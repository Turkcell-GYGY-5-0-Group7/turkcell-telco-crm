# 19.5 Live Verification on the Sprint 15 Kind Cluster: Forged-Header Bypass Attempt Fails; Legitimate Gateway-to-Service and Kafka/Postgres/Redis Traffic Is Unaffected

> Sprint: [Sprint 19 - Service Mesh and mTLS](./README.md)

## Objective

Prove, live on the Sprint 15 Kind cluster, that the residual risk `security-posture.md` Section 8
accepted for the MVP is demonstrably closed and not merely asserted, that the full smoke test passes
unchanged with the mesh and NetworkPolicies in place, and that the existing JWT/RBAC user-identity trust
layer (ADR-011) remains unmodified - this sprint's primary exit gate.

## Subtasks

#### 19.5.1 Live-verify the forged-header bypass attempt is rejected at the mesh policy layer
- ID: 19.5.1
- Title: Live-verify a forged-header bypass attempt is rejected at the mesh policy layer
- Description: On the Sprint 15 Kind cluster, from a pod outside the `api-gateway`'s mesh identity
  (e.g. a throwaway debug pod, or any non-gateway service's pod), attempt a direct HTTP call to a
  downstream domain service's app port carrying forged `X-User-Id`/`X-User-Roles` headers, bypassing
  the gateway entirely. Confirm the call is rejected at the mesh policy layer (19.3.2's
  `AuthorizationPolicy`) before the request ever reaches the service's application code /
  `@PreAuthorize` check - this is the exact residual risk `security-posture.md` Section 8 accepted for
  the MVP, and this task is the live, not merely asserted, proof it is closed, per this sprint's Exit
  Criteria.
- Business Purpose: This is the sprint's primary exit gate and the reason the sprint exists - without a
  live-verified rejection, ADR-026's core claim (closing the header-forgery residual risk) is unproven.
- Inputs: `security-posture.md` Section 8 (the exact residual risk being tested), 19.3.2 (the policy
  under test), Feature 19.4 (NetworkPolicy as the defense-in-depth layer in case a pod is somehow
  unmeshed).
- Outputs: A live-verification record (test pod manifest/command used, request sent, response received
  - rejected at the mTLS/mesh-identity layer, e.g. connection reset or a policy-denied response from
  the proxy, not a 401/403 from the application) added to this sprint's README or a dedicated
  verification note, mirroring the evidentiary standard `deploy/ROLLBACK.md` set for Sprint 15's
  rollback drill.
- Acceptance Criteria:
  - The forged-header request from a non-gateway mesh identity is rejected before reaching application
    code (confirmed via the target service's application logs showing no incoming request, and the mesh
    proxy's logs/metrics showing a denied connection).
  - The same request, sent from the `api-gateway`'s mesh identity (or through the gateway normally, with
    a legitimately-issued JWT, not forged headers), succeeds.
- Dependencies: 19.3.2, 19.4.3
- Complexity: M

#### 19.5.2 Run the full smoke test unchanged with the mesh and NetworkPolicies in place
- ID: 19.5.2
- Title: Run the full smoke test with the mesh and NetworkPolicies in place, unchanged
- Description: Run `deploy/smoke/smoke-test.sh` (Sprint 15.4.3) against the mesh-enabled,
  NetworkPolicy-enforced Kind cluster with no modification to the script itself, confirming legitimate
  traffic - gateway health via Ingress, key-service readiness, an authenticated Keycloak-token read
  through the gateway - passes exactly as it did in Sprint 15, per this sprint's Exit Criteria ("The
  full smoke test... passes unchanged with the mesh and NetworkPolicies in place - proving legitimate
  traffic... is unaffected").
- Business Purpose: Proves the mesh and NetworkPolicy rollout (Features 19.1-19.4) has zero regression
  on the platform's actual, already-verified happy path - the concrete evidence that "zero-trust
  networking" did not become "zero networking."
- Inputs: `deploy/smoke/smoke-test.sh` (unmodified), `deploy/RUNBOOK.md` Section 8 (existing smoke-test
  procedure), Features 19.1-19.4 (everything under test).
- Outputs: A smoke-test run log/record for this sprint (added to the verification note from 19.5.1, or a
  dedicated one), confirming exit code 0.
- Acceptance Criteria:
  - `bash deploy/smoke/smoke-test.sh` exits 0 against the mesh-enabled cluster, with no change to the
    script's source.
  - Every check the script performs (gateway health, key-service readiness, authenticated read) passes.
- Dependencies: 19.5.1
- Complexity: S

#### 19.5.3 Confirm zero change to the JWT/RBAC user-identity trust layer
- ID: 19.5.3
- Title: Confirm zero change to the JWT/RBAC user-identity trust layer (ADR-011)
- Description: Diff-review every service touched by Sprint 19 (chart templates, `SecurityConfig`
  classes, `@PreAuthorize` annotations, mediator `AuthorizationRule` implementations) to confirm none
  were modified - per this sprint's Exit Criteria ("No change to any service's JWT validation,
  `@PreAuthorize` rule, or mediator `AuthorizationRule` - the user-identity trust layer (ADR-011) is
  verified unchanged, per ADR-026 Section 2") and ADR-026 Section 2's explicit warning that conflating
  the two trust layers is "the most likely implementation mistake."
- Business Purpose: Proves ADR-026's central architectural claim - that the mesh is a second, additive,
  orthogonal trust layer, not a replacement for the existing JWT/RBAC model - holds in the actual
  delivered code, not just in the ADR's prose.
- Inputs: ADR-026 Section 2, `architecture/adr/ADR-011-security-foundation.md`, the full diff of every
  change made across Sprint 19 (Features 19.1-19.4).
- Outputs: A verification note confirming the diff scope (chart templates,
  `NetworkPolicy`/`Server`/`AuthorizationPolicy` manifests, RUNBOOK/README docs only - zero `.java`,
  zero `SecurityConfig`, zero `@PreAuthorize`, zero `AuthorizationRule` files touched).
- Acceptance Criteria:
  - A repository-wide diff for the Sprint 19 changeset shows zero modified `.java` files under any
    service's `security`/`config` package or mediator `AuthorizationRule` implementations.
  - `git diff --stat` for the sprint's merge scope contains only `deploy/`, `docs/`, and
    chart-related paths.
- Dependencies: 19.5.1, 19.5.2
- Complexity: S
