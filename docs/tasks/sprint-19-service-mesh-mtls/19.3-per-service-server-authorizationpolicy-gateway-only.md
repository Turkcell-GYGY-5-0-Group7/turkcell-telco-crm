# 19.3 Per-Service Server/AuthorizationPolicy: Only the Gateway's Mesh Identity May Call Downstream Services; /internal/** Remains Edge-Denied

> Sprint: [Sprint 19 - Service Mesh and mTLS](./README.md)
> Feature ID: 19.3
> Subtasks: 19.3.1, 19.3.2, 19.3.3

## Objective

Author a Linkerd `Server` and `AuthorizationPolicy` per service restricting inbound traffic to only the
`api-gateway`'s mesh identity, closing the exact residual risk `security-posture.md` Section 8 accepted
for the MVP (in-cluster header forgery bypassing the gateway), per ADR-026 Section 2 - while confirming
the existing `/internal/**` edge-deny (`GatewaySecurityConfig`) remains unchanged and unaffected.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- Feature 19.2 (mesh active, injection confirmed for all 13 services)
- ADR-026 Service Mesh and mTLS, Section 2 (two distinct trust layers - the precise mechanism this
  feature implements)
- `docs/architecture/security-posture.md` Section 1 and Section 8 (the existing `/internal/**`
  edge-deny and the residual risk being closed)
- 19.3.1

## Subtasks

#### 19.3.1 Server resources per service
- ID: 19.3.1
- Title: Author a Linkerd `Server` resource per service, defining the app port as a policy-enforced
  meshed server
- Description: For each of the 10 domain services and the infra services (`config-server`,
  `discovery-server`) add a `policy.linkerd.io/v1beta1` `Server` resource selecting the service's
  Deployment/pod and app container port (per `docs/architecture/service-catalog.md`'s port table),
  which is the prerequisite object Linkerd's `AuthorizationPolicy` (19.3.2) attaches to - without a
  `Server`, Linkerd's default policy is unrestricted-within-mesh, not deny-by-default.
- Business Purpose: `Server` is the object that switches a service's inbound traffic from Linkerd's
  default "any meshed caller allowed" posture to a policy-enforced posture, the prerequisite for the
  gateway-only restriction ADR-026 Section 2 describes.
- Inputs: ADR-026 Section 1 ("per-service `Server`/`AuthorizationPolicy` resources... devops/
  platform-engineer implementation detail for Sprint 19") and Section 2 (Layer 1 workload-identity
  guarantee), `docs/architecture/service-catalog.md` (per-service ports).
- Outputs: `deploy/helm/telco-service/templates/server.yaml` (a new chart template, one `Server` object
  per service release, values-driven port), rendered for each of the 13 services; the distinction that
  `config-server`/`discovery-server`'s caller set is "all 13 services," not just the gateway, is noted
  explicitly here and carried into 19.3.2's policy authoring rather than force-fit into a single
  gateway-only template.
- Acceptance Criteria:
  - `kubectl -n telco get server` lists one `Server` object per service, each selecting the correct pod
    label and app port.
  - `linkerd -n telco authz deploy/customer-service` (or equivalent inspection command) shows the
    `Server` is active.
- Dependencies: 19.2.1 (mesh must be active and injection confirmed first)
- Complexity: L

#### 19.3.2 AuthorizationPolicy restricting inbound to the gateway's mesh identity
- ID: 19.3.2
- Title: Author `AuthorizationPolicy` + `MeshTLSAuthentication` restricting each domain service's
  `Server` to the `api-gateway` `ServiceAccount`'s mesh identity
- Description: For each domain service's `Server` (19.3.1), add a Linkerd `AuthorizationPolicy`
  referencing a `MeshTLSAuthentication` whose identity references contain only the
  `api-gateway.telco.serviceaccount.identity.linkerd.cluster.local`-style mesh identity Linkerd derives
  from the `api-gateway` pod's `ServiceAccount` (per ADR-026 Section 2's Layer 1 description), so a
  downstream service's `Server` rejects any caller that is not cryptographically the gateway's workload
  identity - independent of what HTTP headers the request carries. Explicitly identify and document
  which non-gateway callers legitimately need direct service-to-service access (if any exist per the
  service-catalog call graph beyond gateway routing - Kafka-consumer-driven flows do not go through the
  gateway and are unaffected by this HTTP-layer policy) so the policy does not inadvertently block a
  legitimate internal call path. Author a distinct, broader policy shape for `config-server`/
  `discovery-server`, whose actual caller set is all 13 services, not the gateway alone.
- Business Purpose: This is ADR-026 Section 2's precise mechanism for closing the residual risk
  `security-posture.md` Section 8 accepted: "an attacker with raw in-cluster network position can no
  longer call a downstream service directly with forged headers, because the downstream service's mesh
  policy rejects any caller that is not cryptographically the gateway's workload identity."
- Inputs: ADR-026 Section 2 (Layer 1 mechanism, quoted above), 19.3.1, `docs/architecture/service-catalog.md`
  (confirming the gateway is the only synchronous HTTP caller of each domain service, per the platform's
  event-driven inter-service design, ADR-009).
- Outputs: `deploy/helm/telco-service/templates/authorizationpolicy.yaml` (new chart template),
  `deploy/helm/telco-service/templates/meshtlsauthentication.yaml` (or inlined identity references),
  rendered per domain service restricting inbound to `api-gateway`'s mesh identity; a distinct policy
  variant for `config-server`/`discovery-server` permitting all 13 services' identities.
- Acceptance Criteria:
  - A request from a non-gateway pod's mesh identity directly to a domain service's app port is
    rejected (verified live in Feature 19.5; this task only authors the policy).
  - A request from the `api-gateway` pod's mesh identity to a domain service succeeds unchanged.
  - `config-server`/`discovery-server`'s `AuthorizationPolicy` permits all 13 services' identities,
    matching their actual caller set, not narrowed to the gateway alone.
- Dependencies: 19.3.1
- Complexity: L

#### 19.3.3 Confirm /internal/** remains edge-denied and unaffected by mesh policy
- ID: 19.3.3
- Title: Verify `/internal/**` edge-deny behavior (`GatewaySecurityConfig`) is unchanged by the new mesh
  policy layer
- Description: `docs/architecture/security-posture.md` Section 1 records that `/internal/**` endpoints
  are already denied at the edge (routed to a local 404 sink, reachable only in-cluster) via
  `GatewaySecurityConfig` - a Layer-2/application-level control, not a mesh control. Confirm the new
  `Server`/`AuthorizationPolicy` resources from 19.3.1/19.3.2 neither weaken this existing edge-deny
  (mesh policy operates at the mTLS/identity layer, beneath and independent of the gateway's
  HTTP-routing-level deny) nor are relied upon as a substitute for it - the two controls are
  complementary, not one replacing the other, mirroring ADR-026 Section 2's "two distinct trust layers"
  framing applied to this specific existing control.
- Business Purpose: Prevents a regression where introducing mesh-level authorization is mistaken for a
  superset of the existing `/internal/**` edge-deny, when in fact they operate at different layers and
  both must remain intact - an explicit verification this sprint's Features table calls out by name
  (19.3), consistent with ADR-026 Section 2's warning that conflating the two trust layers "is the most
  likely implementation mistake."
- Inputs: `docs/architecture/security-posture.md` Sections 1 and "Rationale" (existing `/internal/**`
  edge-deny record), ADR-026 Section 2 (two-layers framing), api-gateway's `GatewaySecurityConfig`
  (existing implementation).
- Outputs: A verification note (no code change expected - this task is confirmatory) recorded in this
  sprint's README or a verification log; if the audit finds any `/internal/**` route inadvertently
  reachable through a newly-authored `AuthorizationPolicy` path, a narrowly-scoped fix to that policy
  (not a `GatewaySecurityConfig` change, since that control is unaffected by the mesh).
- Acceptance Criteria:
  - An external request to any `/internal/**` path continues to receive the existing local-404-sink
    behavior, unchanged, with the mesh active.
  - `GatewaySecurityConfig` has zero code changes attributable to this sprint.
- Dependencies: 19.3.2
- Complexity: S
