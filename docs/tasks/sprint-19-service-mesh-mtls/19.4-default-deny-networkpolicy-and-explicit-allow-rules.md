# 19.4 Default-Deny NetworkPolicy per Namespace + Explicit Allow Rules Matching the Actual Service-Catalog Call Graph

> Sprint: [Sprint 19 - Service Mesh and mTLS](./README.md)
> Feature ID: 19.4
> Subtasks: 19.4.1, 19.4.2, 19.4.3

## Objective

Deliver default-deny Kubernetes `NetworkPolicy` for the `telco` namespace plus explicit allow rules
matching the platform's actual call graph, as the compensating control for pods that are not yet meshed
or bypass the proxy - in scope of this same sprint per ADR-026 Section 3, not deferred.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- Feature 19.1 (mesh install/documentation complete; NetworkPolicy is orthogonal to mesh injection
  status and can proceed independently of Features 19.2/19.3)
- ADR-026 Service Mesh and mTLS, Section 3 (NetworkPolicies in scope, delivered alongside the mesh)
- `docs/architecture/service-catalog.md` (the call graph these rules are derived from)
- 19.4.1

## Subtasks

#### 19.4.1 Default-deny NetworkPolicy baseline
- ID: 19.4.1
- Title: Default-deny `NetworkPolicy` for the `telco` namespace (ingress and egress)
- Description: Add a baseline `NetworkPolicy` selecting all pods in `telco` with empty
  `podSelector: {}`, `policyTypes: [Ingress, Egress]`, and no rules (default-deny both directions), per
  ADR-026 Section 3's decision that default-deny NetworkPolicies are in this ADR's scope, "delivered in
  the same sprint as the mesh rollout," as the compensating control for pods that are not yet meshed or
  bypass the proxy (a non-injected pod, a rollout gap, a misconfigured namespace annotation) - the
  specific gap Linkerd's `Server`/`AuthorizationPolicy` (Feature 19.3) cannot cover because that policy
  only governs the meshed data path.
- Business Purpose: Delivers the network-layer half of the combined "mTLS + default-deny NetworkPolicy"
  production recommendation `security-posture.md` Section 8 named, closing the specific residual risk
  that persists even with a correctly-configured mesh: a pod that is not (yet) meshed reaching another
  pod's application port directly in plain HTTP.
- Inputs: ADR-026 Section 3 (exact rationale, quoted above), `security-posture.md` Section 8 (the
  combined production recommendation this closes).
- Outputs: A default-deny `NetworkPolicy` manifest (e.g.
  `deploy/helm/dependencies/templates/networkpolicy-default-deny.yaml`, or a new small chart if a
  separate release fits the "one chart per concern" convention better - decide and document which),
  applied to the `telco` namespace before any allow rules (19.4.2/19.4.3) are added, so the deny-by-
  default baseline is proven first.
- Acceptance Criteria:
  - With only this NetworkPolicy applied (no allow rules yet), all pod-to-pod traffic in `telco` is
    blocked (verified by a temporary connectivity test before 19.4.2/19.4.3 are applied, then removed
    from the verification record once allow rules restore legitimate traffic).
  - `kubectl -n telco get networkpolicy` shows a default-deny policy selecting all pods.
- Dependencies: 19.1.3 (mesh install/documentation complete)
- Complexity: M

#### 19.4.2 Explicit ingress allow rules matching the service-catalog call graph
- ID: 19.4.2
- Title: Explicit ingress allow rules matching the actual service-catalog call graph
- Description: Add explicit `NetworkPolicy` ingress allow rules restoring every real edge in
  `docs/architecture/service-catalog.md`'s call graph: the Ingress controller -> `api-gateway`
  (external entry), `api-gateway` -> each of the 10 domain services + `config-server`/
  `discovery-server`, and any service -> `config-server`/`discovery-server` (config fetch, service
  registry - every one of the 13 services calls both). Derive rules strictly from the service-catalog
  document rather than from convenience/blanket allows, per this sprint's References ("the service call
  graph Feature 19.4's NetworkPolicy allow rules are derived from").
- Business Purpose: Restores the platform's actual, legitimate traffic on top of the 19.4.1 default-deny
  baseline - without this task the platform is fully non-functional, since default-deny blocks
  everything including required infra traffic.
- Inputs: `docs/architecture/service-catalog.md` (full call graph: gateway routing table, config-server/
  discovery-server as universal callees), ADR-026 Section 3, 19.4.1.
- Outputs: `deploy/helm/telco-service/templates/networkpolicy-ingress.yaml` (per-service `NetworkPolicy`
  allow objects, one per service release, listing permitted source pod selectors, matching the chart's
  existing per-service-release pattern).
- Acceptance Criteria:
  - A request from `api-gateway`'s pod to each domain service's app port succeeds.
  - A request from a domain service to `config-server`'s and `discovery-server`'s ports succeeds.
  - A request from a non-gateway, non-authorized pod directly to a domain service's app port is still
    blocked (default-deny baseline holds for anything not explicitly allowed).
- Dependencies: 19.4.1
- Complexity: L

#### 19.4.3 Explicit egress allow rules for infra dependencies plus DNS
- ID: 19.4.3
- Title: Explicit egress allow rules for each service's own Postgres/Redis/Kafka/MinIO/Keycloak
  dependencies, plus DNS
- Description: Add explicit `NetworkPolicy` egress allow rules per service, restoring only that
  service's own infrastructure dependencies per `docs/architecture/service-catalog.md` Section 5's
  Infrastructure Profile table (e.g. `customer-service` -> Postgres + MinIO; `usage-service` -> Postgres
  + Redis; every Kafka-producing/consuming service -> Kafka + Schema Registry; `identity-service`/
  gateway -> Keycloak), plus the universal cluster-DNS egress allow (CoreDNS on UDP/TCP 53) every pod
  requires regardless of role, and each service's egress to `config-server`/`discovery-server`
  (mirroring the 19.4.2 ingress rule from the other side).
- Business Purpose: Completes the default-deny + explicit-allow model by restoring only the *outbound*
  dependencies each service actually needs, per the specific per-service ownership boundaries in the
  service-catalog - a service that has no legitimate reason to reach another service's database (per
  ADR-006's database-per-service model) cannot do so even if it tried, adding a network-layer
  enforcement of a boundary that was previously only architectural convention.
- Inputs: `docs/architecture/service-catalog.md` Section 5 (per-service infra dependency table),
  `docs/architecture/event-catalog.md` (which services produce/consume which Kafka topics, for the
  Kafka egress rules), ADR-026 Section 3.
- Outputs: `deploy/helm/telco-service/templates/networkpolicy-egress.yaml` (per-service egress allow
  rules, values-driven from each service's `deploy/helm/values/<service>.yaml` declared dependencies), a
  DNS egress allow rule (likely part of the 19.4.1 baseline chart, since it is universal rather than
  per-service).
- Acceptance Criteria:
  - `customer-service`'s pod can reach its own Postgres and MinIO but not another service's Postgres
    instance (verified by a blocked-connection test to a different service's DB).
  - Every pod can resolve in-cluster DNS names (CoreDNS egress allowed universally).
  - The full smoke test's underlying traffic pattern (gateway -> services -> Postgres/Redis/Kafka/
    MinIO/Keycloak) is unblocked (final proof deferred to Feature 19.5's live smoke-test run).
- Dependencies: 19.4.2
- Complexity: L
