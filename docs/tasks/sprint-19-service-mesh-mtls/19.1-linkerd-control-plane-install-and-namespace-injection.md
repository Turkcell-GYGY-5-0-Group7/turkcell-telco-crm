# 19.1 Linkerd Control-Plane Install (deploy/helm/ Release) + Namespace Injection Annotation

> Sprint: [Sprint 19 - Service Mesh and mTLS](./README.md)
> Feature ID: 19.1
> Subtasks: 19.1.1, 19.1.2, 19.1.3

## Objective

Install Linkerd as another `deploy/helm/` release, annotate the `telco` namespace for automatic sidecar
injection, and document the install/verification steps in the RUNBOOK - the foundational control-plane
work every other Sprint 19 feature builds on, per ADR-026 Section 1.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- Sprint 18 - Secret Management (sequenced before this sprint for operational reasons, ADR-026 Section
  4; not a hard technical dependency)
- ADR-026 Service Mesh and mTLS, Section 1 (mesh technology choice) and Section 4 (self-managed trust
  anchor, no Vault dependency)
- `deploy/helm/README.md` (chart-layout and install-order convention this release follows)
- 19.1.1

## Subtasks

#### 19.1.1 Install Linkerd CRDs and control plane as deploy/helm/ releases
- ID: 19.1.1
- Title: Install Linkerd CRDs and control plane as `deploy/helm/` releases
- Description: Add `deploy/helm/linkerd-crds/` and `deploy/helm/linkerd-control-plane/` (Linkerd's
  official two-chart install model: CRDs first, then the control plane with `linkerd-identity`,
  `linkerd-destination`, `linkerd-proxy-injector`) following the same one-chart-per-concern pattern as
  `dependencies`, `telco-service`, and Vault (Sprint 18). Use Linkerd's self-managed trust anchor /
  issuer certificate generation (no external CA), per ADR-026 Section 4 - no dependency on Vault's PKI
  secrets engine.
- Business Purpose: Stands up the mesh control plane every other Sprint 19 feature depends on (sidecar
  injection, per-service policy, live verification).
- Inputs: ADR-026 Section 1 ("Choosing Linkerd") and Section 4 (self-managed trust anchor, no Vault
  dependency), `deploy/helm/README.md` chart-layout convention, Sprint 18's Vault release (18.1.1) as
  the immediately preceding precedent for adding a new security-control-plane `deploy/helm/` release.
- Outputs: `deploy/helm/linkerd-crds/` and `deploy/helm/linkerd-control-plane/` release definitions, a
  `deploy/helm/README.md` chart-layout tree entry and install-order step, `deploy/RUNBOOK.md` Section 2
  readiness wait for `linkerd-identity`/`linkerd-destination`/`linkerd-proxy-injector`.
- Acceptance Criteria:
  - `linkerd check` (Linkerd's own CLI health-check command) reports all control-plane checks green.
  - `helm lint`/`helm template` for the new releases succeed with no external CA/cert-manager dependency
    configured.
- Dependencies: none (first task of Sprint 19; sequenced after Sprint 18 completion for operational
  reasons per ADR-026 Section 4, not a hard technical dependency)
- Complexity: M

#### 19.1.2 Namespace injection annotation
- ID: 19.1.2
- Title: Annotate the `telco` namespace for automatic Linkerd proxy injection
- Description: Add `linkerd.io/inject: enabled` to the `telco` namespace (created by the `dependencies`
  chart per `deploy/helm/README.md`) so every pod scheduled into it is automatically proxy-injected by
  `linkerd-proxy-injector`, with no per-Deployment annotation required across the 13 services' charts -
  zero change to `deployment.yaml` templates, consistent with ADR-026 Section 1's "automatic sidecar
  injection... with zero application code change" framing extended to zero per-service chart-authoring
  change.
- Business Purpose: Delivers automatic sidecar injection (this ADR's core mechanism) without touching
  any of the 13 services' individual Helm values or templates - namespace-level annotation is the
  single point of control.
- Inputs: ADR-026 Section 1 ("automatic sidecar injection"), `deploy/helm/dependencies/` (the
  namespace-creation location the annotation is applied to, or a documented `kubectl annotate` RUNBOOK
  step if the namespace is created outside a chart template).
- Outputs: A namespace annotation, applied either via a `deploy/helm/dependencies/templates/namespace.yaml`
  update or a documented `kubectl annotate namespace telco linkerd.io/inject=enabled` RUNBOOK step -
  confirm which mechanism the namespace actually uses and document precisely.
- Acceptance Criteria:
  - `kubectl get namespace telco -o jsonpath='{.metadata.annotations}'` shows `linkerd.io/inject: enabled`.
  - A newly-created pod in `telco` (after this change) has 2 containers (`app` + `linkerd-proxy`)
    without any per-service chart change.
- Dependencies: 19.1.1
- Complexity: S

#### 19.1.3 RUNBOOK/README mesh install and verification documentation
- ID: 19.1.3
- Title: Document the mesh install step and a mesh-verification command in `deploy/RUNBOOK.md` and
  `deploy/helm/README.md`
- Description: Extend `deploy/RUNBOOK.md`'s deploy sequence (Section 4) and `deploy/helm/README.md`'s
  install-order section with the Linkerd install step (19.1.1) and namespace annotation (19.1.2), plus
  a mesh-verification command (`linkerd check --proxy` and/or `linkerd viz stat deploy -n telco`, if
  `linkerd-viz` is added) matching the "operational-completeness bar Sprint 15 set" this sprint's
  README Sprint Deliverables names explicitly.
- Business Purpose: Keeps the RUNBOOK the single operational entry point (its own stated purpose)
  current with the new mesh dependency, so a fresh environment can be brought up following only that
  document, per Sprint 15's precedent.
- Inputs: `deploy/RUNBOOK.md` Sections 2 and 4, `deploy/helm/README.md` install-order section, this
  sprint's README Sprint Deliverables bullet naming this exact documentation requirement.
- Outputs: `deploy/RUNBOOK.md` updates (install step + verification command), `deploy/helm/README.md`
  updates (chart-layout tree + install order).
- Acceptance Criteria:
  - Following only `deploy/RUNBOOK.md`, an operator can install Linkerd, annotate the namespace, and
    verify the mesh is healthy before proceeding to service verification (Features 19.2/19.5).
- Dependencies: 19.1.1, 19.1.2
- Complexity: S
