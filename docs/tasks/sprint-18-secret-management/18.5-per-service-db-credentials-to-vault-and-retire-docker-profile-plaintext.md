# 18.5 Per-Service DB Credentials into Vault KV v2; Retire the docker-Profile Plaintext DB Block for In-Cluster Deployment

> Sprint: [Sprint 18 - Secret Management](./README.md)
> Feature ID: 18.5
> Subtasks: 18.5.1, 18.5.2, 18.5.3

## Objective

Close the gap ADR-025's Context section identifies - per-service database credentials are not secreted
at all today, baked in plaintext into the `docker`-profile YAML config-server serves - by generating
real per-service DB credentials into Vault KV v2, moving in-cluster deployment off the `docker`
profile's plaintext DB block onto an environment-sourced profile, and syncing the credentials via CSI.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- Feature 18.4 (working Vault/CSI chain proven for the five core secrets)
- Feature 18.2 (per-service Vault policies, already scoped by service prefix)
- ADR-025 Secrets and Key Management, Context section (the plaintext-DB-credential gap) and Section 4
  (interaction with config-server / profile change deferred to this sprint)
- 18.5.1

## Subtasks

#### 18.5.1 Generate and write per-service DB credentials to Vault
- ID: 18.5.1
- Title: Generate real per-service database credentials and write them to Vault KV v2
  (`secret/<service>/db-credentials`)
- Description: For every PostgreSQL-backed service (per `docs/architecture/service-catalog.md` Section
  5's Infrastructure Profile table - all services except `api-gateway`, `discovery-server`,
  `config-server`, plus `notification-service`'s PostgreSQL outbox database even though its primary
  store is MongoDB), generate a real (non-`<service>`/`<service>` dev-default) DB username/password
  pair and write it to Vault KV v2 at `secret/<service>/db-credentials`, per ADR-025 Section 2's
  mapping row for per-service DB credentials. This closes the gap ADR-025's Context section identifies:
  DB credentials are "not secreted at all today... baked in plaintext into the profile-scoped YAML
  config-server serves."
- Business Purpose: Closes the previously-undocumented finding ADR-025's research surfaced - the
  least-protected secret class in the platform today becomes Vault-sourced and per-service
  policy-scoped, the same as every other credential this sprint migrates.
- Inputs: ADR-025 Context section and Section 2's DB-credentials row, `docs/architecture/service-catalog.md`
  Section 5 (which services are PostgreSQL-backed), `microservices/configs/<service>/application-docker.yml`
  files (today's plaintext values, one per service, to be superseded).
- Outputs: A documented `vault kv put secret/<service>/db-credentials username=... password=...`
  procedure per PostgreSQL-backed service (added to the RUNBOOK's Vault section), confirmation that
  each service's existing Vault policy (18.2.2, scoped to `secret/data/<service>/*`) already covers the
  `db-credentials` path with no new policy needed.
- Acceptance Criteria:
  - `vault kv get secret/customer-service/db-credentials` returns a `username`/`password` pair distinct
    from the committed `customer`/`customer` dev default.
  - Every PostgreSQL-backed service (per the service-catalog table) has a corresponding Vault path
    populated.
- Dependencies: 18.4.3 (builds on the working Vault/CSI chain), 18.2.2 (policy paths already scoped per
  service)
- Complexity: M

#### 18.5.2 Audit and adopt an environment-sourced DB-credential profile for in-cluster deployment
- ID: 18.5.2
- Title: Retire the in-cluster use of the `docker` profile's plaintext DB block in favor of `prod` (or a
  new `k8s` profile) that reads DB credentials from the environment
- Description: Per ADR-025 Section 4's explicit deferral to this sprint ("in-cluster services must move
  from the `docker` profile... to a profile that reads DB credentials from the environment... That
  profile change is implementation, not decided further here - it is Sprint 18 scope"), audit each
  service's `application-prod.yml` (which already anticipates
  `${<SERVICE>_DB_URL}`/`${<SERVICE>_DB_USER}`/`${<SERVICE>_DB_PASSWORD}`-style placeholders, e.g.
  `microservices/configs/customer-service/application-prod.yml`) against the `docker` profile it
  replaces, for other environment-specific assumptions not yet audited (per ADR-025's Negative
  consequence: "that audit is explicitly deferred to Sprint 18 implementation"). Decide, per service,
  whether `prod` is safe to activate in-cluster as-is or whether a new `k8s` profile is needed to avoid
  pulling in unrelated `prod`-only assumptions. Update `SPRING_PROFILES_ACTIVE` in each service's
  `deploy/helm/values/<service>.yaml` `config:` map accordingly (currently `dev,docker` per
  `deploy/helm/README.md`).
- Business Purpose: This is the specific action that closes the plaintext-DB-credential gap at the
  application-configuration level - generating a Vault-sourced credential (18.5.1) is meaningless if
  the running Spring profile still bakes in the plaintext `docker`-profile block instead of reading the
  environment-sourced placeholder.
- Inputs: ADR-025 Section 4 and its Negative consequence, `microservices/configs/customer-service/application-docker.yml`
  and `application-prod.yml` (the exact gap this task closes, named explicitly in this sprint's README
  References), the equivalent `application-docker.yml`/`application-prod.yml` pair for every other
  PostgreSQL-backed service.
- Outputs: A per-service audit note (which profile-specific assumptions in `application-prod.yml`
  differ from `application-docker.yml` beyond DB credentials - MinIO endpoint, JWKS URI, etc. - and
  whether they are safe for in-cluster use), `microservices/configs/<service>/application-k8s.yml`
  (new files, only if the audit finds `prod` unsuitable as-is) or confirmation that `prod` is reused,
  `deploy/helm/values/<service>.yaml` `SPRING_PROFILES_ACTIVE` updates (`dev,docker` -> `dev,prod` or
  `dev,k8s`) for every affected service.
- Acceptance Criteria:
  - The audit note explicitly lists, per service, every place `application-prod.yml` diverges from
    `application-docker.yml` and a decision (reuse `prod` / introduce `k8s`) for each.
  - No service's in-cluster `SPRING_PROFILES_ACTIVE` retains the plaintext `docker`-profile DB block
    once this task ships.
- Dependencies: 18.5.1
- Complexity: L

#### 18.5.3 Sync DB credentials via CSI and live-verify
- ID: 18.5.3
- Title: Extend `SecretProviderClass`/`secretObjects` to sync DB-credential keys and live-verify every
  service's environment-sourced DB connection
- Description: Extend each PostgreSQL-backed service's `SecretProviderClass` (from 18.3.2) to also sync
  the `secret/<service>/db-credentials` path from 18.5.1 into the same `<service>-secret` object, under
  the key names the 18.5.2 profile change expects (`CUSTOMER_DB_URL`, `CUSTOMER_DB_USER`,
  `CUSTOMER_DB_PASSWORD` for customer-service, and the equivalent per-service names). Live-verify on the
  Sprint 15 Kind cluster that every affected service connects to its own Postgres database using the
  Vault-sourced credential and the 18.5.2 profile, closing the ADR-025 Context gap end to end.
- Business Purpose: Completes the DB-credential migration end to end - Vault storage (18.5.1) + profile
  change (18.5.2) + CSI delivery (this task) together are what actually gets a real, non-committed DB
  password into a running pod's environment.
- Inputs: 18.3.2 (existing `SecretProviderClass` pattern), 18.5.1 (Vault paths), 18.5.2
  (profile/env-var expectations), `deploy/RUNBOOK.md` Section 2 (live-verification standard).
- Outputs: `SecretProviderClass` updates (additional `secretObjects.data` entries) per PostgreSQL-backed
  service, `deploy/helm/values/<service>.yaml` config additions if `CUSTOMER_DB_URL`-style non-secret
  parts (host/port/dbname) need to move from a hardcoded value to a `config:` entry.
- Acceptance Criteria:
  - Every PostgreSQL-backed service's pod connects successfully to its own database using credentials
    sourced only from Vault (verified by a successful health check and, for at least one service, an
    inspected `env` output confirming the value matches Vault, not the old `customer`/`customer`-style
    default).
  - `microservices/configs/*/application-docker.yml`'s plaintext DB block is no longer the block
    resolved at runtime for any in-cluster service (confirmed via active-profile inspection).
- Dependencies: 18.5.2
- Complexity: M

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met.

## Deliverables

- Real, per-service DB credentials in Vault KV v2, one policy-scoped path per service.
- Every PostgreSQL-backed service moved off the `docker` profile's plaintext DB block onto an
  environment-sourced profile (`prod` or a new `k8s` profile).
- Live-verified, Vault-sourced DB connectivity for every PostgreSQL-backed service on the Sprint 15
  Kind cluster.

## Definition of Done

- All subtasks (18.5.1, 18.5.2, 18.5.3) completed and their acceptance criteria verified.
- All listed dependencies satisfied.
- Live verification on the Sprint 15 Kind cluster passes, and no service's in-cluster deployment reads
  a plaintext DB credential, per the global conventions in [../README.md](../README.md) (Section 3).
