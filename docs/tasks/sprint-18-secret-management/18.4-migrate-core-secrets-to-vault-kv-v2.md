# 18.4 Migrate ENCRYPT_KEY, CUSTOMER_AES_KEY, CONFIG_SERVER_PASSWORD, EUREKA_PASSWORD, REDIS_PASSWORD from Committed Helm Defaults into Vault KV v2

> Sprint: [Sprint 18 - Secret Management](./README.md)
> Feature ID: 18.4
> Subtasks: 18.4.1, 18.4.2, 18.4.3

## Objective

Generate real, non-committed values for the five credentials named in ADR-025 Section 2's mapping
table, write them to Vault KV v2, retire the committed DEV-ONLY defaults in `deploy/helm/values/*.yaml`
for any non-local environment, and live-verify all 13 services boot with the Vault-sourced,
CSI-synced values.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- Feature 18.3 (CSI sync path must exist to consume these values)
- Feature 18.2 (per-service policy paths)
- ADR-025 Secrets and Key Management, Section 2 (what moves to Vault vs stays a K8s Secret/ConfigMap)
- 18.4.1

## Subtasks

#### 18.4.1 Generate and write real secret values into Vault KV v2
- ID: 18.4.1
- Title: Generate non-committed values for `ENCRYPT_KEY`, `CUSTOMER_AES_KEY`, `CONFIG_SERVER_PASSWORD`,
  `EUREKA_PASSWORD`, `REDIS_PASSWORD` and write them to Vault KV v2
- Description: For each of the five credentials named in ADR-025 Section 2's mapping table, generate a
  real value (matching the existing generation convention where documented - `openssl rand -hex 32` for
  `ENCRYPT_KEY` per `deploy/helm/values/config-server.yaml`'s comment; a fresh base64-encoded 32-byte
  value for `CUSTOMER_AES_KEY` matching `AesKeyProvider`'s 32-byte AES-256 validation) and
  `vault kv put` it at the KV v2 path ADR-025 Section 2 specifies (`secret/config-server/encrypt-key`,
  `secret/customer-service/aes-key`, and one path per service for the three shared-credential passwords,
  generated per-environment rather than reused across services).
- Business Purpose: Establishes the platform's first real, non-committed secret material - ADR-025's
  Consequences name this as "a prerequisite for any production environment, not just a hardening
  nicety."
- Inputs: ADR-025 Section 2 (mapping table), `deploy/helm/values/config-server.yaml` and
  `deploy/helm/values/customer-service.yaml` (today's DEV-ONLY committed values and their
  generation-method comments), `AesKeyProvider.java`
  (`microservices/customer-service/src/main/java/com/telco/customer/infrastructure/crypto/AesKeyProvider.java`
  - the 32-byte AES-256 validation the generated key must satisfy).
- Outputs: A documented `vault kv put` procedure per credential (added to the RUNBOOK's Vault section);
  no committed key material anywhere in the repository.
- Acceptance Criteria:
  - `vault kv get secret/config-server/encrypt-key` returns a 64-hex-char value distinct from the
    currently-committed dev default.
  - `vault kv get secret/customer-service/aes-key` returns a base64 value that decodes to exactly 32
    bytes (per `AesKeyProvider`'s validation).
  - None of the five generated values appear in any file under `deploy/helm/` after this task.
- Dependencies: 18.3.3 (SecretProviderClass/sync path must exist to consume these), 18.2.2 (policy
  paths must match)
- Complexity: M

#### 18.4.2 Remove committed DEV-ONLY defaults; update chart and docs guidance
- ID: 18.4.2
- Title: Retire the committed `secrets:` dev-default values in favor of Vault-sourced values for any
  non-local environment
- Description: Update `deploy/helm/values/*.yaml` so the `secrets:` map (still present for the
  `vault.enabled=false` local-dev path from 18.3.3) is clearly documented as local-only, and update
  `deploy/helm/README.md`'s "Config / Secret model" section and `deploy/RUNBOOK.md` Section 3 to
  replace the "DEV-ONLY defaults... override at install time" guidance with the Vault-sourced
  procedure, per this sprint's README Sprint Deliverable: "No committed real secret value anywhere in
  `deploy/helm/values/*.yaml` after this sprint... retired in favor of Vault-sourced values for any
  non-local environment."
- Business Purpose: Closes the "Real secrets from Vault/K8s Secret" line item in
  `security-posture.md` Section 10's hardening checklist, which ADR-025's Supersession section names
  explicitly.
- Inputs: ADR-025 Supersession section, `deploy/helm/README.md` "Config / Secret model" (current
  guidance being replaced), `deploy/RUNBOOK.md` Section 3.
- Outputs: `deploy/helm/README.md` "Config / Secret model" section rewrite, `deploy/RUNBOOK.md`
  Section 3 rewrite, a flagged note for the `security` agent to update
  `docs/architecture/security-posture.md` Section 10's checklist item to reflect closure (out of this
  sprint directory's write scope - coordinate rather than edit directly).
- Acceptance Criteria:
  - `deploy/helm/README.md` and `deploy/RUNBOOK.md` describe the Vault-backed flow as the primary path
    and the static `secrets:` map as explicitly local-dev-only.
  - A coordination note/flag exists for `security-posture.md` Section 10's "Real secrets from
    Vault/K8s Secret" item to be marked closed once this sprint ships.
- Dependencies: 18.4.1
- Complexity: S

#### 18.4.3 Live verification - 13 services boot with Vault-sourced secrets
- ID: 18.4.3
- Title: Live-verify all 13 services start with Vault-sourced, CSI-synced secret values
- Description: On the Sprint 15 Kind cluster (`deploy/RUNBOOK.md` Section 2), install Vault (18.1), the
  CSI stack (18.3.1), per-service policies/roles (18.2), `SecretProviderClass`es (18.3.2), and the
  updated chart (18.3.3), populate Vault with the 18.4.1 values, and confirm every one of the 13
  services' pods reaches `Ready` with its secret environment populated from the synced Secret - the
  same verification standard Sprint 15 itself used, per this sprint's Exit Criteria.
- Business Purpose: Proves the end-to-end Vault -> CSI -> Secret -> `envFrom` chain works for real, not
  just per-component in isolation; this is the sprint's primary exit gate.
- Inputs: Features 18.1-18.3, subtask 18.4.2, `deploy/RUNBOOK.md` Section 2 (cluster bring-up) and
  Section 4 (deploy sequence).
- Outputs: A live-verification record (added to this sprint's README or a dedicated verification note,
  per the pattern Sprint 15 used), an updated `deploy/RUNBOOK.md` install sequence reflecting the new
  Vault + CSI steps ahead of the service install loop.
- Acceptance Criteria:
  - `kubectl -n telco get pods` shows all 13 services `Running`/`Ready` with no
    `CreateContainerConfigError`.
  - For at least `config-server` and `customer-service`, the running pod's environment
    (`kubectl -n telco exec ... -- env`) shows the Vault-generated values from 18.4.1, not the old
    committed dev defaults.
  - Deleting a service's Vault policy (per this sprint's Exit Criteria) causes that service's
    `SecretProviderClass` sync to fail, verified live.
- Dependencies: 18.4.2
- Complexity: M
