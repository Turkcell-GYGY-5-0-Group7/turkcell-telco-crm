# 18.1 Vault Helm Release: Standalone Server, Raft Storage, Unseal Procedure

> Sprint: [Sprint 18 - Secret Management](./README.md)
> Feature ID: 18.1
> Subtasks: 18.1.1, 18.1.2, 18.1.3

## Objective

Stand up HashiCorp Vault in-cluster as a fourth `deploy/helm/` release (alongside `dependencies`,
`telco-service`, and the frontend/BFF chart), in standalone mode with Integrated Storage (Raft),
single node, per ADR-025 Section 1 - the foundational infrastructure every other Sprint 18 feature
(auth method, policies, CSI sync) builds on.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- Sprint 15 (Helm chart layout and install-order convention this release follows)
- ADR-025 Secrets and Key Management, Section 1 (Vault deployment model)
- 18.1.1

## Subtasks

#### 18.1.1 Vault Helm chart release wrapping hashicorp/vault
- ID: 18.1.1
- Title: Add a `deploy/helm/vault/` release wrapping the official `hashicorp/vault` chart in
  standalone/Raft mode
- Description: Vendor or reference the official `hashicorp/vault` Helm chart (as a chart dependency,
  matching the `dependencies` chart's pattern of one release per moving part) and author a
  `deploy/helm/vault/values.yaml` that pins `server.standalone.enabled=true`, `server.ha.enabled=false`
  with Integrated Storage (Raft) as the storage backend and `replicas: 1`, per ADR-025 Section 1's
  "standalone, Integrated Storage (Raft), single node" decision for this platform's current scale. The
  Vault Agent sidecar injector (`injector.enabled`) is explicitly disabled - ADR-025 Section 1 rejects
  it in favor of the CSI-sync delivery model (see Feature 18.3).
- Business Purpose: Establishes the fourth `deploy/helm/` release (per ADR-025's "one chart per
  concern, one release per moving part") that the rest of Sprint 18 depends on; without a running Vault
  server, no auth method, policy, or CSI sync can be configured.
- Inputs: ADR-025 Section 1 ("Vault deployment model"), `deploy/helm/README.md` (chart layout
  convention), `deploy/helm/dependencies/` (existing pattern for a community/vendored Helm chart as a
  `deploy/helm/` release).
- Outputs: `deploy/helm/vault/Chart.yaml` (dependency on `hashicorp/vault`), `deploy/helm/vault/values.yaml`
  (standalone + Raft + injector disabled), a `deploy/helm/README.md` chart-layout tree entry for
  `vault/` and an "Install order" update inserting Vault as an explicit step ahead of any service that
  will consume Vault-sourced secrets.
- Acceptance Criteria:
  - `helm template vault deploy/helm/vault -n telco` renders a single-replica StatefulSet with Raft
    storage and no injector Deployment.
  - `helm lint deploy/helm/vault` passes.
  - `deploy/helm/README.md`'s chart-layout tree and install-order sections list Vault as an explicit,
    ordered step.
- Dependencies: none (first task of Sprint 18)
- Complexity: M

#### 18.1.2 Initialize and unseal Vault; document the operator procedure
- ID: 18.1.2
- Title: Vault initialization (`vault operator init`) and Shamir unseal procedure, documented
- Description: Author the one-time `vault operator init` and `vault operator unseal` procedure for the
  single-node Raft server from 18.1.1, following the "dev-grade in-cluster dependency, hardened later"
  posture ADR-025 Section 1 explicitly accepts for this platform's scale ("a single-node Vault with
  Shamir-sealed manual unseal is an accepted MVP posture"). Document where unseal keys and the root
  token are expected to be stored operationally - explicitly NOT committed to the repository, mirroring
  the "DEV-ONLY defaults... no real secret committed" posture `deploy/helm/README.md` already uses for
  other secrets - and add a `deploy/RUNBOOK.md` section describing the manual init/unseal steps an
  operator runs after `helm install vault`.
- Business Purpose: Vault is unusable (sealed) immediately after install; every downstream Sprint 18
  feature (auth method, policies, CSI sync) requires an initialized, unsealed Vault. Auto-unseal is
  explicitly out of scope for this platform's current scale (ADR-025 Section 1), so a documented manual
  procedure - not automation - is the deliverable.
- Inputs: ADR-025 Section 1 (Shamir-sealed manual unseal as accepted MVP posture), `deploy/RUNBOOK.md`
  Sections 1-2 (prerequisites / cluster bring-up) as the format template for the new section.
- Outputs: A new `deploy/RUNBOOK.md` section ("Vault initialization and unseal") describing `kubectl exec`
  into the Vault pod, `vault operator init`, key-share storage guidance, and `vault operator unseal`
  invocation; a note in `deploy/helm/vault/values.yaml`'s comments (or a `deploy/helm/vault/README.md`)
  that this is a manual, non-automated MVP step.
- Acceptance Criteria:
  - Following only the documented steps, a freshly-installed Vault pod from 18.1.1 reaches
    `Initialized: true, Sealed: false` (`vault status`).
  - The procedure states explicitly that unseal keys and the root token are never committed to the
    repository.
- Dependencies: 18.1.1
- Complexity: S

#### 18.1.3 Vault as a singleton infra dependency - readiness wiring and RUNBOOK bring-up integration
- ID: 18.1.3
- Title: Wire Vault into the Sprint 15 cluster bring-up sequence as a singleton dependency
- Description: Vault, like config-server and discovery-server, is a single-replica singleton at this
  platform's scale (ADR-025 Section 1) - not autoscaled, no PDB, mirroring `deploy/helm/README.md`'s
  "HPA / PDB" section rationale for singletons (a PDB on one replica blocks node drains). Add Vault's
  readiness wait to `deploy/RUNBOOK.md` Section 2 ("Bring up a cluster") alongside the existing
  ingress-nginx/metrics-server waits, so Vault is confirmed ready before Section 4's service install
  loop runs (services will eventually depend on Vault-sourced secrets via Features 18.3/18.4).
- Business Purpose: Prevents a race where `helm upgrade --install <service>` runs before Vault (and its
  CSI sync, landed in Feature 18.3) is ready, which would leave a service's Secret unpopulated and the
  pod stuck in `CreateContainerConfigError`.
- Inputs: `deploy/RUNBOOK.md` Section 2 (existing wait pattern for ingress-nginx/metrics-server),
  `deploy/helm/README.md` "HPA / PDB" section (existing singleton-service precedent for
  config-server/discovery-server).
- Outputs: `deploy/RUNBOOK.md` Section 2 addition (a Vault readiness wait, e.g.
  `kubectl -n telco rollout status statefulset/vault --timeout=180s`), `deploy/helm/vault/values.yaml`
  singleton overrides if the vendored chart's defaults need pinning (`server.ha.replicas: 1`, no PDB).
- Acceptance Criteria:
  - `deploy/RUNBOOK.md`'s cluster bring-up sequence includes a Vault readiness wait before any service
    install step.
  - The Vault StatefulSet has no associated PDB and is not targeted by any HPA.
- Dependencies: 18.1.1, 18.1.2
- Complexity: S
