# 18.3 Secrets Store CSI Driver + Vault CSI Provider, SecretProviderClass per Service, secretObjects Sync to <service>-secret

> Sprint: [Sprint 18 - Secret Management](./README.md)
> Feature ID: 18.3
> Subtasks: 18.3.1, 18.3.2, 18.3.3

## Objective

Install the Secrets Store CSI Driver and the Vault CSI provider, author a `SecretProviderClass` per
service with `secretObjects` sync to the existing `<service>-secret` object, and switch the
`telco-service` chart's Secret source from a static `secrets:` values map to the CSI-synced Secret -
with zero change to any service's `envFrom`, Spring configuration, or Dockerfile, per ADR-025 Section 1.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- Feature 18.1 (Vault installed, initialized, unsealed)
- Feature 18.2 (Kubernetes auth method, per-service policies and roles)
- ADR-025 Secrets and Key Management, Section 1 (Delivery: Secrets Store CSI Driver + Vault CSI
  provider, explicitly not the Agent sidecar injector)
- 18.3.1

## Subtasks

#### 18.3.1 Install Secrets Store CSI Driver + Vault CSI provider
- ID: 18.3.1
- Title: Install the Secrets Store CSI Driver and the Vault CSI provider as `deploy/helm/` releases
- Description: Add the `secrets-store-csi-driver` and `hashicorp/vault-csi-provider` Helm releases (via
  the same vendored/community-chart pattern as 18.1.1's `hashicorp/vault`), deployed as a DaemonSet
  across cluster nodes per the CSI driver's standard install model, ahead of any service's
  `SecretProviderClass`. This is the delivery mechanism ADR-025 Section 1 selects explicitly over the
  Vault Agent sidecar injector, because this platform's Spring Boot config model is env-var driven
  (`@Value`, `${VAR}` placeholders), not file driven.
- Business Purpose: The CSI driver + Vault provider is the runtime component that actually performs the
  `secretObjects` sync from Vault KV v2 into native Kubernetes Secrets - without it, no
  `SecretProviderClass` (18.3.2) has anything to execute against.
- Inputs: ADR-025 Section 1 ("Delivery: Secrets Store CSI Driver + Vault CSI provider..."), the
  Kubernetes Secrets Store CSI Driver project's standard Helm chart, `deploy/helm/README.md`
  chart-layout convention.
- Outputs: `deploy/helm/csi-driver/` (or equivalently named) release wrapping `secrets-store-csi-driver`
  + `vault-csi-provider`, a `deploy/helm/README.md` chart-layout tree entry and install-order step
  (after Vault/18.1, before any service using CSI-synced secrets), a `deploy/RUNBOOK.md` Section 2
  readiness wait for the CSI driver DaemonSet.
- Acceptance Criteria:
  - `kubectl -n telco get daemonset` shows the CSI driver and Vault provider pods `Ready` on every node.
  - `kubectl get csidriver secrets-store.csi.k8s.io` exists in the cluster.
- Dependencies: 18.1.3, 18.2.3
- Complexity: M

#### 18.3.2 Per-service SecretProviderClass with secretObjects sync
- ID: 18.3.2
- Title: Author a `SecretProviderClass` per service mapping each Vault KV path to the existing
  `<service>-secret` key names
- Description: For each service currently consuming a `<service>-secret` object (per
  `deploy/helm/README.md`'s secret-key table), author a `SecretProviderClass`
  (`spec.provider: vault`, `spec.parameters.vaultAddress`, `roleName: <service>`, an `objects` list of
  Vault KV v2 path/key pairs) with a `secretObjects` block that materializes the synced values as an
  ordinary Kubernetes `Secret` named `<service>-secret` with the exact same keys used today
  (`ENCRYPT_KEY`, `CUSTOMER_AES_KEY`, `CONFIG_SERVER_PASSWORD`, `EUREKA_PASSWORD`, `REDIS_PASSWORD`),
  per ADR-025 Section 1's explicit requirement that the resulting Secret preserve today's key names so
  `envFrom.secretRef` is unaffected.
- Business Purpose: This is ADR-025 Section 1's core mechanism claim made concrete: "zero change to
  `deployment.yaml`'s `envFrom`, zero change to any service's Spring config, zero change to any
  Dockerfile - only the Secret *source* changes."
- Inputs: 18.3.1, 18.2.2/18.2.3 (per-service policy/role), `deploy/helm/README.md`'s secret-key table
  (exact key names to preserve), `deploy/helm/telco-service/templates/secret.yaml` (the static Secret
  template this replaces).
- Outputs: `deploy/helm/telco-service/templates/secretproviderclass.yaml` (a new chart template,
  values-driven, rendered only when `.Values.vault.enabled` is set), one values fragment per service
  (`deploy/helm/values/<service>.yaml` additions defining the Vault paths/keys to sync).
- Acceptance Criteria:
  - `kubectl -n telco get secretproviderclass customer-service` exists and its `secretObjects[0].data`
    lists `CUSTOMER_AES_KEY`, `CONFIG_SERVER_PASSWORD`, `EUREKA_PASSWORD`, `REDIS_PASSWORD` with the
    same key casing the chart uses today.
  - Mounting the `SecretProviderClass` as a volume in a test pod produces a synced
    `customer-service-secret` Kubernetes Secret with those same keys populated from Vault.
- Dependencies: 18.3.1
- Complexity: L

#### 18.3.3 Replace the telco-service chart's static Secret source with the CSI-synced Secret
- ID: 18.3.3
- Title: Switch `deployment.yaml`'s Secret dependency from the static `secret.yaml` template to the
  CSI-synced `secretObjects` Secret
- Description: Mount the `SecretProviderClass` from 18.3.2 as a read-only CSI volume on each service's
  pod (triggering the `secretObjects` sync on pod start), retire the static `secrets:` values map and
  `templates/secret.yaml` in favor of the CSI-synced Secret of the same name (`<service>-secret`), and
  confirm `deployment.yaml`'s existing `envFrom.secretRef` line requires zero change - ADR-025 Section
  1's central claim. Gate this per-release with a `vault.enabled` flag so the chart can still render a
  plain dev Secret locally when Vault is unavailable, matching the platform's existing dev-first
  posture.
- Business Purpose: Completes the migration from "static Helm-committed Secret" to "Vault-sourced,
  CSI-synced Secret" without touching any of the 13 services' application code, Dockerfiles, or Spring
  configuration - the zero-blast-radius property ADR-025 Section 1 argues for over the sidecar injector.
- Inputs: 18.3.2, `deploy/helm/telco-service/templates/deployment.yaml` (existing `envFrom` block),
  `deploy/helm/telco-service/templates/secret.yaml` (existing static Secret template being retired).
- Outputs: `deploy/helm/telco-service/templates/deployment.yaml` update (CSI volume + volumeMount,
  gated by `.Values.vault.enabled`), `deploy/helm/telco-service/templates/secret.yaml` update (rendered
  only when `.Values.vault.enabled` is false, for local/dev), `deploy/helm/README.md` "Config / Secret
  model" section rewritten to describe the Vault-backed flow.
- Acceptance Criteria:
  - With `vault.enabled=true`, `helm template <service>` renders no static `secrets:`-sourced Secret
    and instead a CSI volume referencing the service's `SecretProviderClass`; the pod's
    `envFrom.secretRef` is byte-for-byte unchanged from before this feature.
  - With `vault.enabled=false` (default for local dev), the chart behaves exactly as it does today
    (static dev-default Secret).
  - No `Dockerfile` or application `.java`/`.yml` file changes anywhere in the 13 services.
- Dependencies: 18.3.2
- Complexity: M

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met.

## Deliverables

- The CSI driver + Vault CSI provider running as a `deploy/helm/` release.
- One `SecretProviderClass` per service, syncing to a `<service>-secret` object with unchanged key
  names.
- The `telco-service` chart's Secret source switched from static `secrets:` values to CSI-synced,
  gated by a `vault.enabled` flag, with zero change to any service's `envFrom`, Spring config, or
  Dockerfile.

## Definition of Done

- All subtasks (18.3.1, 18.3.2, 18.3.3) completed and their acceptance criteria verified.
- All listed dependencies satisfied.
- `helm lint`/`helm template` are green for both `vault.enabled=true` and `vault.enabled=false`, and
  the CSI sync is live-verified on the Sprint 15 Kind cluster, per the global conventions in
  [../README.md](../README.md) (Section 3).
