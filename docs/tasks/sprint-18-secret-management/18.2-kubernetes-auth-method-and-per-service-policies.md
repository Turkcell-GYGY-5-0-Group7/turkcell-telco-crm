# 18.2 Kubernetes Auth Method + Per-Service Policies Bound to Existing ServiceAccounts

> Sprint: [Sprint 18 - Secret Management](./README.md)
> Feature ID: 18.2
> Subtasks: 18.2.1, 18.2.2, 18.2.3

## Objective

Enable Vault's Kubernetes auth method and author one least-privilege Vault policy and Kubernetes auth
role per service, bound to that service's existing `ServiceAccount`, per ADR-025 Section 1 - replacing
"no service can read another service's secrets, by convention" with a Vault-enforced guarantee.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- Feature 18.1 (Vault must be installed, initialized, and unsealed)
- ADR-025 Secrets and Key Management, Section 1 (Authentication: the Kubernetes auth method)
- ADR-017 Service Template Standard (per-service `ServiceAccount` this feature binds to)
- 18.2.1

## Subtasks

#### 18.2.1 Enable and configure the Kubernetes auth method
- ID: 18.2.1
- Title: Enable Vault's Kubernetes auth method against the Sprint 15 cluster's API server
- Description: Run `vault auth enable kubernetes` and configure it (`vault write auth/kubernetes/config`)
  with the in-cluster Kubernetes API address and the token-reviewer JWT, per ADR-025 Section 1
  ("Authentication: the Kubernetes auth method"). Determine and document the token-reviewer
  `ServiceAccount` (Vault's own `ServiceAccount`, granted the `system:auth-delegator` `ClusterRole` -
  the standard Vault-Kubernetes-auth pattern) required for Vault to validate tokens presented by other
  pods' `ServiceAccount`s.
- Business Purpose: The Kubernetes auth method is the mechanism ADR-025 selects for every service to
  authenticate to Vault without a shared static credential; it is the prerequisite for every
  per-service policy/role in 18.2.2/18.2.3.
- Inputs: ADR-025 Section 1, `deploy/helm/telco-service/templates/serviceaccount.yaml` (the per-service
  `ServiceAccount` this auth method binds to), HashiCorp's Kubernetes auth method documentation
  (external reference, standard `system:auth-delegator` `ClusterRoleBinding` pattern).
- Outputs: A documented `vault auth enable kubernetes` + `vault write auth/kubernetes/config` procedure
  (added to `deploy/RUNBOOK.md`'s Vault section from 18.1.2), a `ClusterRoleBinding` granting Vault's
  `ServiceAccount` `system:auth-delegator` (as part of `deploy/helm/vault/values.yaml`'s
  `server.serviceAccount` config or a companion template).
- Acceptance Criteria:
  - `vault auth list` shows `kubernetes/` enabled.
  - `vault read auth/kubernetes/config` returns a valid `kubernetes_host` pointing at the in-cluster
    API.
- Dependencies: 18.1.2 (Vault must be initialized/unsealed first)
- Complexity: S

#### 18.2.2 Per-service Vault KV v2 policies
- ID: 18.2.2
- Title: Author one least-privilege Vault policy per service, scoped to `secret/data/<service>/*`
- Description: Enable the KV v2 secrets engine at `secret/` and write one Vault policy per service that
  actually consumes a secret today (cross-referenced against `deploy/helm/README.md`'s "Config / Secret
  model" secret-key table - not every service reads every key, e.g. only `customer-service` reads
  `CUSTOMER_AES_KEY`), each granting `read` (and `list` where needed for KV v2 metadata) on exactly that
  service's own path (`secret/data/<service>/*`) and nothing else, per ADR-025 Section 1's "no service
  can read another service's secrets, now enforced by Vault policy instead of by convention."
- Business Purpose: Delivers the least-privilege guarantee ADR-025 names as a core Positive consequence
  - Vault-enforced per-service secret isolation, not just conventional isolation.
- Inputs: ADR-025 Section 1 and Section 2 (per-item ownership table), `deploy/helm/README.md` "Config /
  Secret model" secret-key table (which service reads which key today).
- Outputs: One policy document per service (`vault policy write <service> <path>.hcl`, e.g. under
  `deploy/helm/vault/policies/<service>.hcl`), a service -> Vault path -> policy name mapping table
  added to the RUNBOOK's Vault section.
- Acceptance Criteria:
  - Each policy grants `read`/`list` on only that service's own `secret/data/<service>/*` path.
  - `vault policy read <service>` for every service returns the scoped policy; none grants access to
    another service's path.
- Dependencies: 18.2.1
- Complexity: M

#### 18.2.3 Per-service Kubernetes auth roles binding ServiceAccount to policy
- ID: 18.2.3
- Title: Bind each service's existing `ServiceAccount` to its Vault policy via a Kubernetes auth role
- Description: For each service, `vault write auth/kubernetes/role/<service>` binding
  `bound_service_account_names=<service>`, `bound_service_account_namespaces=telco` (matching the
  `ServiceAccount` name/namespace the `telco-service` chart's `serviceaccount.yaml` template already
  provisions, per ADR-025 Section 1), `policies=<service>` (from 18.2.2), and a short TTL consistent
  with least-privilege token lifetime.
- Business Purpose: Completes the Kubernetes-auth-method chain (auth method 18.2.1 -> policy 18.2.2 ->
  role 18.2.3) so a pod running under, e.g., the `customer-service` `ServiceAccount` can log in to
  Vault and receive only the `customer-service` policy's access - zero new credential to distribute,
  since the pod's own projected `ServiceAccount` token is the credential.
- Inputs: 18.2.1, 18.2.2, `deploy/helm/telco-service/templates/serviceaccount.yaml`
  (`{{ include "telco-service.serviceAccountName" . }}` naming convention).
- Outputs: One `vault write auth/kubernetes/role/<service> ...` command per service (documented in the
  RUNBOOK's Vault section), a verification snippet showing a successful `vault login` using a service's
  projected token.
- Acceptance Criteria:
  - `vault read auth/kubernetes/role/customer-service` shows
    `bound_service_account_names=[customer-service]`, `bound_service_account_namespaces=[telco]`,
    `policies=[customer-service]`.
  - A manual login test (`vault write auth/kubernetes/login role=customer-service jwt=<projected-token>`)
    from inside a running pod using that `ServiceAccount` succeeds and returns a token scoped to the
    `customer-service` policy only.
- Dependencies: 18.2.2
- Complexity: M

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met.

## Deliverables

- Vault Kubernetes auth method enabled and configured against the in-cluster API server.
- One Vault policy per service, scoped to that service's own KV path only.
- One Vault Kubernetes auth role per service, bound to that service's existing `ServiceAccount`.

## Definition of Done

- All subtasks (18.2.1, 18.2.2, 18.2.3) completed and their acceptance criteria verified.
- All listed dependencies satisfied.
- The auth method, policies, and roles are live-verified on the Sprint 15 Kind cluster (a real
  `vault login` from a live pod succeeds and returns the correctly scoped token) per the global
  conventions in [../README.md](../README.md) (Section 3).
