# 23.2 Rule Evaluation Engine (RAPID_SIM_SWAP, MSISDN_CHURN_VELOCITY, SUSPEND_REACTIVATE_VELOCITY)

> Sprint: [Sprint 23 - SIM-Swap / Fraud Detection](./README.md)
> Feature ID: 23.2
> Subtasks: 23.2.1, 23.2.2, 23.2.3, 23.2.4, 23.2.5

## Objective

Consume subscription-service's existing MSISDN/subscription lifecycle events via the inbox, evaluate
the three ADR-029 rules against the rolling `MsisdnLifecycleSignal` log, raise `FraudSignal` rows, and
escalate repeated/high-severity signals into a `FraudCase`. This is the core detection logic of Sprint
23. **No response to a raised signal or opened case ever automatically suspends a subscription or
calls subscription-service's write API** - that constraint (ADR-029 Section 5, design-note.md Section
5) is a hard acceptance criterion of this feature, not an implementation detail left to later
discretion.

## Dependencies

- 23.1 (fraud-service scaffold, `fraud-db` schema, aggregates/repositories).
- ADR-029 Section 4 (rule definitions) and Section 5 (detect-and-alert-only response model - **no
  automated hold**).
- [design-note.md](design-note.md) Section 4 (rule definitions and default thresholds/windows),
  Section 5 (response model), Section 7 (consumed event list).
- Sprint 09 subscription-service (DONE) - publishes `msisdn.allocated.v1`, `msisdn.released.v1`,
  `subscription.activated.v1`, `subscription.suspended.v1` (`docs/api-contracts/subscription-service.md`).
- Reference implementation shape: `microservices/usage-service/src/main/java/com/telco/usage/application/consumer/`
  (inbox event consumer pattern) and `.../application/handler/` (threshold-evaluation command handler
  pattern) - structurally the closest existing analogue named in design-note.md Section 3.

## Subtasks

#### 23.2.1 Lifecycle event ingestion via inbox
- ID: 23.2.1
- Title: Consume `msisdn.allocated.v1`, `msisdn.released.v1`, `subscription.activated.v1`,
  `subscription.suspended.v1` idempotently and append to `MsisdnLifecycleSignal`
- Description: Add four inbox-backed event consumers (one per consumed event type, following
  `usage-service`'s `application/consumer/*EventConsumer.java` naming pattern:
  `MsisdnAllocatedEventConsumer`, `MsisdnReleasedEventConsumer`, `SubscriptionActivatedEventConsumer`,
  `SubscriptionSuspendedEventConsumer`) that deserialize the Avro event, use `InboxService.firstSeen`
  (`starter-inbox`) to guarantee idempotent processing, and insert one `MsisdnLifecycleSignal` row per
  event (event_type mapped from the Avro event name, customerId/msisdn/subscriptionId/occurredAt
  extracted from the payload, ingestedAt = processing time). This subtask ONLY appends to the raw
  signal log; it does not evaluate any rule (23.2.2-23.2.4 read this log).
- Business Purpose: Builds the rolling-window event log the velocity rules query against, without
  fraud-service ever reading `subscription-db` directly (ADR-006, ADR-029 Section 1).
- Inputs: `docs/api-contracts/subscription-service.md` (Events table), design-note.md Section 7
  (consumed event list), `com.telco.platform.inbox.InboxService`/`InboxBehavior`,
  `usage-service/src/main/java/com/telco/usage/application/consumer/SubscriptionActivatedEventConsumer.java`
  (reference).
- Outputs: `microservices/fraud-service/src/main/java/com/telco/fraud/application/consumer/MsisdnAllocatedEventConsumer.java`,
  `.../application/consumer/MsisdnReleasedEventConsumer.java`,
  `.../application/consumer/SubscriptionActivatedEventConsumer.java`,
  `.../application/consumer/SubscriptionSuspendedEventConsumer.java`,
  `.../config/KafkaConsumerConfig.java` (topic/consumer-group wiring, mirroring
  `usage-service/src/main/java/com/telco/usage/config/KafkaConsumerConfig.java`).
- Acceptance Criteria:
  - Publishing `msisdn.allocated.v1`/`msisdn.released.v1`/`subscription.activated.v1`/
    `subscription.suspended.v1` on the topics subscription-service already produces to results in a
    corresponding `MsisdnLifecycleSignal` row in `fraud-db` within the consumer's normal processing
    latency.
  - Replaying the same event (same Kafka key/event id) does not create a duplicate
    `MsisdnLifecycleSignal` row (inbox idempotency verified, ADR-019/ARC-05).
  - fraud-service requires no new subscription-service event and never queries `subscription-db`
    directly (ADR-029 Exit Criteria bullet 2).
- Dependencies: 23.1.3
- Complexity: M

#### 23.2.2 RAPID_SIM_SWAP rule evaluator
- ID: 23.2.2
- Title: Detect an `msisdn.released.v1` followed by an `msisdn.allocated.v1` for the same MSISDN,
  reassigned to a different SimCard/subscription, within the rule's window
- Description: On ingesting an `msisdn.allocated.v1` signal (23.2.1), trigger a command
  (`EvaluateRapidSimSwapCommand` -> `EvaluateRapidSimSwapCommandHandler`, mediator-dispatched per
  ADR-008/ARC-03) that looks up the active `RAPID_SIM_SWAP` `FraudRule` (window_minutes, default 15
  per ADR-029 Section 4) and queries `MsisdnLifecycleSignalRepository` for a prior
  `MSISDN_RELEASED` signal for the same MSISDN within that window, where the new allocation's
  subscriptionId/SimCard differs from the released one's. On a match, create a `FraudSignal` row
  (ruleCode=`RAPID_SIM_SWAP`, severity=HIGH per the rule's configured severity,
  sourceSignalIds=[released signal id, allocated signal id]) and publish `fraud.signal-raised.v1` via
  the outbox (event contract delivered in 23.4; this subtask calls `OutboxService.publish(...)` with
  the event type name, actual Avro payload class exists once 23.4 lands - coordinate ordering with
  23.4 or stub the payload shape per design-note.md Section 7 if built in parallel). If the rule is
  `enabled=false`, no signal is raised.
- Business Purpose: Implements ADR-029's primary detection scenario - the sprint's Exit Criteria
  bullet 1 ("a simulated rapid MSISDN release-then-reallocate for the same customer raises a
  `RAPID_SIM_SWAP` signal").
- Inputs: ADR-029 Section 4 item 1, design-note.md Section 4 item 1, 23.1.3 aggregates/repositories.
- Outputs: `microservices/fraud-service/src/main/java/com/telco/fraud/application/command/EvaluateRapidSimSwapCommand.java`,
  `.../application/handler/EvaluateRapidSimSwapCommandHandler.java`.
- Acceptance Criteria:
  - An `msisdn.released.v1` for MSISDN X followed within 15 minutes (or the configured
    `windowMinutes`) by an `msisdn.allocated.v1` for MSISDN X on a different subscription/SimCard
    raises exactly one `FraudSignal` with `ruleCode=RAPID_SIM_SWAP`.
  - The same allocate/release pair outside the configured window, or reassigned to the *same*
    subscription/SimCard, does NOT raise a signal.
  - Raising a `RAPID_SIM_SWAP` signal performs no write, call, or command against
    subscription-service, and does not suspend or hold the subscription in any way (ADR-029 Section
    5 - detect and alert only).
- Dependencies: 23.2.1
- Complexity: L

#### 23.2.3 MSISDN_CHURN_VELOCITY rule evaluator
- ID: 23.2.3
- Title: Detect more than N allocate/release cycles for the same customer within a rolling 24-hour
  window
- Description: On ingesting any `MsisdnLifecycleSignal` of type `MSISDN_ALLOCATED` or
  `MSISDN_RELEASED` (23.2.1), trigger `EvaluateMsisdnChurnVelocityCommand` ->
  `EvaluateMsisdnChurnVelocityCommandHandler` that looks up the `MSISDN_CHURN_VELOCITY` `FraudRule`
  (window_minutes default 1440 / 24h, threshold_count default 3 per ADR-029 Section 4), counts
  allocate+release signals for the same `customerId` within that rolling window via
  `MsisdnLifecycleSignalRepository`, and, if the count exceeds `thresholdCount`, creates a
  `FraudSignal` (ruleCode=`MSISDN_CHURN_VELOCITY`, severity=MEDIUM per the rule's configured severity,
  sourceSignalIds = the contributing signal ids) and publishes `fraud.signal-raised.v1` via the
  outbox. Re-evaluate on every new qualifying signal, but do not raise a duplicate signal for a
  customer already over threshold within the same window (avoid a signal-per-event flood - one
  `FraudSignal` per distinct threshold breach; a fresh breach after the customer's count later drops
  and re-crosses the threshold within a new window is allowed).
- Business Purpose: Implements the sprint's second-priority rule from ADR-029 Section 4 item 2 -
  catches slower-moving MSISDN churn abuse that `RAPID_SIM_SWAP`'s short window would miss.
- Inputs: ADR-029 Section 4 item 2, design-note.md Section 4 item 2, 23.1.3 aggregates/repositories.
- Outputs: `microservices/fraud-service/src/main/java/com/telco/fraud/application/command/EvaluateMsisdnChurnVelocityCommand.java`,
  `.../application/handler/EvaluateMsisdnChurnVelocityCommandHandler.java`.
- Acceptance Criteria:
  - A customer with 4 (> default threshold 3) allocate/release cycles within a rolling 24-hour window
    raises exactly one `FraudSignal` with `ruleCode=MSISDN_CHURN_VELOCITY`.
  - The same customer with 3 or fewer cycles in the window does NOT raise a signal.
  - Raising this signal performs no write, call, or command against subscription-service (ADR-029
    Section 5).
- Dependencies: 23.2.1
- Complexity: M

#### 23.2.4 SUSPEND_REACTIVATE_VELOCITY rule evaluator (second priority)
- ID: 23.2.4
- Title: Detect unusual `subscription.suspended.v1` / `subscription.activated.v1` cycling for the
  same subscription within a short window
- Description: On ingesting a `SUBSCRIPTION_ACTIVATED` or `SUBSCRIPTION_SUSPENDED` signal (23.2.1),
  trigger `EvaluateSuspendReactivateVelocityCommand` -> its handler, which looks up the
  `SUSPEND_REACTIVATE_VELOCITY` `FraudRule` (window_minutes and threshold_count as seeded in 23.1.2)
  and counts suspend/reactivate transitions for the same `subscriptionId` within the window; on
  breach, creates a `FraudSignal` (ruleCode=`SUSPEND_REACTIVATE_VELOCITY`, severity per the rule's
  configured LOW/MEDIUM value) and publishes `fraud.signal-raised.v1`. Per ADR-029 Section 4 and
  design-note.md Section 4 item 3, this rule is explicitly **second priority - include if
  straightforward**; if implementation surfaces unexpected complexity (for example, distinguishing
  legitimate non-payment suspend/reactivate cycles from abuse), it may be deferred to a follow-up
  sprint without blocking 23.2's other subtasks or the sprint's Exit Criteria, none of which name this
  rule explicitly.
- Business Purpose: Covers the secondary SIM-swap/account-takeover pattern named in ADR-029 - rapid
  suspend/reactivate cycling that RAPID_SIM_SWAP and MSISDN_CHURN_VELOCITY do not observe.
- Inputs: ADR-029 Section 4 item 3, design-note.md Section 4 item 3, 23.1.3 aggregates/repositories.
- Outputs: `microservices/fraud-service/src/main/java/com/telco/fraud/application/command/EvaluateSuspendReactivateVelocityCommand.java`,
  `.../application/handler/EvaluateSuspendReactivateVelocityCommandHandler.java`.
- Acceptance Criteria:
  - A subscription with suspend/reactivate cycling exceeding the seeded threshold within the
    configured window raises exactly one `FraudSignal` with `ruleCode=SUSPEND_REACTIVATE_VELOCITY`.
  - Cycling within normal bounds (at or below threshold) does NOT raise a signal.
  - Raising this signal performs no write, call, or command against subscription-service, and in
    particular never itself re-suspends or re-activates the subscription (ADR-029 Section 5).
- Dependencies: 23.2.1
- Complexity: M

#### 23.2.5 FraudCase escalation
- ID: 23.2.5
- Title: Escalate repeated or high-severity `FraudSignal`s for the same customer into a `FraudCase`
- Description: After any of 23.2.2/23.2.3/23.2.4 creates a `FraudSignal`, evaluate escalation via
  `EscalateFraudCaseCommand` -> its handler: if the signal's severity is HIGH (always escalates on
  first occurrence, matching `RAPID_SIM_SWAP`'s default severity), or if the customer has accumulated
  more than one open-window signal (severity MEDIUM/LOW repeated), open a new `FraudCase`
  (status=OPEN, signalIds including the triggering and any related prior signals for that customer) if
  one is not already OPEN/UNDER_REVIEW for that customer, and publish `fraud.case-opened.v1` via the
  outbox (event contract delivered in 23.4). If an OPEN/UNDER_REVIEW `FraudCase` already exists for
  the customer, attach the new signal to it instead of opening a duplicate case. **This handler MUST
  NOT call subscription-service, MUST NOT invoke any suspend/hold operation, and MUST NOT alter
  subscription state in any way** - opening a `FraudCase` is purely a detect-and-alert action per
  ADR-029 Section 5; the only sanctioned downstream effect is 23.4's ticket-service auto-ticket for
  human agent review, and any subscription suspension remains a manual, out-of-band agent action via
  subscription-service's existing `POST /api/v1/subscriptions/{id}/suspend` endpoint.
- Business Purpose: Turns individual rule hits into an actionable unit of work (a case) without
  fraud-service reinventing case-management workflow, per ADR-029 Section 5 / design-note.md Section
  5's reuse-ticket-service-SLA-machinery decision (23.4).
- Inputs: ADR-029 Section 5, design-note.md Section 5-6, 23.2.2/23.2.3/23.2.4 outputs.
- Outputs: `microservices/fraud-service/src/main/java/com/telco/fraud/application/command/EscalateFraudCaseCommand.java`,
  `.../application/handler/EscalateFraudCaseCommandHandler.java`.
- Acceptance Criteria:
  - A HIGH-severity `RAPID_SIM_SWAP` signal opens a `FraudCase` (status=OPEN) on first occurrence for
    that customer (sprint Exit Criteria bullet 1: "...and, on repetition/severity, opens a
    `FraudCase`").
  - A second qualifying signal for a customer who already has an OPEN/UNDER_REVIEW `FraudCase`
    attaches to the existing case rather than opening a duplicate.
  - **No code path reachable from `EscalateFraudCaseCommandHandler` calls subscription-service or any
    endpoint that suspends, holds, or otherwise mutates a subscription - verified by an explicit test
    asserting zero outbound HTTP/Feign calls to subscription-service from this handler (see 23.5).**
    This is the concrete, testable form of the sprint's Exit Criteria bullet 3 and ADR-029 Section 5.
- Dependencies: 23.2.2, 23.2.3, 23.2.4
- Complexity: M

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met. In addition, at the feature level:

- ADR-029 Section 5's detect-and-alert-only response model holds across all three rules and the
  escalation path: no automated subscription suspension, hold, or any subscription-service mutation
  is ever triggered by fraud-service's rule evaluation or case-escalation logic.
- fraud-service consumes only the four existing subscription-service events named in
  design-note.md Section 7 - no new subscription-service event is required (sprint Exit Criteria
  bullet 2).

## Deliverables

- Idempotent inbox consumption of `msisdn.allocated.v1`, `msisdn.released.v1`,
  `subscription.activated.v1`, `subscription.suspended.v1` into `MsisdnLifecycleSignal`.
- Three rule evaluators (`RAPID_SIM_SWAP`, `MSISDN_CHURN_VELOCITY`, `SUSPEND_REACTIVATE_VELOCITY`)
  raising `FraudSignal` rows against admin-configurable thresholds.
- `FraudCase` escalation logic, detect-and-alert only, with no automated suspension.
