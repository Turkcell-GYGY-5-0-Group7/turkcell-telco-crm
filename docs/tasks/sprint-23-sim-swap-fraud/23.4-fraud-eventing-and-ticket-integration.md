# 23.4 Fraud Eventing and ticket-service Auto-Ticket Integration

> Sprint: [Sprint 23 - SIM-Swap / Fraud Detection](./README.md)
> Feature ID: 23.4
> Subtasks: 23.4.1, 23.4.2, 23.4.3

## Objective

Define and publish the three new fraud-service events (`fraud.signal-raised.v1`,
`fraud.case-opened.v1`, `fraud.case-resolved.v1`) per ADR-019 event governance, and wire the two
documented consumers: ticket-service auto-opens a linked ticket for agent review (reusing its existing
SLA/assignment machinery, the same reuse pattern as ADR-028's dispute-service integration), and
notification-service raises an internal ops/security alert. **Neither consumer, nor any code in this
feature, triggers a subscription suspension** - the ticket is for human review; ADR-029 Section 5 is
explicit that automated holds are out of scope for this phase.

## Dependencies

- 23.1 (outbox wiring scaffold on fraud-service).
- 23.2 (rule evaluators and `EscalateFraudCaseCommandHandler` - the producers of these events).
- 23.3.2 (`ResolveFraudCaseCommandHandler` - the producer of `fraud.case-resolved.v1`).
- ADR-019 Event Contract and Schema Governance; ADR-009 Event Driven Architecture.
- ADR-029 Section 5 / design-note.md Section 5 and 7 (event list, consumers, no-automated-hold
  constraint).
- ADR-028 Dispute and Chargeback (the precedent for the "consume `<domain>.opened.v1`, auto-open a
  linked ticket, reuse existing SLA/assignment machinery" pattern - `dispute.opened.v1` ->
  ticket-service, mirrored here for `fraud.case-opened.v1`).
- Sprint 12 ticket-service (DONE) - `Ticket`/`TicketComment`/`SlaPolicy` aggregates,
  `OpenTicketCommand`/`OpenTicketCommandHandler`, `docs/api-contracts/ticket-service.md`.
- Sprint 12 notification-service (DONE) - existing event-triggered notification pattern.
- `event-integration` agent - coordinate on Avro schema authoring/Schema Registry registration for
  the three new event types (this feature only specifies the contract; formal schema-registry
  governance sits with that agent per the root `CLAUDE.md` agent table).

## Subtasks

#### 23.4.1 Avro contracts and outbox publish wiring for the three fraud events
- ID: 23.4.1
- Title: Define `fraud.signal-raised.v1`, `fraud.case-opened.v1`, `fraud.case-resolved.v1` and wire
  `OutboxService.publish(...)` calls from 23.2/23.3
- Description: Author the Avro schemas for the three new event types following the platform's
  `domain.event.v1` naming and envelope convention (ADR-019, `EventEnvelope` +
  `com.telco.platform.events.*`), matching `subscription-service`'s existing event-payload classes
  (`application/event/*V1.java`, e.g. `SubscriptionActivatedV1.java`) as the authoring reference.
  Payload fields, minimum: `fraud.signal-raised.v1` (signalId, ruleCode, customerId, msisdn,
  subscriptionId, severity, triggeredAt); `fraud.case-opened.v1` (caseId, customerId, signalIds,
  openedAt, highestSeverity); `fraud.case-resolved.v1` (caseId, customerId, status, resolvedAt,
  resolvedBy). Register the schemas with Schema Registry (coordinate with `event-integration` for the
  registration step and any shared event-contracts module placement) and update
  `docs/architecture/event-catalog.md`'s producer/consumer table to list fraud-service as the
  producer of these three events (a documentation change within this feature's normal scope - not an
  ADR/architecture-owned file). Wire `EvaluateRapidSimSwapCommandHandler`,
  `EvaluateMsisdnChurnVelocityCommandHandler`, `EvaluateSuspendReactivateVelocityCommandHandler`
  (23.2.2-23.2.4) to publish `fraud.signal-raised.v1`, `EscalateFraudCaseCommandHandler` (23.2.5) to
  publish `fraud.case-opened.v1`, and `ResolveFraudCaseCommandHandler` (23.3.2) to publish
  `fraud.case-resolved.v1`, all via `OutboxService.publish(aggregateType, aggregateId, eventType,
  payload)` so the DB write and the outbox row are atomic (never call Kafka directly, per the platform
  capability catalog).
- Business Purpose: Makes fraud-service's detection output consumable by the rest of the platform
  (ticket-service, notification-service) without any direct service-to-service coupling, per the
  event-driven backbone (ADR-009).
- Inputs: ADR-019, design-note.md Section 7, `subscription-service/src/main/java/com/telco/subscription/application/event/`
  (payload-class authoring reference), `docs/architecture/event-catalog.md`.
- Outputs: `microservices/fraud-service/src/main/java/com/telco/fraud/application/event/FraudSignalRaisedV1.java`,
  `.../application/event/FraudCaseOpenedV1.java`, `.../application/event/FraudCaseResolvedV1.java`,
  outbox-publish call sites added to the 23.2/23.3.2 handlers, updated
  `docs/architecture/event-catalog.md` row(s) for the three new events.
- Acceptance Criteria:
  - Each of the three event payloads carries at minimum the fields listed above and validates against
    its registered Avro schema.
  - A `RAPID_SIM_SWAP` rule hit results in a `fraud.signal-raised.v1` row in the outbox table in the
    same DB transaction as the `FraudSignal` insert (atomicity verified by an integration test that
    fails the transaction and asserts neither row persists).
  - `docs/architecture/event-catalog.md` lists fraud-service as producer of all three events.
- Dependencies: 23.2.2, 23.2.3, 23.2.4, 23.2.5, 23.3.2
- Complexity: M

#### 23.4.2 ticket-service auto-ticket consumer for fraud.case-opened.v1
- ID: 23.4.2
- Title: Consume `fraud.case-opened.v1` in ticket-service and auto-open a linked ticket
- Description: In ticket-service, add an inbox-backed consumer (`FraudCaseOpenedEventConsumer`,
  following the naming already used for domain-event consumers elsewhere in the platform, e.g.
  `subscription-service`'s `PaymentCompletedEventConsumer`) that deserializes `fraud.case-opened.v1`
  and dispatches ticket-service's existing `OpenTicketCommand` (`application/command/OpenTicketCommand.java`
  -> `OpenTicketCommandHandler`) to create a `Ticket` linked to the fraud case (carry `caseId` as an
  external-reference field on the ticket, category/type indicating fraud review, e.g. a
  `FRAUD_REVIEW` ticket category or equivalent extension of the existing `Ticket` type taxonomy - do
  not introduce a parallel ticket-creation path; reuse `OpenTicketCommandHandler` so SLA
  auto-assignment (`SlaPolicy`) and existing `ticket.opened.v1`/`ticket.sla-breached.v1` machinery
  apply unchanged, exactly as ADR-028 does for `dispute.opened.v1`). This subtask is a planning
  specification only in this sprint's documentation; the actual ticket-service source change is
  implemented by `domain-engineer`/`event-integration` when this task is picked up - do not implement
  it as part of authoring this task file.
- Business Purpose: Realizes the sprint's core downstream effect - a `FraudCase` becomes actionable
  agent work via ticket-service's existing case-management workflow, without fraud-service
  reinventing ticketing/SLA logic (ADR-029 Section 5, mirroring the ADR-028 dispute-service
  precedent).
- Inputs: `docs/api-contracts/ticket-service.md`, ADR-028 (reuse-pattern precedent),
  `ticket-service/src/main/java/com/telco/ticket/application/command/OpenTicketCommand.java`,
  `ticket-service/src/main/java/com/telco/ticket/domain/Ticket.java`,
  `ticket-service/src/main/java/com/telco/ticket/domain/SlaPolicy.java` (existing aggregates this
  consumer must not duplicate or bypass).
- Outputs (future, when built - not created by this planning task):
  `microservices/ticket-service/src/main/java/com/telco/ticket/application/consumer/FraudCaseOpenedEventConsumer.java`,
  any minimal `Ticket` category/reference-field addition needed to link a ticket back to a `caseId`
  (Flyway migration in ticket-service if a new column is required), updated
  `docs/api-contracts/ticket-service.md` Events table (`fraud.case-opened.v1` added to the Consume
  row).
- Acceptance Criteria:
  - A `fraud.case-opened.v1` event results in exactly one new `Ticket` in ticket-service, SLA-assigned
    through the existing `OpenTicketCommandHandler` path (no parallel/duplicate ticket-creation
    logic), with a retrievable link back to the originating `caseId`.
  - Replaying the same `fraud.case-opened.v1` event does not create a second ticket (inbox
    idempotency).
  - The auto-opened ticket does not, itself or via any downstream effect, call subscription-service
    or suspend the subscription - it is agent-facing work only (ADR-029 Section 5).
- Dependencies: 23.4.1
- Complexity: M

#### 23.4.3 notification-service ops/security alert consumer
- ID: 23.4.3
- Title: Consume `fraud.case-opened.v1` in notification-service for an internal ops/security alert
- Description: In notification-service, add a consumer for `fraud.case-opened.v1` that raises an
  internal ops/security notification (channel per notification-service's existing internal-alert
  pattern, distinct from customer-facing SMS/email) so a security/ops responder is aware a fraud case
  was opened, independent of the ticket-service queue. This subtask is a planning specification only
  in this sprint's documentation; the actual notification-service source change is implemented when
  this task is picked up - do not implement it as part of authoring this task file.
- Business Purpose: Gives operations/security visibility into fraud detections in near-real-time,
  separate from the agent ticket queue, per design-note.md Section 5.
- Inputs: design-note.md Section 5 and 7, existing notification-service event-consumer pattern
  (Sprint 12).
- Outputs (future, when built - not created by this planning task): a
  `FraudCaseOpenedEventConsumer` (or equivalently named) in notification-service's consumer package,
  an internal-alert notification template.
- Acceptance Criteria:
  - A `fraud.case-opened.v1` event results in exactly one internal ops/security alert notification.
  - This alert is informational only - it triggers no subscription-service call and no automated
    suspension (ADR-029 Section 5).
- Dependencies: 23.4.1
- Complexity: S

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met. In addition, at the feature level:

- `fraud.signal-raised.v1`, `fraud.case-opened.v1`, and `fraud.case-resolved.v1` are published only
  via the outbox (never a direct Kafka call), atomically with the triggering DB write.
- Every consumer of these events (ticket-service, notification-service) is informational/agent-facing
  only; none suspends, holds, or otherwise mutates a subscription - the sprint's detect-and-alert-only
  constraint (ADR-029 Section 5) holds end-to-end through this feature.

## Deliverables

- Three new Avro-defined, schema-registered fraud-service events, published via the outbox.
- ticket-service auto-ticket consumer for `fraud.case-opened.v1`, reusing existing SLA/assignment
  machinery (no parallel ticketing logic).
- notification-service internal ops/security alert consumer for `fraud.case-opened.v1`.
