# 23.5 Tests (Unit, Integration, Cross-Service Acceptance)

> Sprint: [Sprint 23 - SIM-Swap / Fraud Detection](./README.md)
> Feature ID: 23.5
> Subtasks: 23.5.1, 23.5.2, 23.5.3

## Objective

Prove the sprint's three Exit Criteria objectively: rolling-window rule evaluation is correct at the
boundary, the full inbox-consume -> rule-evaluate -> outbox-publish -> ticket-auto-open chain works
end to end, and - critically - that no code path in this sprint's deliverable ever automatically
suspends a subscription (ADR-029 Section 5). The no-automated-suspension property is treated as a
first-class, explicitly asserted test requirement, not an incidental side effect of other tests
passing.

## Dependencies

- 23.1, 23.2, 23.3, 23.4 (this feature tests the full built sprint; it does not gate their individual
  Definition of Done, each of which already carries its own acceptance criteria, but it is the
  sprint-level verification pass).
- ADR-013 (Testcontainers for integration tests) - see `docs/architecture/platform-capabilities.md`
  Section 3 (`platform-test` Testcontainers fixtures - reuse if available).
- `docs/tasks/README.md` Section 3 (global build/test conventions).
- Sprint 14 testing-and-hardening patterns (acceptance-suite conventions already established for
  cross-service flows, e.g. the onboarding saga acceptance test) as the structural reference for
  23.5.3's cross-service test.

## Subtasks

#### 23.5.1 Rule-evaluator unit tests (rolling-window boundary scenarios)
- ID: 23.5.1
- Title: Unit-test `RAPID_SIM_SWAP`, `MSISDN_CHURN_VELOCITY`, and `SUSPEND_REACTIVATE_VELOCITY`
  evaluators at their window/threshold boundaries
- Description: Add JUnit 5 unit tests (mocked repositories, no Spring context) for
  `EvaluateRapidSimSwapCommandHandler`, `EvaluateMsisdnChurnVelocityCommandHandler`, and
  `EvaluateSuspendReactivateVelocityCommandHandler` (23.2.2-23.2.4) covering: exactly-at-window-edge
  (signal at `windowMinutes` - 1 second vs. + 1 second), exactly-at-threshold vs. one-above/one-below
  threshold, disabled-rule short-circuit (no signal raised when `FraudRule.enabled=false`), and the
  RAPID_SIM_SWAP same-subscription-reassignment exclusion (release+allocate to the *same*
  subscription/SimCard must not raise a signal). Also unit-test
  `EscalateFraudCaseCommandHandler` (23.2.5): HIGH-severity first occurrence opens a case,
  MEDIUM/LOW-severity repetition escalates, and attaching to an existing OPEN/UNDER_REVIEW case
  rather than duplicating.
- Business Purpose: Rule thresholds are hand-tuned per ADR-029's Consequences ("expect a higher
  false-positive/false-negative rate than the eventual ML version") - boundary correctness is the
  primary defense against that risk being worse than intended.
- Inputs: 23.2.2-23.2.5 handler implementations, design-note.md Section 4 (default thresholds/windows
  used as the primary boundary-test fixtures).
- Outputs: `microservices/fraud-service/src/test/java/com/telco/fraud/application/handler/EvaluateRapidSimSwapCommandHandlerTest.java`,
  `.../application/handler/EvaluateMsisdnChurnVelocityCommandHandlerTest.java`,
  `.../application/handler/EvaluateSuspendReactivateVelocityCommandHandlerTest.java`,
  `.../application/handler/EscalateFraudCaseCommandHandlerTest.java`.
- Acceptance Criteria:
  - Each of the four handlers has passing tests for at-threshold, above-threshold, below-threshold,
    and (for the two window-based rules) at-window-edge inside/outside cases.
  - `RAPID_SIM_SWAP`'s same-subscription-reassignment exclusion is explicitly asserted (a swap back to
    the identical subscription/SimCard is not fraud).
  - `EscalateFraudCaseCommandHandler`'s no-duplicate-case-per-customer behavior is explicitly
    asserted.
- Dependencies: 23.2.2, 23.2.3, 23.2.4, 23.2.5
- Complexity: M

#### 23.5.2 Integration tests (Testcontainers - inbox to outbox, and the fraud-case API)
- ID: 23.5.2
- Title: Integration-test the full inbox-consume -> rule-evaluate -> outbox-publish flow and the
  fraud-case/rule-config API, against real PostgreSQL
- Description: Add Spring Boot integration tests using Testcontainers PostgreSQL (ADR-013,
  `platform-test` fixtures if available per the capability catalog) that: (a) publish a Kafka message
  (or invoke the consumer directly against a real inbox-backed transaction) for
  `msisdn.released.v1` followed by `msisdn.allocated.v1` for the same MSISDN within the default
  15-minute window and assert a `FraudSignal` row and a corresponding outbox row for
  `fraud.signal-raised.v1` both exist, in the same transaction; (b) drive enough allocate/release
  events for one customer to cross the `MSISDN_CHURN_VELOCITY` threshold and assert the same; (c)
  exercise `GET /api/v1/fraud-cases`, `GET /api/v1/fraud-cases/{id}`, and
  `POST /api/v1/fraud-cases/{id}/resolve` (23.3.1/23.3.2) end to end against a seeded case; (d)
  exercise `PUT /api/v1/fraud-rules/{code}` (23.3.3) and assert the updated threshold takes effect on
  the next evaluation without a restart.
- Business Purpose: Verifies the atomic DB-write-plus-outbox-row guarantee (ARC-05/NFR-11) and the
  API surface under real persistence, not mocks - the same rigor Sprint 14 applied platform-wide.
- Inputs: 23.1-23.4 full feature set, `docs/architecture/platform-capabilities.md` Section 3
  (`platform-test` Testcontainers fixtures).
- Outputs: `microservices/fraud-service/src/test/java/com/telco/fraud/RuleEvaluationIntegrationTest.java`,
  `.../FraudCaseApiIntegrationTest.java`, `.../FraudRuleConfigApiIntegrationTest.java`.
- Acceptance Criteria:
  - The RAPID_SIM_SWAP integration scenario produces a `FraudSignal` and an outbox row for
    `fraud.signal-raised.v1` in the same DB transaction (verified by asserting both exist after commit
    and neither exists if the test forces a rollback).
  - The MSISDN_CHURN_VELOCITY scenario crossing the seeded threshold (default > 3 in 24h) produces the
    expected signal; one fewer event does not.
  - `POST /api/v1/fraud-cases/{id}/resolve` correctly transitions status and publishes
    `fraud.case-resolved.v1`.
  - `PUT /api/v1/fraud-rules/{code}` changes take effect on the next evaluation within the same test,
    without restarting the Spring context.
- Dependencies: 23.5.1
- Complexity: L

#### 23.5.3 Cross-service acceptance test and no-automated-suspension regression test
- ID: 23.5.3
- Title: End-to-end RAPID_SIM_SWAP-to-auto-ticket acceptance scenario, plus an explicit
  no-automated-suspension assertion
- Description: Following the Sprint 14 cross-service acceptance-suite pattern, add a test (or test
  scenario within the existing live-stack acceptance harness, if one exists per Sprint 14's
  infrastructure) that: (1) simulates the sprint's canonical scenario - `msisdn.released.v1` followed
  by `msisdn.allocated.v1` for the same MSISDN, reassigned to a different subscription, within the
  RAPID_SIM_SWAP window - and asserts, end to end, that fraud-service raises a `FraudSignal`, opens a
  `FraudCase`, and ticket-service (23.4.2) auto-opens a linked `Ticket` visible via
  `GET /api/v1/tickets/{id}`; (2) asserts, as an explicit negative/regression check, that at no point
  in this scenario (or in the MSISDN_CHURN_VELOCITY / SUSPEND_REACTIVATE_VELOCITY / case-resolve
  scenarios from 23.5.1/23.5.2) does subscription-service's `POST /api/v1/subscriptions/{id}/suspend`
  endpoint get called, mocking or spying on that boundary (e.g. WireMock/MockServer standing in for
  subscription-service, or an assertion against subscription-service's own audit log / state showing
  no SUSPENDED transition occurred as a side effect of the fraud scenario) - the direct, executable
  form of the sprint's Exit Criteria bullet 3 and ADR-029 Section 5's central constraint.
- Business Purpose: This is the sprint's most important test - it is the one that would catch a
  regression where a future change accidentally wires fraud detection to an automated suspend call,
  silently reintroducing the ADR-029-Section-5-deferred behavior from ADVANCED.md Section 4.4.
- Inputs: ADR-029 Section 5 (Exit Criteria source), Sprint 23 README Exit Criteria, Sprint 14
  acceptance-suite / live-stack infra as the structural reference,
  `docs/api-contracts/subscription-service.md` (the endpoint that must NOT be called).
- Outputs: `microservices/fraud-service/src/test/java/com/telco/fraud/acceptance/RapidSimSwapToAutoTicketAcceptanceTest.java`
  (or the equivalent location within the shared acceptance-suite module if Sprint 14 established one),
  including the explicit no-automated-suspension assertion described above.
- Acceptance Criteria:
  - The simulated rapid MSISDN release-then-reallocate scenario raises a `RAPID_SIM_SWAP` signal,
    opens a `FraudCase`, and results in a ticket-service `Ticket` linked to that case - matching the
    sprint's Exit Criteria bullet 1 verbatim.
  - The test explicitly asserts zero calls to subscription-service's
    `POST /api/v1/subscriptions/{id}/suspend` (or any subscription-state-mutating endpoint) across
    the full scenario, and fails loudly if that assertion is ever violated.
  - The test explicitly asserts fraud-service made zero direct queries against `subscription-db`
    (ADR-029 Exit Criteria bullet 2 - read-only relative to subscription-service).
- Dependencies: 23.5.2, 23.4.2
- Complexity: L

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met. In addition, at the feature level:

- **The "detect and alert only, no automated subscription suspension" constraint (ADR-029 Section 5)
  is verified by an explicit, named test assertion (23.5.3), not merely implied by the absence of a
  suspend call in the implementation.**
- Test coverage spans unit (rule boundaries), integration (real Postgres, inbox/outbox atomicity, API
  surface), and cross-service acceptance (fraud-service -> ticket-service), matching the layered test
  strategy already established in Sprint 14.

## Deliverables

- Unit tests for all four rule/escalation handlers at their threshold/window boundaries.
- Testcontainers integration tests for the inbox-to-outbox flow and the fraud-case/rule-config API.
- A cross-service acceptance test proving the RAPID_SIM_SWAP -> FraudCase -> auto-ticket chain, and an
  explicit regression test asserting no automated subscription suspension ever occurs.
