# 23.1 fraud-service Scaffold and Schema

> Sprint: [Sprint 23 - SIM-Swap / Fraud Detection](./README.md)
> Feature ID: 23.1
> Subtasks: 23.1.1, 23.1.2, 23.1.3

## Objective

Stand up a new `fraud-service` module (CQRS + Mediator, ADR-004) from the ADR-017 service template,
wired to a new `fraud-db` (database-per-service, ADR-006), with the four aggregates the rest of the
sprint builds on: `MsisdnLifecycleSignal`, `FraudRule`, `FraudSignal`, `FraudCase`. This task delivers
scaffold and schema only - no rule evaluation, no eventing, no API (those are 23.2/23.3/23.4).

## Dependencies

- ADR-029 Fraud Detection MVP Scope (Sections 1-3 - service boundary, architecture mode, data
  ownership).
- [design-note.md](design-note.md) Sections 2-3 and 6 (aggregate shapes) and Section 8 (data
  ownership).
- ADR-017 Service Template Standard.
- `docs/architecture/platform-capabilities.md` Section 2 (New-service checklist).
- Sprint 03 (`microservices/service-template`, platform starters - DONE, unblocks all new services).
- Reference shape: `microservices/usage-service` (closest existing CQRS + Mediator analogue - inbox
  consumer + threshold evaluation + outbox publish) and `microservices/subscription-service` (event
  producer this service consumes from).

## Subtasks

#### 23.1.1 Scaffold fraud-service from the ADR-017 template
- ID: 23.1.1
- Title: Copy the service template into `microservices/fraud-service` and register it in the reactor
  build
- Description: Copy `microservices/service-template` (ADR-017) into `microservices/fraud-service`
  with base package `com.telco.fraud`, `pom.xml` inheriting `platform-bom` (never hardcode a
  dependency version, ADR-003/ADR-020), and register the new module in the parent reactor `pom.xml`.
  Add the mandatory starters (`starter-api`, `starter-security`, `starter-observability`) plus the
  optional starters this service needs per its architecture mode: `starter-mediator` (CQRS +
  Mediator, ADR-004), `starter-outbox` (publishes `fraud.signal-raised.v1` /
  `fraud.case-opened.v1` / `fraud.case-resolved.v1`, delivered in 23.4), `starter-inbox` (consumes
  subscription-service events idempotently, delivered in 23.2). Do NOT depend on `platform-core`
  directly (ADR-018) - starters only. Write `README.md` and `CLAUDE.md` declaring
  `Architecture Mode: CQRS + MEDIATOR` (ADR-004) and `Infrastructure Profile: PostgreSQL (fraud-db,
  primary store, event-emitting service per ADR-006); Redis optional cache for hot per-customer
  velocity counters, explicitly not source of truth` (design-note.md Section 8), proposed port 9013,
  and a bounded-context statement matching ADR-029 Section 1 (read-only relative to
  subscription-service; never accesses `subscription-db` directly, ADR-006).
- Business Purpose: Establishes the service boundary decision already made in ADR-029 (a new,
  lightweight service rather than logic bolted onto subscription-service) as a buildable module, so
  23.2-23.5 have a home.
- Inputs: ADR-017, ADR-029 Section 1-3, `microservices/service-template/`, root `pom.xml`.
- Outputs: `microservices/fraud-service/pom.xml`, `microservices/fraud-service/README.md`,
  `microservices/fraud-service/CLAUDE.md`,
  `microservices/fraud-service/src/main/java/com/telco/fraud/FraudServiceApplication.java`,
  `microservices/fraud-service/src/main/resources/application.yml` (minimal bootstrap only, per the
  ADR-010 pattern used by `subscription-service`/`usage-service` - runtime config stays in
  config-server), parent `pom.xml` module entry.
- Acceptance Criteria:
  - `mvn -pl fraud-service -am compile` succeeds as part of the reactor build.
  - `fraud-service` starts, registers with discovery-server, and reports `UP` on
    `/actuator/health` (no business endpoints required yet).
  - `README.md`/`CLAUDE.md` declare exactly one Architecture Mode (CQRS + MEDIATOR) and the
    Infrastructure Profile, matching the `docs/architecture/service-catalog.md` convention used by
    every other service.
  - `fraud-service` depends only on platform starters, never on `platform-core` modules directly
    (verified by `pom.xml` dependency list, ADR-018).
- Dependencies: none (first task of Sprint 23)
- Complexity: S

#### 23.1.2 fraud-db Flyway schema and default rule seed
- ID: 23.1.2
- Title: Create the `fraud_rule`, `msisdn_lifecycle_signal`, `fraud_signal`, `fraud_case` tables and
  seed the three default `FraudRule` rows
- Description: Add Flyway migrations under `fraud-service/src/main/resources/db/migration`
  (`V1__fraud_schema.sql`, `V2__fraud_rule_seed.sql`, following the `V<n>__<name>.sql` naming already
  used by `subscription-service`) creating the four tables from design-note.md Section 6:
  `fraud_rule` (code PK, window_minutes, threshold_count, severity, enabled), `msisdn_lifecycle_signal`
  (id, event_type, customer_id, msisdn, subscription_id, occurred_at, ingested_at - the raw ingested
  event log rolling-window queries run against), `fraud_signal` (id, rule_code FK, customer_id, msisdn,
  subscription_id, severity, triggered_at, source_signal_ids), `fraud_case` (id, customer_id, status,
  signal_ids, opened_at, resolved_at nullable, resolved_by nullable). `spring.flyway.locations` MUST
  include `classpath:db/migration,classpath:db/migration/platform` so the platform outbox (V900) and
  inbox (V901) tables apply (per the ADR-017 checklist and the pattern documented in
  `subscription-service/CLAUDE.md`). Seed exactly the three `FraudRule` rows from ADR-029 Section 4 /
  design-note.md Section 4: `RAPID_SIM_SWAP` (window_minutes=15, threshold_count=1, severity=HIGH,
  enabled=true), `MSISDN_CHURN_VELOCITY` (window_minutes=1440, threshold_count=3, severity=MEDIUM,
  enabled=true), `SUSPEND_REACTIVATE_VELOCITY` (window_minutes=60, threshold_count=2, severity=LOW,
  enabled=true) - thresholds are admin-tunable at runtime via 23.3's rule-config API without a
  redeploy; only the rule codes themselves are fixed at this MVP stage.
- Business Purpose: Gives the rule-evaluation engine (23.2) and the fraud-case API (23.3) a concrete,
  query-ready schema with sane MVP defaults, avoiding a boolean-expression rule engine
  (explicitly rejected as over-engineering in ADR-029's Alternatives Considered).
- Inputs: design-note.md Section 6 (aggregate shapes) and Section 4 (rule defaults), ADR-029 Section
  4, `subscription-service/src/main/resources/db/migration/V1__subscription.sql` (migration-naming
  reference).
- Outputs: `microservices/fraud-service/src/main/resources/db/migration/V1__fraud_schema.sql`,
  `microservices/fraud-service/src/main/resources/db/migration/V2__fraud_rule_seed.sql`.
- Acceptance Criteria:
  - `fraud-service` starts against a real `fraud-db` PostgreSQL 17 instance with all four migrations
    plus the platform outbox/inbox migrations applying cleanly (Flyway `info` shows no pending/failed
    migrations).
  - `SELECT * FROM fraud_rule` returns exactly the three seeded rows with the ADR-029 default
    thresholds/windows/severities above.
  - No table in `fraud-db` duplicates or reads from `subscription-db` (ADR-006 database-per-service;
    fraud-service is read-only relative to subscription-service, ADR-029 Section 1).
- Dependencies: 23.1.1
- Complexity: M

#### 23.1.3 Domain aggregates and repositories
- ID: 23.1.3
- Title: Implement the `MsisdnLifecycleSignal`, `FraudRule`, `FraudSignal`, `FraudCase` JPA aggregates
  and their Spring Data repositories
- Description: Under `domain/`, implement the four aggregates as JPA entities matching 23.1.2's
  schema exactly (field-for-field, including the `FraudCase.status` enum `OPEN | UNDER_REVIEW |
  CONFIRMED | DISMISSED` from design-note.md Section 6), following the domain-entities-in-JPA
  convention already used by `subscription-service` (`domain/Subscription.java`,
  `domain/MsisdnPool.java`). Under `infrastructure/`, implement the corresponding Spring Data
  repositories (`MsisdnLifecycleSignalRepository`, `FraudRuleRepository`, `FraudSignalRepository`,
  `FraudCaseRepository`) with the rolling-window query methods 23.2's rule evaluators need (for
  example: find `MsisdnLifecycleSignal` rows for a given MSISDN/customerId within an
  `occurredAt` window). No rule-evaluation logic, no command/query handlers, and no controllers in
  this subtask - aggregates and data access only; business logic is delivered by 23.2/23.3.
- Business Purpose: Provides the typed, queryable domain model 23.2 (rule evaluation) and 23.3
  (fraud-case API) build directly on top of, without either of those tasks re-deriving the schema.
- Inputs: design-note.md Section 6 (exact aggregate field lists), 23.1.2's schema,
  `subscription-service/src/main/java/com/telco/subscription/domain/` (entity-authoring reference),
  `subscription-service/src/main/java/com/telco/subscription/infrastructure/` (repository-authoring
  reference).
- Outputs: `microservices/fraud-service/src/main/java/com/telco/fraud/domain/MsisdnLifecycleSignal.java`,
  `.../domain/MsisdnLifecycleEventType.java` (enum: `MSISDN_ALLOCATED | MSISDN_RELEASED |
  SUBSCRIPTION_SUSPENDED | SUBSCRIPTION_ACTIVATED`), `.../domain/FraudRule.java`,
  `.../domain/FraudRuleCode.java` (enum: `RAPID_SIM_SWAP | MSISDN_CHURN_VELOCITY |
  SUSPEND_REACTIVATE_VELOCITY`), `.../domain/FraudSeverity.java` (enum: `LOW | MEDIUM | HIGH`),
  `.../domain/FraudSignal.java`, `.../domain/FraudCase.java`, `.../domain/FraudCaseStatus.java`,
  `.../infrastructure/MsisdnLifecycleSignalRepository.java`,
  `.../infrastructure/FraudRuleRepository.java`, `.../infrastructure/FraudSignalRepository.java`,
  `.../infrastructure/FraudCaseRepository.java`.
  Also, `.../infrastructure/EventEnvelope` note: no Avro payload classes are created here, only when
  the events are defined in 23.4.
- Acceptance Criteria:
  - Every field in design-note.md Section 6's aggregate shapes is represented on the corresponding
    entity, with types matching the 23.1.2 schema (compile-time verified by JPA metamodel /
    integration test loading a seeded row).
  - Repository rolling-window query methods exist for at least: MSISDN-scoped lookback (for
    `RAPID_SIM_SWAP`) and customerId-scoped lookback (for `MSISDN_CHURN_VELOCITY` and
    `SUSPEND_REACTIVATE_VELOCITY`).
  - No command/query handler, controller, or event consumer/publisher exists yet in this subtask
    (scope boundary with 23.2/23.3/23.4).
- Dependencies: 23.1.2
- Complexity: M
