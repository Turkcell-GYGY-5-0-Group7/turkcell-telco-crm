# 23.3 Fraud Case API

> Sprint: [Sprint 23 - SIM-Swap / Fraud Detection](./README.md)
> Feature ID: 23.3
> Subtasks: 23.3.1, 23.3.2, 23.3.3

## Objective

Expose the read/manage surface for `FraudCase`s (list, view, resolve) and an admin endpoint to tune
`FraudRule` thresholds without a redeploy, per `/api/v1` and `ApiResult<T>` conventions (ADR-015).
None of these endpoints ever suspend, hold, or otherwise mutate a subscription; resolving a case is a
fraud-service-local status change only, consistent with ADR-029 Section 5's detect-and-alert-only
model.

## Dependencies

- 23.1 (`FraudRule`/`FraudSignal`/`FraudCase` aggregates and repositories).
- 23.2 (rule evaluation and case escalation - this API reads and resolves cases 23.2 produces).
- ADR-029 Section 4 (admin-configurable rule thresholds) and Section 5 (no automated hold - resolving
  a case must not suspend a subscription).
- ADR-015 API standards (`/api/v1`, `ApiResult<T>`, `PageResult<T>`, `Idempotency-Key`).
- `docs/architecture/platform-capabilities.md` Section 1 (`ApiResult`, `PageResult`,
  `GlobalExceptionHandler`, `PlatformException` subtypes - reuse, do not re-implement).

## Subtasks

#### 23.3.1 Fraud case list/view endpoints
- ID: 23.3.1
- Title: `GET /api/v1/fraud-cases` (paged list) and `GET /api/v1/fraud-cases/{id}` (view with
  signals)
- Description: Implement `GET /api/v1/fraud-cases` returning a `PageResult<FraudCaseSummaryResponse>`
  filterable by `status` and `customerId` (query params), and `GET /api/v1/fraud-cases/{id}` returning
  a `FraudCaseDetailResponse` including the case's linked `FraudSignal`s (via `signalIds`) and, for
  each signal, its contributing `MsisdnLifecycleSignal` ids for audit/investigation traceability
  (design-note.md Section 6). Both dispatch via `Mediator.query(...)` to
  `GetFraudCaseQuery`/`GetFraudCasesQuery` handlers (ARC-03); controllers are thin (ADR-008). Both
  endpoints require a valid JWT with an agent/fraud-analyst role (RBAC) - fraud case data is
  sensitive and not customer-self-service.
- Business Purpose: Gives fraud-review agents the investigation surface referenced by the
  auto-opened ticket (23.4) - the agent's actual review workflow starts here.
- Inputs: 23.1.3 aggregates/repositories, ADR-015, `docs/architecture/platform-capabilities.md`
  Section 1 (`PageResult<T>`, `ApiResult<T>`).
- Outputs: `microservices/fraud-service/src/main/java/com/telco/fraud/api/FraudCaseController.java`,
  `.../application/query/GetFraudCaseQuery.java`, `.../application/query/GetFraudCasesQuery.java`,
  `.../application/handler/GetFraudCaseQueryHandler.java`,
  `.../application/handler/GetFraudCasesQueryHandler.java`,
  `.../application/dto/FraudCaseSummaryResponse.java`, `.../application/dto/FraudCaseDetailResponse.java`,
  `.../application/dto/FraudSignalResponse.java`, `.../api/FraudSecurityConfig.java` (RBAC rule for
  the agent/fraud-analyst role).
- Acceptance Criteria:
  - `GET /api/v1/fraud-cases?status=OPEN` returns only OPEN cases, wrapped in
    `ApiResult<PageResult<FraudCaseSummaryResponse>>`.
  - `GET /api/v1/fraud-cases/{id}` for an unknown id returns `ApiResult.failure` with a 404
    (`ResourceNotFoundException`, reused from platform-common per the capability catalog - not
    hand-rolled).
  - `GET /api/v1/fraud-cases/{id}` for a known id returns the case plus its linked signals.
  - Both endpoints reject a caller without the agent/fraud-analyst role (403).
- Dependencies: 23.1.3
- Complexity: M

#### 23.3.2 Fraud case resolution endpoint
- ID: 23.3.2
- Title: `POST /api/v1/fraud-cases/{id}/resolve`
- Description: Implement the resolve endpoint: request body carries `status` (`CONFIRMED` or
  `DISMISSED`, per the `FraudCase.status` enum from design-note.md Section 6 - `UNDER_REVIEW` is set
  separately, e.g. when an agent starts investigating, if that transition is needed; this subtask
  covers the terminal `CONFIRMED`/`DISMISSED` resolution) and an optional resolution note. The
  handler (`ResolveFraudCaseCommand` -> `ResolveFraudCaseCommandHandler`) sets `resolvedAt`,
  `resolvedBy` (from `CurrentUserProvider`/`UserContextHolder`, platform-common), transitions
  `status`, and publishes `fraud.case-resolved.v1` via the outbox (event contract from 23.4).
  **This endpoint performs a status change on `FraudCase` only. It MUST NOT call
  subscription-service, and MUST NOT itself suspend, hold, or otherwise mutate the subscription in
  any way** - per ADR-029 Section 5, even a `CONFIRMED` fraud case does not trigger an automated
  suspension; the reviewing agent, having confirmed the case through this API, takes any suspension
  action separately and manually via subscription-service's existing
  `POST /api/v1/subscriptions/{id}/suspend` endpoint. This is a hard scope boundary of this subtask,
  not an implementation detail.
- Business Purpose: Closes the fraud-review loop (case opened -> agent investigates via 23.3.1's
  ticket-linked view -> agent resolves via this endpoint) while keeping the actual suspension decision
  a deliberate, auditable, human action, per the ADR-029 rationale (avoiding false-positive-driven
  customer harm).
- Inputs: ADR-029 Section 5, design-note.md Section 5-6, `com.telco.platform.common.context.CurrentUserProvider`.
- Outputs: `microservices/fraud-service/src/main/java/com/telco/fraud/application/command/ResolveFraudCaseCommand.java`,
  `.../application/handler/ResolveFraudCaseCommandHandler.java`,
  `.../application/dto/ResolveFraudCaseRequest.java`, controller method added to
  `FraudCaseController.java` (23.3.1).
- Acceptance Criteria:
  - `POST /api/v1/fraud-cases/{id}/resolve` with `status=CONFIRMED` transitions the case to
    `CONFIRMED`, sets `resolvedAt`/`resolvedBy`, and publishes `fraud.case-resolved.v1`.
  - `POST /api/v1/fraud-cases/{id}/resolve` with `status=DISMISSED` transitions the case to
    `DISMISSED` and publishes `fraud.case-resolved.v1`; no signal or subscription state is altered
    beyond the `FraudCase` row itself.
  - Resolving a case, in either outcome, makes zero outbound calls to subscription-service (verified
    by an explicit test asserting no HTTP/Feign call to subscription-service originates from this
    handler - see 23.5). This is the API-level enforcement of ADR-029 Section 5's "no automated hold"
    constraint.
  - Only a caller with the agent/fraud-analyst role may resolve a case (403 otherwise).
- Dependencies: 23.3.1, 23.2.5 (a `FraudCase` must exist to resolve)
- Complexity: M

#### 23.3.3 Admin rule-threshold configuration API
- ID: 23.3.3
- Title: `GET /api/v1/fraud-rules` and `PUT /api/v1/fraud-rules/{code}`
- Description: Implement `GET /api/v1/fraud-rules` (list all `FraudRule` rows: code, windowMinutes,
  thresholdCount, severity, enabled) and `PUT /api/v1/fraud-rules/{code}` allowing an admin to update
  `windowMinutes`, `thresholdCount`, `severity`, and `enabled` for one of the three fixed rule codes
  (`RAPID_SIM_SWAP`, `MSISDN_CHURN_VELOCITY`, `SUSPEND_REACTIVATE_VELOCITY`) without a redeploy, per
  ADR-029 Section 4 ("Thresholds ... stored per rule code so ops can tune without a redeploy; the rule
  *codes* themselves are fixed at this MVP stage"). `PUT` MUST reject an unknown/new rule code (404 -
  adding a genuinely new rule type remains a code change, not a config change) and MUST require an
  admin role (stricter than the agent/fraud-analyst role on 23.3.1/23.3.2).
- Business Purpose: Lets operations tune false-positive/false-negative rates (ADR-029 Consequences:
  "expect a higher false-positive/false-negative rate than the eventual ML version") without
  redeploying fraud-service, while keeping the rule set itself intentionally fixed and small (ADR-029
  Alternatives Considered - a full rule-expression engine was explicitly rejected).
- Inputs: ADR-029 Section 4, design-note.md Section 4 and Section 6 (`FraudRule` shape).
- Outputs: `microservices/fraud-service/src/main/java/com/telco/fraud/api/FraudRuleController.java`,
  `.../application/query/GetFraudRulesQuery.java`, `.../application/command/UpdateFraudRuleCommand.java`,
  `.../application/handler/GetFraudRulesQueryHandler.java`,
  `.../application/handler/UpdateFraudRuleCommandHandler.java`,
  `.../application/dto/FraudRuleResponse.java`, `.../application/dto/UpdateFraudRuleRequest.java`.
- Acceptance Criteria:
  - `GET /api/v1/fraud-rules` returns all three seeded rules (23.1.2) with their current
    threshold/window/severity/enabled values.
  - `PUT /api/v1/fraud-rules/RAPID_SIM_SWAP` with a new `windowMinutes` takes effect on the next rule
    evaluation (23.2.2) without a service restart.
  - `PUT /api/v1/fraud-rules/{unknownCode}` returns 404.
  - Only a caller with the admin role may call `PUT`; `GET` is available to the agent/fraud-analyst
    role.
- Dependencies: 23.1.3
- Complexity: S

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met. In addition, at the feature level:

- No endpoint in this feature (list, view, resolve, or rule config) ever calls subscription-service
  or otherwise triggers a subscription suspension/hold - resolving a case as `CONFIRMED` is a
  `FraudCase` status change and an event publish only, per ADR-029 Section 5.

## Deliverables

- `GET /api/v1/fraud-cases`, `GET /api/v1/fraud-cases/{id}`, `POST /api/v1/fraud-cases/{id}/resolve`.
- `GET /api/v1/fraud-rules`, `PUT /api/v1/fraud-rules/{code}`.
- RBAC: agent/fraud-analyst role for case read/resolve, admin role for rule-threshold writes.
