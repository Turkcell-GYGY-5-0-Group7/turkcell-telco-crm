## Subtasks

#### 16.4.1 BFF onboarding composition endpoints
- ID: 16.4.1
- Title: Implement `GET /bff/v1/onboarding/catalog` and `POST /bff/v1/onboarding/order`
- Description: Implement the two onboarding endpoints stubbed in 16.1.2. `GET .../catalog` composes
  `GET /api/v1/tariffs` and `GET /api/v1/addons?tariffCode=...` (product-catalog-service) into a
  UI-shaped catalog. `POST .../order` orchestrates, in order: `POST /api/v1/customers` (register,
  customer-service) or reuse of an existing customer, KYC document upload
  (`POST /api/v1/customers/{id}/documents`), and `POST /api/v1/orders` (order-service, with a
  BFF-generated `Idempotency-Key` passed through), returning the order/saga state for the UI to poll
  via `GET /api/v1/orders/{id}`. Payment (`POST /api/v1/payments`) is invoked from the wizard's final
  step, not folded into a single BFF call, so the UI can show a distinct "processing payment" state.
- Business Purpose: Realizes the AC-01 onboarding flow (register -> KYC -> order -> pay -> activate)
  through a single UI-facing composition surface (ADR-022; web-bff contract).
- Inputs: `docs/api-contracts/web-bff.md`, `docs/api-contracts/customer-service.md`,
  `docs/api-contracts/product-catalog-service.md`, `docs/api-contracts/order-service.md`,
  `docs/api-contracts/payment-service.md`, AC-01 (`docs/tasks/README.md` Section 4.3).
- Outputs: `microservices/web-bff/src/main/java/com/telco/webbff/controller/OnboardingController.java`,
  `microservices/web-bff/src/main/java/com/telco/webbff/service/OnboardingCompositionService.java`,
  request/response DTOs under `microservices/web-bff/src/main/java/com/telco/webbff/dto/onboarding/`.
- Acceptance Criteria:
  - `GET /bff/v1/onboarding/catalog` returns tariffs and addons in one call, with no client-side
    fan-out required.
  - `POST /bff/v1/onboarding/order` registers a customer (or reuses one), uploads the KYC document,
    places the order with a mandatory `Idempotency-Key`, and returns an order id and status usable
    for polling; a replayed call with the same key returns the original result (order-service
    idempotency).
  - No domain logic or persistence lives in web-bff (ADR-022) - it only composes and forwards.
- Dependencies: 16.1.2, 16.3.3
- Complexity: L

#### 16.4.2 SvelteKit onboarding wizard UI
- ID: 16.4.2
- Title: Build the multi-step onboarding wizard at `/onboarding`
- Description: Replace the `/onboarding` placeholder (16.2.2) with a multi-step wizard: registration
  form (name, TCKN/VKN, contact), KYC document upload, tariff/addon selection (from
  `GET /bff/v1/onboarding/catalog`), order confirmation, and a payment step that calls
  `POST /api/v1/payments` (via the gateway, since payment is not yet BFF-composed per
  `docs/api-contracts/web-bff.md`) with the order id and a client-generated `Idempotency-Key`. Poll
  order status after payment until the saga reaches a terminal state (activated or compensated).
- Business Purpose: The user-facing half of AC-01; the sprint's primary demo path.
- Inputs: 16.4.1 (BFF endpoints), `docs/api-contracts/payment-service.md`,
  `docs/api-contracts/order-service.md` (saga status semantics).
- Outputs: `frontend/web/src/routes/onboarding/+page.svelte` and step components under
  `frontend/web/src/routes/onboarding/` (e.g. `register.svelte`, `kyc.svelte`, `catalog.svelte`,
  `payment.svelte`, `confirmation.svelte`), `frontend/web/src/lib/onboarding/` state store.
- Acceptance Criteria:
  - A user completes registration, KYC upload, tariff/addon selection, order placement, and payment
    entirely through the UI without leaving `/onboarding`.
  - The final step reflects the subscription's activated state (or a clear failure/compensation
    message) sourced from polling, not an optimistic assumption.
- Dependencies: 16.4.1
- Complexity: L

#### 16.4.3 Saga failure and compensation UX
- ID: 16.4.3
- Title: Surface order/payment failure and compensation outcomes in the wizard
- Description: Handle the order-service compensation path (payment failure -> refund -> order
  CANCELLED) and customer KYC rejection in the UI with actionable error states (retry payment,
  restart KYC), sourced from the order/subscription status the wizard already polls (16.4.2) and
  from `ApiError` responses surfaced through the BFF.
- Business Purpose: A demo-quality UI must show failure paths, not just the happy path (AC-01 covers
  both compensation and success).
- Inputs: `docs/api-contracts/order-service.md` (compensation notes),
  `docs/api-contracts/subscription-service.md` (`subscription.activation-failed.v1`), ADR-015
  (`ApiError` shape).
- Outputs: Error-state components/handling within `frontend/web/src/routes/onboarding/`.
- Acceptance Criteria:
  - A simulated payment failure surfaces a clear "payment failed, order cancelled" state in the UI
    (not a raw stack trace or generic error).
  - A KYC rejection routes the user back to a corrective step rather than a dead end.
- Dependencies: 16.4.2
- Complexity: M
