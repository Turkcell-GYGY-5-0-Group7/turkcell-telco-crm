## Subtasks

#### 16.1.1 Web-bff HTTP client wiring and config bootstrap
- ID: 16.1.1
- Title: Wire web-bff's outbound HTTP client to the API gateway and finish its config-server bootstrap
- Description: Turn the existing `microservices/web-bff` skeleton (`WebBffApplication.java`,
  `application.yml`, `pom.xml`) into a working service: add an outbound HTTP client bean
  (`WebClient`/`RestClient`, or OpenFeign, consistent with ADR-005) configured to call the API
  gateway base URL sourced from config-server, relay the caller's `Authorization` bearer token on
  every outbound call, and expose `/actuator/health` for container/K8s probes per the ADR-014
  pattern established in Sprint 15. web-bff calls only the gateway (`/api/v1/**`); it never calls a
  domain-service port directly (ADR-022, ADR-011 Section 2).
- Business Purpose: Establishes the composition layer's transport so downstream composition
  endpoints (16.4, 16.5) have a single, token-relaying HTTP client to build on (ADR-011 Section 5,
  ADR-022).
- Inputs: ADR-022, ADR-011 Section 5, ADR-005, `docs/api-contracts/web-bff.md`,
  `docs/api-contracts/api-gateway.md`, `microservices/web-bff/` (existing skeleton).
- Outputs: `microservices/web-bff/src/main/java/com/telco/webbff/config/GatewayClientConfig.java`
  (or equivalent client bean), a token-relay interceptor/filter, an `application.yml` gateway
  base-url property, updated `microservices/web-bff/README.md` status.
- Acceptance Criteria:
  - `web-bff` starts, registers with discovery-server, and reports `UP` on `/actuator/health`.
  - A call through the client to a gateway route (e.g. `GET /api/v1/tariffs`) with a bearer token
    succeeds and the token is present on the outbound request (verified by a test or captured log).
  - No hardcoded gateway URL; the base URL is sourced from config-server per the platform convention.
- Dependencies: none (first task of Sprint 16)
- Complexity: M

#### 16.1.2 /bff/v1 controller skeleton and UI-shaped DTOs
- ID: 16.1.2
- Title: Add the `/bff/v1` controller layer and placeholder UI-shaped response DTOs
- Description: Implement the five composition endpoints from `docs/api-contracts/web-bff.md`
  (`GET /bff/v1/home`, `GET /bff/v1/onboarding/catalog`, `POST /bff/v1/onboarding/order`,
  `GET /bff/v1/account`, `GET /bff/v1/invoices`) as controller methods returning UI-shaped DTOs, not
  `ApiResult<T>` - the BFF contract explicitly exempts BFF responses from that wrapper (see
  `docs/api-contracts/web-bff.md` Notes). The composition logic itself is stubbed here and completed
  in 16.4 (onboarding endpoints) and 16.5 (home/account/invoices endpoints); this subtask only
  establishes routes, request validation, and response shapes so the frontend (16.2/16.3) can
  integrate against a stable surface.
- Business Purpose: Gives the frontend a stable, contract-shaped surface to integrate against before
  the full composition logic lands, decoupling frontend and BFF-composition work.
- Inputs: `docs/api-contracts/web-bff.md`, ADR-022, ADR-015 (API standards - the BFF is the
  documented exception to `ApiResult<T>`).
- Outputs: `microservices/web-bff/src/main/java/com/telco/webbff/controller/BffController.java`
  (or one controller per resource area), `microservices/web-bff/src/main/java/com/telco/webbff/dto/`
  (`HomeResponse`, `OnboardingCatalogResponse`, `OnboardingOrderRequest`/`Response`,
  `AccountResponse`, `InvoicesResponse`).
- Acceptance Criteria:
  - Each of the 5 endpoints in the web-bff contract exists, is mapped under `/bff/v1`, requires a
    valid JWT (delegates to `starter-security`), and returns a stub/placeholder body matching the
    documented shape.
  - Springdoc/OpenAPI is exposed for `web-bff` (ARC-08), covering all 5 routes.
- Dependencies: 16.1.1
- Complexity: M

#### 16.1.3 Gateway route and CORS for the web-bff/frontend origin
- ID: 16.1.3
- Title: Add the `/bff/v1/**` gateway route and CORS policy for the SvelteKit origin
- Description: Extend the api-gateway routing table (`docs/api-contracts/api-gateway.md`) with a new
  `/bff/v1/**` -> web-bff (port 9020) entry, and configure CORS to allow the SvelteKit dev origin
  (`http://localhost:3000`, the `telco-web` Keycloak redirect origin). This changes api-gateway
  routing configuration, which is owned outside this sprint; coordinate with `tech-lead`/
  `architecture` before merging the route change.
- Business Purpose: The gateway is the only externally exposed service (ADR-011); the BFF and
  frontend cannot be reached without a routed, CORS-safe path through it.
- Inputs: `docs/api-contracts/api-gateway.md`, `docs/architecture/keycloak-and-auth.md` (telco-web
  client, redirect URI), ADR-011, ADR-022.
- Outputs: api-gateway route configuration update (Spring Cloud Gateway route/CORS config), updated
  `docs/api-contracts/api-gateway.md` routing table row for `/bff/v1/**`.
- Acceptance Criteria:
  - A request to `http://localhost:8080/bff/v1/home` with a valid bearer token reaches `web-bff`
    through the gateway.
  - A CORS preflight (`OPTIONS`) from `http://localhost:3000` to a `/bff/v1/**` route succeeds.
- Dependencies: 16.1.1, 16.1.2
- Complexity: S
