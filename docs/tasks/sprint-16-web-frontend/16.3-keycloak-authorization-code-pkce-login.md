# 16.3 Keycloak Authorization Code + PKCE Login

> Sprint: [Sprint 16 - Web Frontend](./README.md)
> Feature ID: 16.3
> Subtasks: 16.3.1, 16.3.2, 16.3.3

## Objective

Deliver "Keycloak Authorization Code + PKCE Login" within Sprint 16 - Web Frontend.

## Subtasks

#### 16.3.1 PKCE login/callback/logout flow
- ID: 16.3.1
- Title: Integrate `oidc-client-ts` against the `telco-web` Keycloak client
- Description: Implement Authorization Code + PKCE login using `oidc-client-ts` (ADR-022's stated
  library), targeting the `telco-crm` realm's `telco-web` public client (redirect
  `http://localhost:3000/*`, no client secret - the frontend never sees one). Implement `/login`
  (redirect to Keycloak), an OIDC callback route, silent token renewal, and `/logout` (Keycloak
  end-session). Store tokens in memory/session storage, not `localStorage`, to limit XSS exposure.
- Business Purpose: Delivers the sprint's core auth requirement - a real Keycloak login against the
  platform's identity provider (ADR-011, ADR-022; closes the "Web client PKCE login" checklist item
  in `docs/architecture/keycloak-and-auth.md` Section 9).
- Inputs: `docs/architecture/keycloak-and-auth.md`, ADR-011 Section 5, ADR-022,
  `infra/docker/keycloak/realm/realm-export.json` (telco-web client config).
- Outputs: `frontend/web/src/lib/auth/oidc.ts` (client config + login/callback/logout/renew),
  `frontend/web/src/routes/login/+page.svelte` (wired to redirect),
  `frontend/web/src/routes/auth/callback/+page.svelte`.
- Acceptance Criteria:
  - Visiting `/login` redirects to Keycloak; completing login as a seeded realm user returns to the
    app with a valid access + refresh token pair.
  - `/logout` ends the Keycloak SSO session and clears local token state.
  - Silent renewal keeps the session alive past the access-token lifespan (3600s) without a full
    redirect, until the SSO session's idle/max timeout.
- Dependencies: 16.2.2
- Complexity: L

#### 16.3.2 Route guard and protected navigation
- ID: 16.3.2
- Title: Guard `/onboarding`, `/account`, `/invoices` behind an authenticated session
- Description: Add a SvelteKit route guard (layout-level `load` check or hooks) that redirects
  unauthenticated users to `/login` and preserves the originally requested route for post-login
  return.
- Business Purpose: Enforces the exit criterion that onboarding/account flows are only reachable
  after a real Keycloak login (Sprint 16 README exit criteria).
- Inputs: 16.3.1 auth state, `frontend/web/src/routes/*/+page.svelte` from 16.2.2.
- Outputs: `frontend/web/src/hooks.client.ts` (or per-route `+layout.ts` guards).
- Acceptance Criteria:
  - An unauthenticated request to `/account` (or `/onboarding`, `/invoices`) redirects to `/login`
    and returns to the originally requested route after a successful login.
  - An authenticated user reaches all three routes without a redirect loop.
- Dependencies: 16.3.1
- Complexity: M

#### 16.3.3 Bearer-token propagation through the BFF to the gateway
- ID: 16.3.3
- Title: Attach the access token to every BFF call and verify gateway-side validation
- Description: Wire the API client wrapper (16.2.2) to attach `Authorization: Bearer <access_token>`
  to every `/bff/v1/**` call, and confirm web-bff (16.1.1) relays that same token unmodified to the
  gateway, which validates it against the realm JWKS and forwards `X-User-Id`/`X-User-Roles`
  downstream (gateway-behind-trust, ADR-011). Handle `401` responses by triggering silent renewal or
  a redirect to `/login`.
- Business Purpose: Closes the token-authority chain end to end - browser to BFF to gateway to
  services - satisfying ADR-011 Section 2's flow and the Sprint 16 exit criterion that no request
  reaches a domain service without a validated Keycloak token.
- Inputs: 16.1.1 (web-bff token relay), `docs/architecture/keycloak-and-auth.md` Sections 1 and 5.
- Outputs: `frontend/web/src/lib/api/client.ts` update (bearer attach + 401 handling).
- Acceptance Criteria:
  - A logged-in user's call to `GET /bff/v1/account` reaches the gateway with a valid bearer token,
    and the gateway-validated identity headers are visible on the downstream domain-service request
    (verified via a trace/log, not just a `200` response).
  - An expired/invalid token results in either a transparent silent renewal or a clean redirect to
    `/login`, never a raw `401` rendered to the user.
- Dependencies: 16.3.1, 16.1.2
- Complexity: M
