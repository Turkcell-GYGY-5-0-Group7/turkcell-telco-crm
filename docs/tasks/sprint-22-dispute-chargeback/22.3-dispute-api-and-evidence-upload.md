# 22.3 Dispute API and Evidence Upload (MinIO)

> Sprint: [Sprint 22 - Invoice Dispute / Chargeback](./README.md)
> Feature ID: 22.3
> Subtasks: 22.3.1, 22.3.2, 22.3.3

## Objective

Expose the dispute workflow (open, submit evidence, resolve, withdraw, read) as an external
`/api/v1/disputes` surface (ADR-015), backed by 22.2's commands/queries, with evidence binaries stored
in MinIO and only object references persisted in `dispute-db` - mirroring customer-service's KYC
document pattern (design-note.md Section 7, ADR-006).

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- 22.1 (dispute-service scaffold, `dispute-db`)
- 22.2 (commands/handlers this controller dispatches to via the mediator)
- ADR-015 API standards (`ApiResult<T>`, `/api/v1` prefix)
- ADR-011 Security (JWT/RBAC), ADR-021 (audit-mandated, PII masking)
- `docs/architecture/platform-capabilities.md` Section 3 (object storage is a "planned" capability -
  a local MinIO adapter is acceptable now, flagged for later migration to `starter-storage`)

## Subtasks

#### 22.3.1 `/api/v1/disputes` controller and queries
- ID: 22.3.1
- Title: Add `DisputeController` mapping the dispute lifecycle to `/api/v1/disputes`
- Description: Add a thin controller (HTTP -> `Mediator` -> `ApiResult<T>`, ADR-008/ADR-015, no
  business logic) exposing: `POST /api/v1/disputes` (open, dispatches `OpenDisputeCommand`),
  `POST /api/v1/disputes/{id}/evidence` (submit evidence metadata after 22.3.2's upload, dispatches
  `SubmitEvidenceCommand`), `POST /api/v1/disputes/{id}/resolve` (agent resolution; body selects
  `CUSTOMER` or `MERCHANT` outcome, dispatching `ResolveDisputeCustomerCommand` or
  `ResolveDisputeMerchantCommand`), `POST /api/v1/disputes/{id}/withdraw` (dispatches
  `WithdrawDisputeCommand`), `GET /api/v1/disputes/{id}` and `GET /api/v1/disputes?customerId=...`
  (new `GetDisputeQuery`/`GetDisputesByCustomerQuery` + handlers, read-only, `dispute-db` only).
  Request/response DTOs live under `application/dto/`; controller performs validation
  (`@Valid`) only, per the pattern in
  `microservices/payment-service/src/main/java/com/telco/payment/api/PaymentController.java`.
- Business Purpose: Gives customers/agents (via web-bff or a future admin console) and other services
  an externally callable surface for the dispute workflow defined in 22.2, completing the
  ADR-015-compliant edge of dispute-service.
- Inputs: ADR-015, ADR-008, 22.2 commands, `docs/api-contracts/` conventions (new file
  `docs/api-contracts/dispute-service.md` to be authored alongside this task when implemented),
  `microservices/payment-service/.../api/PaymentController.java`,
  `microservices/order-service/.../api/OrderController.java` (thin-controller reference).
- Outputs: `microservices/dispute-service/src/main/java/com/telco/dispute/api/DisputeController.java`,
  `.../application/dto/OpenDisputeRequest.java`, `.../application/dto/ResolveDisputeRequest.java`,
  `.../application/dto/WithdrawDisputeRequest.java`, `.../application/dto/DisputeResponse.java`,
  `.../application/query/GetDisputeQuery.java`, `.../application/query/GetDisputesByCustomerQuery.java`,
  `.../application/handler/GetDisputeQueryHandler.java`,
  `.../application/handler/GetDisputesByCustomerQueryHandler.java`,
  `docs/api-contracts/dispute-service.md` (new contract doc, authored when this task is executed -
  out of scope for this planning pass).
- Acceptance Criteria:
  - All five endpoints exist under `/api/v1/disputes`, are mapped via the `Mediator`, and return
    `ApiResult<T>` (ADR-015) - no `ApiResult`/error type is hand-rolled (reuse per
    `docs/architecture/platform-capabilities.md` Section 1).
  - `POST /api/v1/disputes` with neither `invoiceId` nor `paymentId` returns a 422/`BusinessRuleException`
    response (delegated to the 22.2.1 domain guard, not re-implemented in the controller).
  - `GET /api/v1/disputes/{id}` returns 404 (`ResourceNotFoundException`) for an unknown id.
  - Springdoc/OpenAPI is exposed for dispute-service (ARC-08), covering all five routes.
- Dependencies: 22.2.2
- Complexity: M

#### 22.3.2 MinIO evidence upload and presigned retrieval
- ID: 22.3.2
- Title: Add `DisputeEvidenceStorage` (MinIO adapter) and the evidence upload endpoint
- Description: Add a `POST /api/v1/disputes/{id}/evidence/upload` (multipart) endpoint that stores
  the uploaded file in a new `dispute-evidence` MinIO bucket and returns an `objectRef`, which the
  caller then supplies to 22.3.1's `POST /api/v1/disputes/{id}/evidence` to create the
  `DisputeEvidence` row - or fold both into a single multipart endpoint if simpler; either shape is
  acceptable as long as raw bytes are never persisted in `dispute-db` (design-note.md Section 7,
  ADR-006). Implement `DisputeEvidenceStorage` (interface) and `MinioDisputeEvidenceStorage`
  (implementation) mirroring
  `microservices/customer-service/src/main/java/com/telco/customer/infrastructure/storage/MinioDocumentStorage.java`
  exactly: `store(objectKey, content, contentType)` and `presignedGetUrl(objectKey, ttl)`, both
  guarded by a `minio` Resilience4j circuit breaker/retry, both throwing
  `DependencyFailureException` on MinIO unavailability (do not invent a new exception type - reuse
  per `docs/architecture/platform-capabilities.md` Section 1). Add
  `GET /api/v1/disputes/{id}/evidence/{evidenceId}/download-url` returning a time-limited presigned
  URL, mirroring customer-service's KYC document download pattern.
- Business Purpose: Lets a customer or agent attach proof (chat logs, receipts, PSP correspondence)
  to a dispute without dispute-service's database ever holding binary content, consistent with the
  platform's existing object-storage pattern (reuse-before-build).
- Inputs: design-note.md Section 7,
  `microservices/customer-service/.../infrastructure/storage/MinioDocumentStorage.java`,
  `microservices/customer-service/.../infrastructure/storage/DocumentStorage.java`,
  `microservices/customer-service/.../domain/Document.java` (structural reference),
  `docs/architecture/platform-capabilities.md` Section 3 (object storage flagged as a planned
  `starter-storage` capability - build locally now, isolate for later extraction, same note
  customer-service already carries).
- Outputs: `microservices/dispute-service/src/main/java/com/telco/dispute/infrastructure/storage/DisputeEvidenceStorage.java`,
  `.../infrastructure/storage/MinioDisputeEvidenceStorage.java`,
  `.../infrastructure/config/MinioConfig.java`, updated `DisputeController` (or a dedicated
  `DisputeEvidenceController`) with the upload/download-url endpoints, `application.yml`
  `minio.bucket.dispute-evidence` property.
- Acceptance Criteria:
  - An uploaded file is stored in the `dispute-evidence` MinIO bucket; the `dispute_evidence` row
    persists only `objectRef`/`submittedBy`/`submittedAt` - no binary column exists in the
    22.1.2 migration.
  - `GET .../download-url` returns a presigned URL that expires (time-limited, matching
    customer-service's TTL convention) rather than a permanent public link.
  - A MinIO outage surfaces as `DependencyFailureException` (503-mapped via `GlobalExceptionHandler`),
    not an unhandled exception.
- Dependencies: 22.3.1
- Complexity: M

#### 22.3.3 RBAC, audit logging, and OpenAPI on the dispute endpoints
- ID: 22.3.3
- Title: Apply role-based access control and mandatory audit logging to every state-changing dispute
  endpoint
- Description: Add `@PreAuthorize` rules consistent with ADR-011/ADR-021: `SUBSCRIBER` may open a
  dispute for their own `customerId` (enforced via `UserContextHolder`, not a client-supplied claim),
  submit evidence, and withdraw their own dispute; `ADMIN`/an agent role may resolve
  (`CUSTOMER`/`MERCHANT`) any dispute. Reads (`GET`) are available to `ADMIN` and to the owning
  `SUBSCRIBER`. Wire an `AuditLogWriter` call (added in 22.2.2) on every state-changing endpoint path,
  since dispute-service is audit-mandated (ADR-028 Section 2, ADR-021, NFR-12) - this closes the loop
  between the API layer and the audit rows the domain handlers already write. Add a
  `DisputeSecurityConfig` and `DisputeAccessDeniedAdvice`, mirroring
  `microservices/order-service/src/main/java/com/telco/order/api/OrderSecurityConfig.java` and
  `OrderAccessDeniedAdvice.java`.
- Business Purpose: Prevents a customer from opening or resolving a dispute on someone else's
  invoice/payment and provides the defensible audit trail a financial dispute resolution requires
  (ADR-028 Section 2's audit-mandated designation).
- Inputs: ADR-011, ADR-021, `microservices/order-service/.../api/OrderSecurityConfig.java`,
  `microservices/order-service/.../api/OrderAccessDeniedAdvice.java`,
  `com.telco.platform.common.context.UserContextHolder` (platform-provided, reuse per
  `docs/architecture/platform-capabilities.md` Section 1).
- Outputs: `microservices/dispute-service/src/main/java/com/telco/dispute/api/DisputeSecurityConfig.java`,
  `.../api/DisputeAccessDeniedAdvice.java`, updated `DisputeController`
  `@PreAuthorize` annotations.
- Acceptance Criteria:
  - A `SUBSCRIBER` JWT for customer A cannot open, view, or withdraw a dispute belonging to
    customer B (403, not silently scoped).
  - `POST .../resolve` requires `ADMIN` (or the designated agent role); a `SUBSCRIBER` call is
    rejected with 403.
  - Every successful `POST` on a dispute (open/evidence/resolve/withdraw) produces exactly one
    `audit_log` row with actor, action, and entity id populated.
- Dependencies: 22.3.1
- Complexity: M

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete

- RBAC and mandatory audit logging on all state-changing dispute endpoints.
