# 22.1 dispute-service Scaffold and Schema

> Sprint: [Sprint 22 - Invoice Dispute / Chargeback](./README.md)
> Feature ID: 22.1
> Subtasks: 22.1.1, 22.1.2, 22.1.3

## Objective

Stand up `dispute-service` as a new, independently deployable microservice per the ADR-017 template
and ADR-028's service-boundary decision: its own Maven module, its own `dispute-db`
(database-per-service, ADR-006), and the JPA aggregate skeletons for Dispute, DisputeEvidence, and
DisputeStateHistory - with no state-machine behavior yet (that is 22.2) and no HTTP surface yet (that
is 22.3).

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- [ADR-028 Dispute and Chargeback](../../../architecture/adr/ADR-028-dispute-and-chargeback.md)
  (Status: Proposed - service boundary, Domain Orchestration mode, audit-mandated, `dispute-db`
  ownership)
- [design-note.md](design-note.md) Section 7 (aggregate shapes)
- ADR-017 Service Template Standard, `docs/architecture/platform-capabilities.md` Section 2
  (new-service checklist)
- `microservices/service-template` (copy source), `microservices/reference-service` (worked example)
- None within this sprint (first task of Sprint 22)

## Subtasks

#### 22.1.1 Module scaffold, service identity, and platform starters
- ID: 22.1.1
- Title: Copy the ADR-017 service template into `microservices/dispute-service` and wire platform
  starters
- Description: Copy `microservices/service-template` into a new `microservices/dispute-service`
  Maven module (add it to the root `pom.xml` `<modules>` list), rename the base package to
  `com.telco.dispute`, and add `DisputeServiceApplication.java`. Configure `application.yml` with
  only the `spring.config.import` bootstrap to config-server (runtime config is centralized per the
  platform convention) and the service port `9012` (proposed in ADR-028; confirm with `tech-lead`/
  `architecture` before the port is registered in `docs/architecture/service-catalog.md`). Add a
  `Dockerfile` matching the Sprint 15 container pattern used by the other Domain Orchestration
  services. Declare in `README.md` and `CLAUDE.md`: **Architecture Mode: Domain Orchestration**
  (ADR-004, per ADR-028 Section 2) and **Infrastructure Profile: PostgreSQL (`dispute-db`) + MinIO
  (dispute evidence objects)** (ADR-006, per ADR-028 Section 3). Mandatory starters per ADR-017
  Section 5: `starter-api`, `starter-security`, `starter-observability`; optional starters needed by
  this service's domain: `starter-mediator` (CQRS command/query dispatch, ADR-008),
  `starter-outbox` (dispute-service is an event producer only, per ADR-028 Section 6), `starter-inbox`
  (not required for 22.1 - dispute-service does not consume events itself in this sprint; billing/
  payment/ticket-service are the consumers, scoped in 22.4/22.5/22.6). Do not add a dependency on
  `platform-core` directly (ADR-018).
- Business Purpose: Establishes dispute-service as a real, buildable, deployable unit before any
  business logic lands, so 22.2-22.6 have a stable module to build on and CI can validate the module
  from day one.
- Inputs: ADR-017, ADR-028 Sections 1-3, `docs/architecture/platform-capabilities.md` Section 2,
  `microservices/service-template/`, `microservices/reference-service/` (worked example),
  `microservices/billing-service/README.md` and `microservices/payment-service/README.md` (as
  Domain Orchestration mode examples).
- Outputs: `microservices/dispute-service/pom.xml`, root `pom.xml` module entry,
  `microservices/dispute-service/src/main/java/com/telco/dispute/DisputeServiceApplication.java`,
  `microservices/dispute-service/src/main/resources/application.yml`,
  `microservices/dispute-service/Dockerfile`, `microservices/dispute-service/README.md`,
  `microservices/dispute-service/CLAUDE.md`.
- Acceptance Criteria:
  - `dispute-service` builds as part of the root Maven reactor (`mvn -pl dispute-service -am
    package`) and starts locally, registers with discovery-server, and reports `UP` on
    `/actuator/health`.
  - `README.md`/`CLAUDE.md` explicitly declare Architecture Mode = Domain Orchestration and
    Infrastructure Profile = PostgreSQL + MinIO, matching ADR-028.
  - `dispute-service`'s `pom.xml` inherits `platform-bom` and depends only on platform starters
    (`starter-api`, `starter-security`, `starter-observability`, `starter-mediator`,
    `starter-outbox`) - no direct `platform-core` dependency (ADR-018 verified).
- Dependencies: none (first subtask of Sprint 22)
- Complexity: M

#### 22.1.2 dispute-db schema and Flyway migrations
- ID: 22.1.2
- Title: Create the `dispute-db` PostgreSQL database and Flyway migrations for
  disputes/dispute_evidence/dispute_state_history
- Description: Author the initial Flyway migration(s) under
  `microservices/dispute-service/src/main/resources/db/migration` creating three tables per
  design-note.md Section 7: `disputes` (id, invoice_id nullable, payment_id nullable, customer_id,
  status, reason_code, disputed_amount, resolution_amount nullable, opened_at, resolved_at nullable,
  closed_at nullable), `dispute_evidence` (id, dispute_id FK, submitted_by, object_ref, submitted_at),
  `dispute_state_history` (id, dispute_id FK, from_status, to_status, changed_by, changed_at, note).
  Add an `audit_log` table mirroring the pattern already used by order-service/payment-service
  (`AuditLog.java`/`AuditLogRepository.java`) since dispute-service is audit-mandated (ADR-028
  Section 2, ADR-021, NFR-12). Configure `spring.flyway.locations` to include
  `classpath:db/migration,classpath:db/migration/platform` so the platform outbox table applies
  (`docs/architecture/platform-capabilities.md` Section 2 item 5). This is a **new, independently
  owned database** - dispute-service never gains access to `billing-db` or `payment-db` (ADR-006
  cross-service data rule, ADR-028 Section 3).
- Business Purpose: Gives dispute-service its own system of record for the dispute workflow, isolated
  from billing-service's and payment-service's ledgers, satisfying ADR-006 and the ADR-028 decision
  that dispute-service coordinates via events/REST only.
- Inputs: design-note.md Section 7 (exact aggregate field lists), ADR-006, ADR-028 Section 3,
  `microservices/order-service/src/main/java/com/telco/order/domain/model/AuditLog.java` and
  `microservices/order-service/src/main/java/com/telco/order/infrastructure/persistence/AuditLogRepository.java`
  (pattern reference).
- Outputs:
  `microservices/dispute-service/src/main/resources/db/migration/V1__create_disputes_schema.sql`
  (or split per-table migrations), updated `application.yml` `spring.flyway.locations`.
- Acceptance Criteria:
  - `mvn -pl dispute-service flyway:migrate` (or Testcontainers-driven Flyway on startup) creates
    `disputes`, `dispute_evidence`, `dispute_state_history`, and `audit_log` tables plus the platform
    outbox table with no errors.
  - Table columns match design-note.md Section 7 exactly (including nullability of `invoice_id`/
    `payment_id`/`resolution_amount`/`resolved_at`/`closed_at`).
  - No migration in this task touches `billing-db` or `payment-db`.
- Dependencies: 22.1.1
- Complexity: M

#### 22.1.3 JPA aggregate skeletons and repositories
- ID: 22.1.3
- Title: Add `Dispute`, `DisputeEvidence`, `DisputeStateHistory` JPA entities and their repositories
  (structure only, no state-machine methods)
- Description: Add the three JPA aggregate classes under
  `microservices/dispute-service/src/main/java/com/telco/dispute/domain/` mapping 1:1 to the 22.1.2
  migration, following the field-and-factory-method style already used by
  `microservices/order-service/src/main/java/com/telco/order/domain/model/Order.java` and
  `microservices/payment-service/src/main/java/com/telco/payment/domain/Payment.java` (private
  constructor + static factory, no setters, JPA-only no-arg protected constructor). Add a
  `DisputeStatus` enum with the seven states from ADR-028 Section 4 /
  design-note.md Section 4 (`OPENED, UNDER_REVIEW, EVIDENCE_SUBMITTED, RESOLVED_CUSTOMER,
  RESOLVED_MERCHANT, WITHDRAWN, CLOSED`). This subtask creates the entity shape and a `create()`
  factory only (`OPENED` state); all other transitions and the transition-guard logic are 22.2's
  scope, not this one - keep this task strictly structural. Add
  `DisputeRepository`/`DisputeEvidenceRepository`/`DisputeStateHistoryRepository` (Spring Data JPA).
- Business Purpose: Provides the persistence shape 22.2's state machine and 22.3's API will build on,
  without prematurely encoding transition logic that belongs to 22.2.
- Inputs: design-note.md Section 4 and Section 7, ADR-028 Section 4,
  `microservices/order-service/.../domain/model/Order.java`,
  `microservices/payment-service/.../domain/Payment.java` (structural reference only - do not copy
  business logic).
- Outputs: `microservices/dispute-service/src/main/java/com/telco/dispute/domain/Dispute.java`,
  `.../domain/DisputeEvidence.java`, `.../domain/DisputeStateHistory.java`,
  `.../domain/DisputeStatus.java`,
  `.../infrastructure/persistence/DisputeRepository.java`,
  `.../infrastructure/persistence/DisputeEvidenceRepository.java`,
  `.../infrastructure/persistence/DisputeStateHistoryRepository.java`.
- Acceptance Criteria:
  - `Dispute.create(...)` persists a row in `OPENED` status with `invoiceId` and/or `paymentId` set;
    creating a `Dispute` with neither set throws `BusinessRuleException` (design-note.md Section 4:
    "OPENED requires at least one of invoiceId or paymentId").
  - Entities carry no Spring annotations beyond JPA mapping (domain logic stays framework-free, per
    the pattern in `Payment.java`/`Order.java`).
  - A repository-level test (`@DataJpaTest` or Testcontainers) proves round-trip persistence for all
    three entities.
- Dependencies: 22.1.2
- Complexity: M
