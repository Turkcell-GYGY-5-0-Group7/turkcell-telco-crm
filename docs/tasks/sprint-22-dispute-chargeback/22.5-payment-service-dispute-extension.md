# 22.5 payment-service Dispute Extension

> Sprint: [Sprint 22 - Invoice Dispute / Chargeback](./README.md)
> Feature ID: 22.5
> Subtasks: 22.5.1, 22.5.2, 22.5.3

## Objective

Add a small, payment-service-owned schema and behavior extension - a `disputed` flag on
`Payment`/`PaymentAttempt` - so payment-service can suppress auto-retry/auto-settlement while a
dispute is under review, and reuse its **existing** `POST /api/v1/payments/{id}/refund` capability
(unchanged) to issue a real refund on `dispute.resolved-customer.v1` when the disputed amount was
already paid. This task extends payment-service only; it does not modify dispute-service, and it does
not build a second refund mechanism (ADR-028 Section 5: "Track 2 reuses payment-service's refund
machinery rather than building a parallel one").

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- **22.2 (hard prerequisite):** the exact `dispute.opened.v1` and `dispute.resolved-customer.v1` /
  `dispute.resolved-merchant.v1` payload shapes frozen in 22.2.3 - this task's inbox consumers are
  written against those field names and cannot be finalized before 22.2.3 lands.
- [ADR-028 Dispute and Chargeback](../../../architecture/adr/ADR-028-dispute-and-chargeback.md)
  Sections 3, 5
- [design-note.md](design-note.md) Sections 5, 7
- `microservices/payment-service/` existing `Payment`/`PaymentAttempt`/`RefundPaymentCommand`/
  `RefundPaymentCommandHandler` (the refund capability being reused, not rebuilt)

## Subtasks

#### 22.5.1 `Payment`/`PaymentAttempt` disputed-flag migration
- ID: 22.5.1
- Title: Add a payment-service-owned Flyway migration for a `disputed` flag
- Description: Add a migration under `microservices/payment-service/src/main/resources/db/migration`
  adding a `disputed BOOLEAN NOT NULL DEFAULT FALSE` column to `payments` (the aggregate root - see
  `microservices/payment-service/src/main/java/com/telco/payment/domain/Payment.java`; `PaymentAttempt`
  itself does not need its own flag if retry logic reads the parent `Payment`, but confirm at
  implementation time whether `PaymentRetryScheduler` operates at the attempt or payment level and add
  the column to `payment_attempts` too if the scheduler queries attempts directly). This is
  payment-service's own migration, in payment-service's own `db/migration` directory, under
  payment-service's own write ownership - dispute-service does not author or apply this migration
  (ADR-028 Section 3).
- Business Purpose: Gives payment-service the minimal schema surface needed to represent "this
  payment is under dispute, do not auto-retry/auto-settle it," without dispute-service reaching into
  `payment-db` (ADR-006 cross-service data rule).
- Inputs: design-note.md Section 7 ("a disputed flag on Payment/PaymentAttempt in payment-service"),
  `microservices/payment-service/.../domain/Payment.java`,
  `microservices/payment-service/.../application/scheduler/PaymentRetryScheduler.java`.
- Outputs: `microservices/payment-service/src/main/resources/db/migration/V{n}__add_disputed_flag.sql`,
  updated `Payment.java` (`disputed` field + `markDisputed()`/`clearDisputed()` methods - no setter).
- Acceptance Criteria:
  - Migration applies cleanly against the existing `payment-db` schema; `disputed` defaults to
    `false` for every existing and newly created payment.
  - `Payment.markDisputed()`/`clearDisputed()` are the only way to change the flag (no public setter),
    consistent with the entity's existing `markCompleted()`/`markFailed()`/`markRefunded()` style.
- Dependencies: none within this feature (depends on 22.2.3 for the event shape used in 22.5.2, not
  on this migration)
- Complexity: S

#### 22.5.2 Dispute-opened / dispute-resolved-merchant inbox consumers (hold only, no financial mutation)
- ID: 22.5.2
- Title: Add `DisputeOpenedPaymentConsumer` and `DisputeResolvedMerchantPaymentConsumer`, and suppress
  retry/auto-settlement for disputed payments
- Description: Add two Kafka inbox consumers on the `dispute.events` topic, mirroring the inbox
  pattern used elsewhere in the platform (type-filter on the `eventType` header, `InboxService.firstSeen(...)`
  dedup, dispatch via `Mediator`): `DisputeOpenedPaymentConsumer` consumes `dispute.opened.v1` and, only
  when the event carries a non-null `paymentId` (a payment-originated or already-linked dispute),
  dispatches a new `MarkPaymentDisputedCommand(paymentId)` whose handler calls `Payment.markDisputed()`
  and **performs no PSP call and no state-machine transition** (`PaymentStatus` is untouched);
  `DisputeResolvedMerchantPaymentConsumer` consumes `dispute.resolved-merchant.v1` and dispatches
  `ClearPaymentDisputedCommand(paymentId)`, calling `Payment.clearDisputed()`, again with no PSP call.
  Update `PaymentRetryScheduler` to skip any payment/attempt with `disputed = true`, per design-note.md
  Section 5 ("suppressing auto-retry/auto-settlement while under review"). Both consumers no-op
  silently if `paymentId` is null on the event payload (an invoice-only dispute with no payment
  reference yet).
- Business Purpose: Implements the "provisional" half of the provisional-credit model on the
  payment side: pausing retry/settlement activity on a contested payment without ever calling the PSP
  or mutating `PaymentStatus`, so payment-service remains the uncorrupted system of record for
  completed/refunded payments (ADR-028 Section 5).
- Inputs: 22.2.3 (frozen `dispute.opened.v1`/`dispute.resolved-merchant.v1` shapes), design-note.md
  Section 5, `microservices/billing-service/.../infrastructure/consumer/PaymentCompletedBillingConsumer.java`
  (cross-service inbox-consumer pattern reference),
  `microservices/payment-service/.../application/scheduler/PaymentRetryScheduler.java`.
- Outputs: `microservices/payment-service/src/main/java/com/telco/payment/infrastructure/consumer/DisputeOpenedPaymentConsumer.java`,
  `.../infrastructure/consumer/DisputeResolvedMerchantPaymentConsumer.java`,
  `.../application/command/MarkPaymentDisputedCommand.java`,
  `.../application/command/ClearPaymentDisputedCommand.java`, corresponding handlers, updated
  `PaymentRetryScheduler.java` to filter out `disputed = true` payments.
- Acceptance Criteria:
  - **Provisional-hold invariant (explicit, load-bearing):** processing `dispute.opened.v1` changes
    `Payment.disputed` from `false` to `true` and changes **no other Payment field** - `status`,
    `amount`, `attempts` are bit-for-bit identical before and after, and no `PspAdapter` method is
    invoked. A test MUST assert this by diffing the full `Payment` row and asserting zero interactions
    with `PspAdapter` (mock verification), not merely asserting `disputed` was set.
  - A disputed payment in `FAILED` status is skipped by `PaymentRetryScheduler`'s next run; an
    otherwise-identical non-disputed `FAILED` payment is still retried as today.
  - `dispute.resolved-merchant.v1` restores `disputed = false` and, like `dispute.opened.v1`, triggers
    no PSP call.
  - Both consumers use `InboxService.firstSeen(...)` for idempotent redelivery handling.
- Dependencies: 22.5.1
- Complexity: M

#### 22.5.3 Dispute-resolved-customer inbox consumer: reuse the existing refund flow when already paid
- ID: 22.5.3
- Title: Add `DisputeResolvedCustomerPaymentConsumer` invoking the existing `RefundPaymentCommand`
  when the disputed payment is already `COMPLETED`
- Description: Add a third inbox consumer on `dispute.events`, consuming `dispute.resolved-customer.v1`.
  If the referenced payment's current `status` is `COMPLETED` (the "already paid" branch of ADR-028
  Section 5), dispatch the **existing, unmodified**
  `microservices/payment-service/src/main/java/com/telco/payment/application/command/RefundPaymentCommand.java`
  (`paymentId`, a reason string such as `"dispute:" + disputeId`, and a fresh inbox-safe `messageId`
  derived from the dispute event's own message key, matching the idempotency-key discipline already
  used at every other `RefundPaymentCommand` call site) via the `Mediator` - do **not** write a new
  refund code path; the entire point of this subtask is that
  `RefundPaymentCommandHandler` (COMPLETED -> REFUNDED, PSP mock refund call, existing
  `payment.refunded.v1` publication) runs completely unchanged. If the payment's status is anything
  other than `COMPLETED` (e.g. it was never charged, or billing-service's 22.4.3 unpaid-invoice branch
  already applied), this consumer must explicitly **no-op** (log and return) rather than attempt a
  refund on an ineligible payment - `Payment.markRefunded()`'s existing guard
  ("Only COMPLETED payments can be refunded") is the second line of defense, not the primary check.
- Business Purpose: Implements the real half of the provisional-credit model on the payment side by
  reusing payment-service's existing refund capability end-to-end, exactly as ADR-028 mandates,
  instead of duplicating PSP-refund logic inside a new code path.
- Inputs: 22.2.3 (frozen `dispute.resolved-customer.v1` shape, carrying both `invoiceId` and
  `paymentId`), ADR-028 Section 5, design-note.md Section 5,
  `microservices/payment-service/.../application/command/RefundPaymentCommand.java`,
  `microservices/payment-service/.../application/handler/RefundPaymentCommandHandler.java`
  (unmodified; reused as-is),
  `microservices/payment-service/.../api/PaymentController.java` (`POST /api/v1/payments/{id}/refund`
  - documents the same capability this consumer reuses internally via the mediator, not via HTTP).
- Outputs: `microservices/payment-service/src/main/java/com/telco/payment/infrastructure/consumer/DisputeResolvedCustomerPaymentConsumer.java`.
  No changes to `RefundPaymentCommand.java`, `RefundPaymentCommandHandler.java`, or `PaymentController.java`.
- Acceptance Criteria:
  - **Provisional-hold-vs-real-financial-action invariant (explicit, load-bearing):** a PSP refund
    call and a `PaymentStatus` transition to `REFUNDED` happen, and only happen, in response to
    `dispute.resolved-customer.v1` on a `COMPLETED` payment - no other consumer in 22.5 ever calls
    `PspAdapter.refund(...)` or `Payment.markRefunded()`. A test MUST prove that `dispute.opened.v1`
    and `dispute.resolved-merchant.v1` produce zero `PspAdapter` interactions (cross-referenced with
    22.5.2's acceptance criteria) while `dispute.resolved-customer.v1` on a `COMPLETED` payment is the
    only path that refunds.
  - `dispute.resolved-customer.v1` on a non-`COMPLETED` payment results in zero refund attempts from
    this consumer (billing-service's adjustment path in 22.4.3 owns the unpaid case).
  - `RefundPaymentCommandHandler.java`'s source is unmodified by this subtask (diff-verifiable) -
    confirming the refund mechanism itself is reused, not forked.
  - `payment.refunded.v1` (the existing event, unchanged shape) is published exactly as it is for an
    admin-initiated refund; downstream consumers (order-service's existing
    `PaymentRefundedEventConsumer`, notification-service) require no changes to react to a
    dispute-triggered refund.
  - The consumer is idempotent via `InboxService.firstSeen(...)`; redelivery of the same
    `dispute.resolved-customer.v1` message does not trigger a second refund attempt (and even if it
    did, `Payment.markRefunded()`'s existing REFUNDED-state guard rejects it).
- Dependencies: 22.5.2
- Complexity: M

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met. In addition, at the feature level:

- **Provisional-hold-vs-real-financial-action invariant:** `Payment.disputed = true` (from
  `dispute.opened.v1`) is exclusively a suppression flag - it never calls the PSP and never changes
  `PaymentStatus`; it only pauses `PaymentRetryScheduler`. Only `dispute.resolved-customer.v1`, and
  only when the payment is `COMPLETED`, triggers a real refund - via the pre-existing, unmodified
  `RefundPaymentCommand`/`RefundPaymentCommandHandler`, never a new refund implementation.
  `dispute.resolved-merchant.v1` is always a no-financial-change flag release.

## Deliverables

- `Payment.disputed` flag, via payment-service's own Flyway migration.
- `DisputeOpenedPaymentConsumer`, `DisputeResolvedMerchantPaymentConsumer`,
  `DisputeResolvedCustomerPaymentConsumer` inbox consumers.
- `PaymentRetryScheduler` updated to skip disputed payments.
- Reuse (zero modification) of the existing `POST /api/v1/payments/{id}/refund` /
  `RefundPaymentCommand` capability for the paid-and-uphold case.

## Definition of Done

- All subtasks (22.5.1, 22.5.2, 22.5.3) completed and their acceptance criteria verified.
- All listed dependencies satisfied, including 22.2.3's frozen event contracts.

- `RefundPaymentCommandHandler.java` and `PaymentController.java`'s existing refund endpoint are
  unmodified by this feature.
