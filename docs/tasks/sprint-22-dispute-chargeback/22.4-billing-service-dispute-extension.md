# 22.4 billing-service Dispute Extension

> Sprint: [Sprint 22 - Invoice Dispute / Chargeback](./README.md)
> Feature ID: 22.4
> Subtasks: 22.4.1, 22.4.2, 22.4.3

## Objective

Add a small, billing-service-owned schema and behavior extension - `Invoice.disputeStatus` as a hold
flag, and a `CREDIT`/`ADJUSTMENT` `InvoiceLine` type - so billing-service can react to dispute-service's
events without dispute-service ever writing to `billing-db` (ADR-028 Section 3). This is billing-service's
own feature task, using billing-service's own migrations and its own write ownership; dispute-service
is never modified by this task.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- **22.2 (hard prerequisite):** the exact `dispute.opened.v1` and `dispute.resolved-customer.v1` /
  `dispute.resolved-merchant.v1` payload shapes frozen in 22.2.3 - this task's inbox consumers are
  written against those field names and cannot be finalized before 22.2.3 lands.
- [ADR-028 Dispute and Chargeback](../../../architecture/adr/ADR-028-dispute-and-chargeback.md)
  Sections 3, 5
- [design-note.md](design-note.md) Sections 5, 7
- `microservices/billing-service/` existing `Invoice`/`InvoiceLine`/`BillCycle` aggregates and
  `PaymentCompletedBillingConsumer` (inbox pattern reference)

## Subtasks

#### 22.4.1 `Invoice.disputeStatus` and `CREDIT`/`ADJUSTMENT` `InvoiceLine` type migration
- ID: 22.4.1
- Title: Add a billing-service-owned Flyway migration for the dispute hold flag and adjustment line
  type
- Description: Add a migration under `microservices/billing-service/src/main/resources/db/migration`
  adding a nullable `dispute_status` column to `invoices` (values: `null`/`NONE`, `ON_HOLD`) and
  extending whatever discriminates `InvoiceLine` type today (currently `InvoiceLine` has no explicit
  `type` column - see `microservices/billing-service/src/main/java/com/telco/billing/domain/InvoiceLine.java`;
  add one, e.g. `line_type VARCHAR` defaulting to `USAGE`/`RECURRING` for existing rows, with a new
  `ADJUSTMENT` value) so a dispute credit line is distinguishable from ordinary usage/recurring lines
  in invoice PDFs and reporting. This is billing-service's own migration, in billing-service's own
  `db/migration` directory, under billing-service's own write ownership - dispute-service does not
  author or apply this migration (ADR-028 Section 3).
- Business Purpose: Gives billing-service the minimal schema surface needed to represent "this
  invoice is under dispute" and "this line exists because of a dispute resolution," without
  dispute-service reaching into `billing-db` (ADR-006 cross-service data rule).
- Inputs: design-note.md Section 7 ("Invoice.disputeStatus + a CREDIT/ADJUSTMENT InvoiceLine type"),
  `microservices/billing-service/.../domain/Invoice.java`,
  `microservices/billing-service/.../domain/InvoiceLine.java`.
- Outputs: `microservices/billing-service/src/main/resources/db/migration/V{n}__add_dispute_status_and_adjustment_line_type.sql`,
  updated `Invoice.java` (`disputeStatus` field + `InvoiceDisputeStatus` enum: `NONE`, `ON_HOLD`),
  updated `InvoiceLine.java` (`lineType` field + `InvoiceLineType` enum: existing usage/recurring
  categorization plus `ADJUSTMENT`).
- Acceptance Criteria:
  - Migration applies cleanly against the existing `billing-db` schema (Flyway, no data loss on
    existing invoices/lines - `dispute_status` defaults to `NONE`, `line_type` backfills existing rows
    to a non-`ADJUSTMENT` value).
  - `Invoice.getDisputeStatus()` defaults to `NONE` for every pre-existing and newly created invoice
    that has never been disputed.
- Dependencies: none within this feature (depends on 22.2.3 for the event shape used in 22.4.2, not
  on this migration)
- Complexity: S

#### 22.4.2 Dispute-opened / dispute-resolved-merchant inbox consumers (hold only, no financial mutation)
- ID: 22.4.2
- Title: Add `DisputeOpenedBillingConsumer` and `DisputeResolvedMerchantBillingConsumer`, and exclude
  `ON_HOLD` invoices from dunning
- Description: Add two Kafka inbox consumers on the `dispute.events` topic, mirroring
  `microservices/billing-service/src/main/java/com/telco/billing/infrastructure/consumer/PaymentCompletedBillingConsumer.java`'s
  shape (type-filter on the `eventType` header first, `InboxService.firstSeen(...)` dedup, dispatch a
  command via the `Mediator`): `DisputeOpenedBillingConsumer` consumes `dispute.opened.v1` and
  dispatches a new `PlaceInvoiceOnDisputeHoldCommand(invoiceId)` whose handler sets
  `Invoice.disputeStatus = ON_HOLD` and **does nothing else** - no change to `subTotal`, `tax`,
  `grandTotal`, or `status`; `DisputeResolvedMerchantBillingConsumer` consumes
  `dispute.resolved-merchant.v1` and dispatches `ClearInvoiceDisputeHoldCommand(invoiceId)`, setting
  `disputeStatus` back to `NONE` with, likewise, no financial change (ADR-028 Section 5: "simply
  clears the hold, no financial change"). Update the overdue/dunning check
  (`MarkInvoicesOverdueCommandHandler` / `BillingScheduler`) to skip any invoice with
  `disputeStatus == ON_HOLD`, so `invoice.overdue.v1` is suppressed while a dispute is under review, per
  design-note.md Section 5. Both consumers no-op silently if `invoiceId` is null on the event payload
  (a payment-originated dispute with no invoice reference).
- Business Purpose: Implements the "provisional" half of the provisional-credit model: pausing
  collections on a contested invoice without ever touching billing-service's financial totals, so
  billing-service remains the uncorrupted system of record for invoiced amounts (ADR-028 Section 5).
- Inputs: 22.2.3 (frozen `dispute.opened.v1`/`dispute.resolved-merchant.v1` shapes), design-note.md
  Section 5, `microservices/billing-service/.../infrastructure/consumer/PaymentCompletedBillingConsumer.java`,
  `microservices/billing-service/.../application/handler/MarkInvoicesOverdueCommandHandler.java`,
  `microservices/billing-service/.../infrastructure/config/BillingScheduler.java`.
- Outputs: `microservices/billing-service/src/main/java/com/telco/billing/infrastructure/consumer/DisputeOpenedBillingConsumer.java`,
  `.../infrastructure/consumer/DisputeResolvedMerchantBillingConsumer.java`,
  `.../application/command/PlaceInvoiceOnDisputeHoldCommand.java`,
  `.../application/command/ClearInvoiceDisputeHoldCommand.java`, corresponding handlers, updated
  `MarkInvoicesOverdueCommandHandler.java` (or equivalent dunning-check call site) to filter out
  `ON_HOLD` invoices.
- Acceptance Criteria:
  - **Provisional-hold invariant (explicit, load-bearing):** processing `dispute.opened.v1` changes
    `Invoice.disputeStatus` from `NONE` to `ON_HOLD` and changes **no other Invoice or InvoiceLine
    field** - `grandTotal`/`subTotal`/`tax`/`status` are bit-for-bit identical before and after. A
    test MUST assert this by diffing the full `Invoice` row pre/post consumption, not merely asserting
    `disputeStatus` was set.
  - An `ON_HOLD` invoice is excluded from the next bill-run's overdue/dunning pass even if its
    `dueDate` has passed; a non-`ON_HOLD` invoice past its due date is still marked overdue as today.
  - `dispute.resolved-merchant.v1` restores `disputeStatus = NONE` and, like `dispute.opened.v1`,
    changes no financial field.
  - Both consumers use `InboxService.firstSeen(...)` for idempotent redelivery handling, matching the
    existing `PaymentCompletedBillingConsumer` pattern (no home-grown dedup logic).
- Dependencies: 22.4.1
- Complexity: M

#### 22.4.3 Dispute-resolved-customer inbox consumer: real credit/adjustment when unpaid
- ID: 22.4.3
- Title: Add `DisputeResolvedCustomerBillingConsumer` applying a real `ADJUSTMENT`/`CREDIT`
  `InvoiceLine` when the disputed invoice is still unpaid
- Description: Add a third inbox consumer on `dispute.events`, consuming `dispute.resolved-customer.v1`.
  If the referenced invoice's current `status` is not `PAID` (i.e., the money has not moved yet - the
  "unpaid" branch of ADR-028 Section 5), dispatch a new `ApplyDisputeAdjustmentCommand(invoiceId,
  resolutionAmount)` whose handler adds a **negative** `InvoiceLine` (`lineType = ADJUSTMENT`,
  `unitPrice` negative or `lineTotal` negative depending on the existing `InvoiceLine.of(...)`
  factory's sign convention - keep the invoice's `grandTotal` recomputation consistent with how
  `InvoiceLine.of(...)` already mutates the parent `Invoice`) for `resolutionAmount`, and clears
  `disputeStatus` back to `NONE`. If the invoice is already `PAID`, this consumer must explicitly
  **no-op** (log and return) - the paid case is payment-service's responsibility (22.5.3), and
  billing-service must never issue a second, competing resolution for the same dispute. Use the event
  payload's `invoiceId`/`paymentId` pair (per 22.2.3) to make this branch decision deterministically
  from the event itself, not from a cross-service call.
- Business Purpose: Implements the real half of the provisional-credit model - the only point where
  billing-service actually changes money owed, gated strictly behind the
  `dispute.resolved-customer.v1` event that only `ResolveDisputeCustomerCommand` (22.2.2) can produce.
- Inputs: 22.2.3 (frozen `dispute.resolved-customer.v1` shape, carrying both `invoiceId` and
  `paymentId`), ADR-028 Section 5, design-note.md Section 5,
  `microservices/billing-service/.../domain/InvoiceLine.java` (`InvoiceLine.of(...)` factory and
  sign/rounding convention), `microservices/billing-service/.../domain/Invoice.java`.
- Outputs: `microservices/billing-service/src/main/java/com/telco/billing/infrastructure/consumer/DisputeResolvedCustomerBillingConsumer.java`,
  `.../application/command/ApplyDisputeAdjustmentCommand.java`, corresponding handler.
- Acceptance Criteria:
  - **Provisional-hold-vs-real-financial-action invariant (explicit, load-bearing):** a real
    `InvoiceLine` of type `ADJUSTMENT` is created, and `Invoice.grandTotal` changes, **only** in
    response to `dispute.resolved-customer.v1` on an unpaid invoice - no other consumer in 22.4 ever
    creates an `InvoiceLine` or changes `grandTotal`. A test MUST prove that `dispute.opened.v1` and
    `dispute.resolved-merchant.v1` leave `InvoiceLine` count and `grandTotal` unchanged (cross-referenced
    with 22.4.2's acceptance criteria) while `dispute.resolved-customer.v1` on an unpaid invoice is the
    only path that adds a line and changes the total.
  - `dispute.resolved-customer.v1` on an already-`PAID` invoice results in zero `InvoiceLine` changes
    from this consumer (payment-service's refund path in 22.5.3 owns that case).
  - The consumer is idempotent via `InboxService.firstSeen(...)`; redelivery of the same
    `dispute.resolved-customer.v1` message does not create a second adjustment line.
- Dependencies: 22.4.2
- Complexity: M

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met. In addition, at the feature level:

- **Provisional-hold-vs-real-financial-action invariant:** `Invoice.disputeStatus = ON_HOLD` (from
  `dispute.opened.v1`) is exclusively a hold flag - it never changes `grandTotal`/`subTotal`/`tax`/
  `status`, and it excludes the invoice from dunning without excusing eventual payment. Only
  `dispute.resolved-customer.v1`, and only when the invoice is still unpaid, creates a real
  `ADJUSTMENT`/`CREDIT` `InvoiceLine` and changes `grandTotal`. `dispute.resolved-merchant.v1` is
  always a no-financial-change hold release.

## Deliverables

- `Invoice.disputeStatus` hold flag and `InvoiceLine.lineType` (`ADJUSTMENT`) extension, via
  billing-service's own Flyway migration.
- `DisputeOpenedBillingConsumer`, `DisputeResolvedMerchantBillingConsumer`,
  `DisputeResolvedCustomerBillingConsumer` inbox consumers.
- Dunning/overdue check updated to exclude `ON_HOLD` invoices.

## Definition of Done

- All subtasks (22.4.1, 22.4.2, 22.4.3) completed and their acceptance criteria verified.
- All listed dependencies satisfied, including 22.2.3's frozen event contracts.
- Build and tests green per the global conventions in [../README.md](../README.md) Section 3.
- No code in this feature writes to `dispute-db` or calls dispute-service's API directly; all
  dispute-awareness in billing-service is inbox-consumer-driven (ADR-006, ADR-009).
