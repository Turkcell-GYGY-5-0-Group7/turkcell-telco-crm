# 22.2 Dispute State Machine and Orchestration

> Sprint: [Sprint 22 - Invoice Dispute / Chargeback](./README.md)
> Feature ID: 22.2
> Subtasks: 22.2.1, 22.2.2, 22.2.3

## Objective

Implement the Dispute state machine (ADR-028 Section 4, design-note.md Section 4) inside
dispute-service's domain model, the application-layer commands/handlers that drive each transition,
and - critically - the exact event contracts (ADR-028 Section 6, design-note.md Section 8) that
billing-service (22.4), payment-service (22.5), and ticket-service (22.6) will consume. This feature
does **not** call billing-service or payment-service directly; dispute-service only ever publishes
events through its outbox (ADR-009, ADR-028 Section 3/5). Getting the provisional-hold invariant right
here - `UNDER_REVIEW`/`dispute.opened.v1` is a hold signal, never a financial instruction - is the
single most important correctness property of this feature and of the sprint.

Aggregated from the subtasks below (prerequisite task IDs and enablers):

## Subtasks

#### 22.2.1 Dispute state machine domain logic
- ID: 22.2.1
- Title: Implement the transition methods and guards on `Dispute` per ADR-028 Section 4
- Description: Extend `Dispute` (from 22.1.3) with transition methods enforcing the state machine
  exactly as specified:
  `OPENED -> UNDER_REVIEW -> EVIDENCE_SUBMITTED -> (loop) UNDER_REVIEW`,
  `UNDER_REVIEW -> RESOLVED_CUSTOMER | RESOLVED_MERCHANT -> CLOSED`,
  `OPENED | UNDER_REVIEW -> WITHDRAWN -> CLOSED`. Each transition method
  (`beginReview()`, `submitEvidence()`, `resolveCustomer(resolutionAmount)`,
  `resolveMerchant()`, `withdraw()`, `close()`) throws `BusinessRuleException` on an illegal
  transition (mirroring the guard style in
  `microservices/billing-service/src/main/java/com/telco/billing/domain/Invoice.java`'s
  `issue()`/`markPaid()`), and appends a `DisputeStateHistory` row (`fromStatus`, `toStatus`,
  `changedBy`, `changedAt`, `note`) on every successful transition. `resolveCustomer` requires a
  non-null, positive `resolutionAmount`; `resolveMerchant` takes none (ADR-028 Section 5: "simply
  clears the hold, no financial change"). Add a unit test class exercising every legal transition and
  every illegal transition from every state (exhaustive state-machine table test).
- Business Purpose: Encodes the authoritative dispute lifecycle as an enforceable domain invariant so
  no application code path (API, event consumer, admin tool) can drive the aggregate into an invalid
  state - the same discipline `Invoice`/`Payment`/`Order` already apply to their own lifecycles.
- Inputs: ADR-028 Section 4, design-note.md Section 4,
  `microservices/billing-service/.../domain/Invoice.java`,
  `microservices/payment-service/.../domain/Payment.java` (state-guard pattern reference).
- Outputs: updated `microservices/dispute-service/src/main/java/com/telco/dispute/domain/Dispute.java`,
  `microservices/dispute-service/src/test/java/com/telco/dispute/domain/DisputeStateMachineTest.java`.
- Acceptance Criteria:
  - Every transition listed in ADR-028 Section 4 succeeds and appends exactly one
    `DisputeStateHistory` row; every other (state, transition) pair throws `BusinessRuleException`.
  - `resolveCustomer(null)` and `resolveCustomer(BigDecimal.ZERO or negative)` throw
    `BusinessRuleException`.
  - `DisputeStateMachineTest` covers all 7 states x 6 transition methods (42 cases, most asserting
    rejection) per the exhaustive-table style of
    `microservices/order-service/src/test/java/com/telco/order/domain/OrderStateMachineTest.java`.
- Dependencies: 22.1.3
- Complexity: L

#### 22.2.2 Application commands, handlers, and the dispute event contracts
- ID: 22.2.2
- Title: Implement `OpenDisputeCommand`, `SubmitEvidenceCommand`, `ResolveDisputeCustomerCommand`,
  `ResolveDisputeMerchantCommand`, `WithdrawDisputeCommand`, `CloseDisputeCommand` and their handlers,
  publishing the six `dispute.*.v1` events via the outbox
- Description: Add one `Command<R>` (ADR-008/CQRS) and one `CommandHandler` per transition in
  22.2.1, each: (1) loads the `Dispute` aggregate, (2) invokes the corresponding domain transition
  method, (3) persists via `DisputeRepository`, (4) writes an `audit_log` row (dispute-service is
  audit-mandated, ADR-028 Section 2) via an `AuditLogWriter` mirroring
  `microservices/payment-service/src/main/java/com/telco/payment/application/AuditLogWriter.java`,
  and (5) publishes the matching event through `OutboxService.publish("dispute", disputeId, eventType,
  payload)` - never calls Kafka directly (`docs/architecture/platform-capabilities.md` Section 1).
  `OpenDisputeCommand` moves a new `Dispute` straight to `UNDER_REVIEW` and publishes
  `dispute.opened.v1`; `SubmitEvidenceCommand` records a `DisputeEvidence` row (object ref supplied by
  22.3's upload flow) and publishes `dispute.evidence-submitted.v1`; `ResolveDisputeCustomerCommand`
  publishes `dispute.resolved-customer.v1`; `ResolveDisputeMerchantCommand` publishes
  `dispute.resolved-merchant.v1`; `WithdrawDisputeCommand` publishes `dispute.withdrawn.v1`;
  `CloseDisputeCommand` publishes `dispute.closed.v1`. Event payload shapes are fixed by 22.2.3 and
  MUST be treated as a stable contract once merged, since 22.4/22.5/22.6 build inbox consumers against
  them.
- Business Purpose: This is the orchestration core of the sprint: it turns the domain state machine
  into the event stream that billing-service, payment-service, and ticket-service react to, entirely
  through the outbox/inbox pattern (ADR-009, ADR-019) with no direct cross-service calls or shared
  database access (ADR-006).
- Inputs: ADR-028 Sections 5-6, design-note.md Sections 5, 8, `docs/architecture/platform-capabilities.md`
  Section 1 (`OutboxService`, `Mediator`),
  `microservices/payment-service/src/main/java/com/telco/payment/application/handler/RefundPaymentCommandHandler.java`
  (handler + outbox + audit pattern reference - reuse the shape, not the payment logic).
- Outputs: `microservices/dispute-service/src/main/java/com/telco/dispute/application/command/`
  (six `*Command.java`), `.../application/handler/` (six `*CommandHandler.java`),
  `.../application/event/` (`DisputeOpenedEvent.java`, `DisputeEvidenceSubmittedEvent.java`,
  `DisputeResolvedCustomerEvent.java`, `DisputeResolvedMerchantEvent.java`,
  `DisputeWithdrawnEvent.java`, `DisputeClosedEvent.java`), `.../application/AuditLogWriter.java`.
- Acceptance Criteria:
  - **Provisional-hold invariant (explicit, load-bearing):** `OpenDisputeCommandHandler` only writes
    to `dispute-db` and the dispute-service outbox; it makes no HTTP call, no Kafka call outside the
    outbox, and no write of any kind to `billing-db`/`payment-db`. `dispute.opened.v1` is the only
    signal a hold ever travels on - it is documentation-level, machine-verifiable proof that opening
    a dispute cannot itself mutate any financial ledger (ADR-028 Section 5, design-note.md Section 5).
  - `ResolveDisputeCustomerCommandHandler` is the only handler that publishes
    `dispute.resolved-customer.v1`, and it is the only dispute-service-originated event that
    downstream consumers (22.4, 22.5) are permitted to treat as authorization for a real financial
    action; `dispute.resolved-merchant.v1` clears the hold with **no** financial change downstream by
    contract, not by consumer-side omission.
  - Each of the six commands is wrapped in a single DB transaction with its outbox insert (platform
    `TransactionBehavior`, per `docs/architecture/platform-capabilities.md` Section 1) - no
    handler ever publishes to Kafka directly.
  - Unit tests (`*CommandHandlerTest`) cover each handler's happy path and its illegal-transition
    rejection (delegated from 22.2.1), asserting the correct event type and payload are placed on the
    outbox.
- Dependencies: 22.2.1
- Complexity: L

#### 22.2.3 Event contract definition (payload shapes, versioning, event-catalog draft)
- ID: 22.2.3
- Title: Fix the exact payload shape of all six `dispute.*.v1` events and stage the
  `docs/architecture/event-catalog.md` registry entries
- Description: Define, as the authoritative contract for the rest of the sprint, the JSON/Avro field
  list for each event per design-note.md Section 5 and Section 8:
  `dispute.opened.v1 { disputeId, invoiceId, paymentId, customerId, disputedAmount, reasonCode,
  openedAt }`; `dispute.evidence-submitted.v1 { disputeId, evidenceId, submittedBy, submittedAt }`;
  `dispute.resolved-customer.v1 { disputeId, invoiceId, paymentId, resolutionAmount, resolvedAt }`;
  `dispute.resolved-merchant.v1 { disputeId, invoiceId, paymentId, resolvedAt }`;
  `dispute.withdrawn.v1 { disputeId, withdrawnAt }`; `dispute.closed.v1 { disputeId, closedAt }`.
  Follow `domain.event.v1` naming (ADR-019), `aggregate_type = "dispute"` (lowercase, routes to the
  `dispute.events` Kafka topic per `docs/architecture/event-catalog.md` Section header rules - a
  PascalCase `aggregate_type` silently misroutes, per the documented order-service lesson). This
  subtask is a **hard prerequisite** for 22.4 and 22.5: their inbox consumers are written against
  these exact field names, and for 22.6's Avro/Schema-Registry registration. Produce a short
  event-contract note (payload table per event) inside this feature's PR/description for 22.4/22.5/
  22.6 implementers to consume; the formal Avro schema and `event-catalog.md` edit are 22.6's scope
  (do not edit `event-catalog.md` in this subtask - draft the shapes here, register them in 22.6).
- Business Purpose: Freezes the cross-service contract early so billing-service (22.4) and
  payment-service (22.5) - both of which depend on this task - can be implemented in parallel against
  a stable shape, avoiding rework.
- Inputs: ADR-028 Section 6, design-note.md Sections 5 and 8, ADR-019, `docs/architecture/event-catalog.md`
  Section 1 (naming/versioning rules) as reference only (not edited here).
- Outputs: the six `application/event/*Event.java` DTOs from 22.2.2 finalized with these exact
  fields; no `docs/` edits in this subtask (the event-catalog.md registration is 22.6.1).
- Acceptance Criteria:
  - Each of the six event DTOs matches its field list above exactly (name, nullability, type).
  - `aggregate_type` used in every `OutboxService.publish(...)` call in 22.2.2 is the lowercase
    literal `"dispute"`.
  - `dispute.resolved-customer.v1` carries both `invoiceId` and `paymentId` (nullable each, mirroring
    the Dispute aggregate) so a consumer can determine which of billing-service/payment-service should
    act, per ADR-028 Section 5's "depending on where the money currently sits" rule.
- Dependencies: 22.2.2
- Complexity: S

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met. In addition, at the feature level:

- The provisional-hold-vs-real-financial-action invariant holds end to end within dispute-service:
  no code path reachable from `OpenDisputeCommand`, `SubmitEvidenceCommand`,
  `ResolveDisputeMerchantCommand`, or `WithdrawDisputeCommand` performs, or could be mistaken for
  performing, a financial mutation; only `ResolveDisputeCustomerCommand`'s resulting event
  (`dispute.resolved-customer.v1`) is a valid trigger for a real credit/refund, and that action itself
  is performed by billing-service/payment-service (22.4/22.5), never by dispute-service.

## Deliverables

- Fully guarded `Dispute` state machine with `DisputeStateHistory` audit trail.
- Six commands/handlers publishing six versioned `dispute.*.v1` events via the outbox.
- A frozen event-contract shape for `dispute.opened.v1`, `dispute.evidence-submitted.v1`,
  `dispute.resolved-customer.v1`, `dispute.resolved-merchant.v1`, `dispute.withdrawn.v1`,
  `dispute.closed.v1`, ready for 22.4/22.5/22.6 to consume.
