# 22.6 Dispute Eventing and ticket-service Integration

> Sprint: [Sprint 22 - Invoice Dispute / Chargeback](./README.md)
> Feature ID: 22.6
> Subtasks: 22.6.1, 22.6.2, 22.6.3

## Objective

Close out the sprint: formally register the six `dispute.*.v1` events as versioned Avro contracts in
the Schema Registry and `docs/architecture/event-catalog.md` (ADR-019), add ticket-service's
`dispute.opened.v1` auto-ticket consumer reusing its existing SLA/assignment machinery (ADR-028
Section 4, design-note.md Section 6 - reuse-before-build), and add the cross-service unit/integration/
acceptance test suite proving the end-to-end dispute lifecycle - including the
provisional-hold-vs-real-financial-action invariant - holds across dispute-service, billing-service,
payment-service, and ticket-service.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- 22.2 (event contracts and orchestration), 22.3 (API surface for the acceptance test to drive),
  22.4 (billing-service hold/adjustment consumers), 22.5 (payment-service hold/refund consumers) -
  all four must be functionally complete before the end-to-end acceptance test in 22.6.3 can pass.
- ADR-019 Event Contract and Schema Governance, ADR-009 Event Driven Architecture
- `docs/architecture/event-catalog.md` (registry to extend), `microservices/ticket-service/` existing
  `Ticket`/`TicketComment`/`SlaPolicy` aggregates and `OpenTicketCommand`

## Subtasks

#### 22.6.1 Avro schema registration and event-catalog.md update
- ID: 22.6.1
- Title: Register `dispute.opened.v1` through `dispute.closed.v1` as Avro schemas and update the
  event catalog
- Description: Translate the six event DTOs frozen in 22.2.3 into Avro schemas registered with the
  Schema Registry (ADR-019), following the same `EventEnvelope` + `*V1` record convention used by the
  other domains (`com.telco.platform.events.*`, per `docs/architecture/platform-capabilities.md`
  Section 1). Confirm `aggregate_type = "dispute"` (lowercase) routes outbox rows to the
  `dispute.events` Kafka topic via the Debezium `EventRouter`, matching the documented convention in
  `docs/architecture/event-catalog.md`'s header rules. Add a new row to
  `docs/architecture/event-catalog.md` Section 2 for each of the six events (Producer: dispute-service;
  Consumers: billing-service, payment-service, ticket-service, notification-service per ADR-028
  Section 6/design-note.md Section 8) and, if the platform's saga-sequence diagrams (Section 3 of that
  file) are extended elsewhere in the doc, add a short dispute-resolution sequence block alongside the
  existing "Saga: New-Line Order" one. This subtask is the only one in Sprint 22 that touches
  `docs/architecture/event-catalog.md`; do not edit it in 22.2/22.4/22.5.
- Business Purpose: Makes the dispute event contract a first-class, governed part of the platform's
  event catalog (ADR-019) rather than an implicit convention living only in dispute-service's code,
  so future services can discover and safely consume it.
- Inputs: 22.2.3 (frozen payload shapes), ADR-019, `docs/architecture/event-catalog.md` (current
  structure and Section 2 table format).
- Outputs: Avro schema files (`.avsc`) for the six events under the platform event-contracts module
  (mirrors existing domain event schemas), updated `docs/architecture/event-catalog.md` Section 2
  (six new rows) and, if applicable, a new saga-sequence subsection.
- Acceptance Criteria:
  - Each of the six `dispute.*.v1` Avro schemas registers with the Schema Registry without a
    compatibility violation (ADR-019 backward-compatibility check).
  - `docs/architecture/event-catalog.md` Section 2 lists all six events with correct Producer/
    Consumers/Purpose columns matching ADR-028 Section 6.
  - dispute-service's outbox rows route to the `dispute.events` topic (verified via a
    Testcontainers/embedded-Kafka assertion, not just code inspection).
- Dependencies: 22.2.3
- Complexity: M

#### 22.6.2 ticket-service auto-ticket consumer on `dispute.opened.v1`
- ID: 22.6.2
- Title: Add `DisputeOpenedTicketConsumer`, reusing `OpenTicketCommand` and existing SLA/assignment
  machinery
- Description: Add a Kafka inbox consumer in ticket-service on `dispute.events`, consuming
  `dispute.opened.v1` and dispatching the **existing**
  `microservices/ticket-service/src/main/java/com/telco/ticket/application/command/OpenTicketCommand.java`
  (`customerId`, `category = "DISPUTE"`, a priority derived from `disputedAmount`/policy, and a
  generated `subject` such as `"Dispute opened: " + disputeId`) via the `Mediator` - do not add a new
  ticket workflow or a parallel SLA timer; this is exactly the reuse ADR-028 Section 4 and
  design-note.md Section 6 mandate ("ticket-service already owns SLA tracking... reusing existing
  SLA/assignment machinery instead of duplicating it in dispute-service"). Because `Ticket` (see
  `microservices/ticket-service/src/main/java/com/telco/ticket/domain/Ticket.java`) has no field to
  correlate back to the originating dispute, add a small, ticket-service-owned migration and column
  (e.g. `external_ref VARCHAR` on `tickets`, nullable, populated with the dispute id for `DISPUTE`
  category tickets) - this is ticket-service's own schema addition under its own migration and write
  ownership, matching the same "small, service-owned extension" pattern already used in 22.4/22.5; it
  is not a change to dispute-service. Existing SLA due-date computation (`SlaPolicy`,
  `slaDueAt`) applies unchanged to the `DISPUTE` category, per reuse-before-build.
- Business Purpose: Ensures every opened dispute is immediately visible to a support agent through
  the platform's existing, SLA-tracked ticketing workflow, without dispute-service reinventing agent
  assignment or SLA timers (ADR-028 Section 4, reuse-before-build).
- Inputs: 22.2.3/22.6.1 (frozen and registered `dispute.opened.v1` shape), design-note.md Section 6,
  `microservices/ticket-service/.../application/command/OpenTicketCommand.java`,
  `microservices/ticket-service/.../application/handler/OpenTicketCommandHandler.java`,
  `microservices/ticket-service/.../domain/Ticket.java`, `microservices/ticket-service/.../domain/SlaPolicy.java`.
- Outputs: `microservices/ticket-service/src/main/java/com/telco/ticket/infrastructure/consumer/DisputeOpenedTicketConsumer.java`,
  `microservices/ticket-service/src/main/resources/db/migration/V{n}__add_external_ref_to_tickets.sql`,
  updated `Ticket.java` (`externalRef` field, nullable).
- Acceptance Criteria:
  - `dispute.opened.v1` results in exactly one new `Ticket` with `category = "DISPUTE"`,
    `externalRef` equal to the dispute id, and an `slaDueAt` computed by the existing `SlaPolicy`
    machinery (no new SLA logic added).
  - No change is made to `OpenTicketCommand.java`, `OpenTicketCommandHandler.java`, or `SlaPolicy.java`
    - the consumer calls the existing command as-is (diff-verifiable), confirming reuse rather than
    duplication.
  - The consumer is idempotent via `InboxService.firstSeen(...)`; redelivery of the same
    `dispute.opened.v1` message does not open a second ticket for the same dispute.
- Dependencies: 22.6.1
- Complexity: M

#### 22.6.3 Cross-service test suite: unit, integration, and end-to-end acceptance
- ID: 22.6.3
- Title: Add the Sprint 22 test suite proving the dispute lifecycle and the provisional-hold invariant
  hold end to end
- Description: Add three layers of tests, matching the platform's existing conventions
  (`docs/tasks/README.md` Section 3, and the acceptance-test pattern in
  `microservices/acceptance-tests/src/test/java/com/telco/acceptance/`): (1) unit tests already
  scoped per-subtask in 22.2/22.4/22.5 (`DisputeStateMachineTest`, per-handler tests) - this subtask
  verifies coverage is complete, not re-authors them; (2) Testcontainers integration tests per service
  proving each inbox consumer's outbox-to-inbox round trip (dispute-service publishes ->
  billing-service/payment-service/ticket-service consumes -> expected state change, using an embedded
  Kafka/Testcontainers harness consistent with
  `microservices/order-service/src/test/java/com/telco/order/routing/OutboxRoutingRegressionTest.java`'s
  approach); (3) one new end-to-end acceptance test,
  `microservices/acceptance-tests/src/test/java/com/telco/acceptance/dispute/DisputeResolutionAcceptanceIT.java`,
  driving the full lifecycle through the live stack: open a dispute against a paid invoice ->
  `UNDER_REVIEW` (assert invoice `ON_HOLD`, no PDF/grandTotal change, payment `disputed=true`, a
  `DISPUTE` ticket auto-opened) -> submit evidence -> resolve `CUSTOMER` (assert a real refund fires
  via the existing refund path and `payment.refunded.v1` is observed) -> `CLOSED`; a second scenario
  resolves `MERCHANT` and asserts zero financial change end to end.
- Business Purpose: Sprint 22's stated Exit Criteria (README.md) require proof - not just code - that
  a `RESOLVED_CUSTOMER` outcome produces a real credit/refund, a `RESOLVED_MERCHANT` outcome produces
  none, and the invoice is excluded from dunning while `ON_HOLD`; this suite is that proof.
- Inputs: `docs/tasks/README.md` Section 3 (global test conventions),
  `microservices/acceptance-tests/src/test/java/com/telco/acceptance/ac01/NewSubscriberOnboardingAcceptanceIT.java`
  and `.../support/` helpers (pattern reference), Sprint 22 README.md "Exit Criteria",
  `microservices/order-service/src/test/java/com/telco/order/routing/OutboxRoutingRegressionTest.java`
  (outbox-routing regression pattern).
- Outputs: `microservices/dispute-service/src/test/java/com/telco/dispute/` (integration tests),
  `microservices/billing-service/src/test/java/com/telco/billing/infrastructure/consumer/DisputeConsumersIntegrationTest.java`,
  `microservices/payment-service/src/test/java/com/telco/payment/infrastructure/consumer/DisputeConsumersIntegrationTest.java`,
  `microservices/ticket-service/src/test/java/com/telco/ticket/infrastructure/consumer/DisputeOpenedTicketConsumerTest.java`,
  `microservices/acceptance-tests/src/test/java/com/telco/acceptance/dispute/DisputeResolutionAcceptanceIT.java`.
- Acceptance Criteria:
  - **Provisional-hold-vs-real-financial-action invariant, proven end to end:** the acceptance test
    asserts zero financial-field change on both `dispute.opened.v1` and `dispute.resolved-merchant.v1`
    paths (invoice `grandTotal`, payment `status`/PSP-call count), and exactly one real financial
    action (adjustment line xor refund, never both) on the `dispute.resolved-customer.v1` path,
    selected correctly based on paid/unpaid state.
  - `DisputeResolutionAcceptanceIT` covers both the `RESOLVED_CUSTOMER` (paid-invoice-refund) and
    `RESOLVED_MERCHANT` (no-change) branches as two distinct scenarios.
  - All Sprint 22 README.md "Exit Criteria" bullets are demonstrably satisfied by this suite (link the
    test method to the exit criterion it proves in a comment or Javadoc).
  - Full build (`mvn verify` across dispute-service, billing-service, payment-service, ticket-service,
    acceptance-tests) is green.
- Dependencies: 22.4, 22.5, 22.6.2
- Complexity: L
