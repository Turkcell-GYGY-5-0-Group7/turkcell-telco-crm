## 14.2.1 PII-at-rest encryption audit (NFR-06)

### What was checked
- Every PII data element that NFR-06 mandates for encryption at rest: customer TCKN/VKN and payment
  card data.
- JPA entities, `AttributeConverter` implementations, key sourcing, and Flyway DDL.
- Runtime confirmation that the stored column holds ciphertext (not plaintext).

### Findings (evidence)
- Customer TCKN/VKN is AES-256-GCM encrypted at rest:
  - `microservices/customer-service/src/main/java/com/telco/customer/infrastructure/crypto/IdentityNumberCryptoConverter.java`
    - `AES/GCM/NoPadding`, fresh random 12-byte IV per encryption prepended to ciphertext, 128-bit GCM
      auth tag, Base64-encoded for storage (lines 32-64).
  - Applied explicitly on the entity field via `@Convert` (no `autoApply`):
    `microservices/customer-service/src/main/java/com/telco/customer/domain/Customer.java:57-60`
    (`identity_number_enc`, column holds ciphertext, domain holds plaintext).
  - DDL stores the encrypted column only:
    `microservices/customer-service/src/main/resources/db/migration/V1__customer.sql:12`
    (`identity_number_enc TEXT NOT NULL`); the header comment states plaintext is never stored (line 2).
- Key sourcing is from secrets, not hardcoded:
  - `microservices/customer-service/src/main/java/com/telco/customer/infrastructure/crypto/AesKeyProvider.java:26`
    reads `${customer.crypto.aes-key}` and enforces a 32-byte AES-256 key (lines 27-31).
  - Production and staging inject the key from environment/secret with no default:
    `microservices/configs/customer-service/application-prod.yml:16` and `application-staging.yml:16`
    (`aes-key: ${CUSTOMER_AES_KEY}`). Only the local base profile carries a dev default
    (`microservices/configs/customer-service/application.yml:39`), which is acceptable for local dev.
- DB-level ciphertext confirmation exists as an automated test against a real PostgreSQL container:
  - `microservices/customer-service/src/test/java/com/telco/customer/infrastructure/persistence/CustomerRepositoryTest.java:94-109`
    (`identityNumberIsStoredAsCiphertextNotPlaintext`) issues a native
    `SELECT identity_number_enc` and asserts the raw column value is not the plaintext TCKN while the
    entity view round-trips to plaintext.
  - `microservices/customer-service/src/test/java/com/telco/customer/infrastructure/crypto/IdentityNumberCryptoConverterTest.java`
    asserts ciphertext != plaintext and does not contain the plaintext (lines 23-27), round-trips
    (34), and is non-deterministic across encryptions due to random IV (39-46).
- Payment card data: NONE is persisted. payment-service integrates a PSP (`MockPspAdapter`) and stores
  only order id, customer id, amount, status, and idempotency key
  (`microservices/payment-service/src/main/java/com/telco/payment/domain/Payment.java`,
  `.../domain/PaymentAttempt.java`, `.../application/dto/ChargePaymentRequest.java`). No PAN, CVV, or
  expiry is accepted, stored, or logged. This "don't store card data; delegate to the PSP" model is the
  correct PCI-DSS posture and removes the need for card-at-rest encryption in this service.
- identity-service `email` is PII but is NOT mandated for encryption at rest by NFR-06 (which lists
  TCKN and card number only). It is stored in plaintext and masked in telemetry via `@Sensitive`
  (`microservices/identity-service/src/main/java/com/telco/identity/domain/User.java:51-53`). `username`
  is an operational login identifier, plaintext, not encrypted. Both are acceptable under NFR-06.

### Result: PASS
All PII columns that NFR-06 mandates for encryption are ciphertext at rest (proven by an automated
native-query DB inspection), keys come from secrets in staging/prod, and payment holds no card data.
Key rotation is documented in `docs/architecture/security-posture.md` (Section on key management).

### Notes
- `AesKeyProvider` supports a single active key (no key id / envelope). A rotation procedure is
  documented in the posture doc; a zero-downtime rotation would benefit from a key-id column and
  dual-key decrypt. Recorded as a recommendation, not a gap (the MVP scope is a single active key).

## 14.2.2 PII telemetry masking audit (ADR-021)

### What was checked
- That the platform masking primitives cover all four required PII types (TCKN, card number, MSISDN,
  email) and that masking is applied across logs, trace attributes, and metric labels.

### Findings (evidence)
- Central pattern set covers all four required types plus IBAN:
  `platform/platform-core/common/src/main/java/com/telco/platform/common/masking/PiiMasker.java:27-42`
  - EMAIL (30), IBAN (33), PAN / card number (36), MSISDN (39), TCKN (42). Patterns are applied
    longest-format-first so an earlier mask breaks a later digit run (89-102).
- Two-layer masking (ADR-021):
  - Layer A (structural): `@Sensitive` + Jackson module masks annotated fields in structured JSON.
    `platform/platform-autoconfigure/src/main/java/com/telco/platform/autoconfigure/masking/SensitiveMaskingModule.java`,
    annotation `platform/platform-core/common/src/main/java/com/telco/platform/common/masking/Sensitive.java`.
    Applied on `Customer.identityNumber` (`@Sensitive(PARTIAL)`, Customer.java:58) and `User.email`
    (`@Sensitive(EMAIL)`, User.java:51).
  - Layer B (free-text backstop): Logback converters mask rendered messages and stack traces.
    `platform/platform-starters/starter-observability/src/main/java/com/telco/platform/starter/observability/logging/PiiMaskingConverter.java`
    and `PiiMaskingThrowableConverter.java`, registered as `%mask` / `%maskEx` in
    `platform/platform-starters/starter-observability/src/main/resources/logback-spring.xml:27-30`.
- Traces and metrics: a source scan found NO PII placed into span attributes or metric tags anywhere
  (`grep` for `.tag(`, `Tags.of`, `setAttribute`, `Attributes.of` intersected with tckn/identity/
  msisdn/email/card/iban/phone/pan returned no matches under `microservices/**/src/main`). The
  effective control for traces/metrics is that PII is never used as a span attribute or metric label
  in the first place, which the code upholds.

### Result: PASS (with two minor recommendations)

Recommendation 1 (documented, not a code gap): The JSON path that actually ships to Loki uses
`LogstashEncoder`, which serialises the MDC and the raw message via its own JsonProvider chain and does
NOT execute Logback pattern conversion words. Therefore `%mask` / `%maskEx` (Layer B) do not run on the
JSON appender output; Layer B only masks pattern-based appenders. This is called out honestly in the
platform logback comments (`logback-spring.xml:16-22`) and the reference-service variant
(`microservices/reference-service/src/main/resources/logback-spring.xml:17-21`). For JSON logs the
primary defense is Layer A (`@Sensitive`). This is sound as long as every PII field is annotated and
free-text log statements do not interpolate raw PII. Recommendation: keep a lint/review rule that no
`log.*` statement interpolates a raw PII value, since such a value in a JSON message body would bypass
Layer B.

Recommendation 2 (defense-in-depth): subscription-service `SimCard.msisdn`
(`microservices/subscription-service/src/main/java/com/telco/subscription/domain/SimCard.java:27-28`)
is not annotated `@Sensitive`. MSISDN in a rendered log message is caught by the Layer B free-text
backstop, but if a subscription/SIM object were ever serialised structurally to a JSON log, the field
would not be masked by Layer A. Recommendation: add `@Sensitive(MaskStrategy.PARTIAL)` to the `msisdn`
field for symmetry with `Customer.identityNumber` and `User.email`. Left as a recommendation rather
than a change because MSISDN is not currently serialised into any structured log and the field owner is
the domain-engineer.

### Acceptance
Seeded PII values do not appear unmasked: TCKN and email are masked structurally (Layer A) and all four
types are masked in rendered log text (Layer B); no PII is written to trace attributes or metric labels.

## 14.2.3 Audit-log completeness (NFR-12)

### What was checked
Every state-changing command handler in the four mandated services (identity, customer, payment,
subscription) was enumerated and checked for an `AuditLogWriter.log(...)` call that carries actor,
action, entity, and correlationId. The shared audit shape is identical across services:
`AuditLog(id, actorId, action, entity, entityId, details, correlationId, createdAt)`; the writer pulls
`actorId` from `UserContextHolder` and `correlationId` from `CorrelationContextHolder` and runs inside
the caller's (mediator) transaction (e.g.
`microservices/customer-service/src/main/java/com/telco/customer/application/AuditLogWriter.java:40-67`).

### identity-service - PASS
All state-changing handlers audit:
- `CreateUserCommandHandler.java:63` (`USER_CREATED`), `DeleteUserCommandHandler.java:67`
  (`USER_DELETED`), `AssignRolesCommandHandler.java:59` (`ROLES_ASSIGNED`),
  `RemoveRolesCommandHandler.java:59` (`ROLES_REMOVED`).
- `correlation_id` column added by `V2__audit_log_correlation.sql`.

### customer-service - PARTIAL (GAP-FLAGGED for address sub-resource)
Audited: `RegisterCustomerCommandHandler.java:50` (`CUSTOMER_REGISTERED`),
`UpdateCustomerCommandHandler.java:48` (`CUSTOMER_UPDATED`),
`ApproveKycCommandHandler.java:59` (`CUSTOMER_KYC_APPROVED`),
`RejectKycCommandHandler.java:52` (`CUSTOMER_KYC_REJECTED`),
`DeleteCustomerCommandHandler.java:34` (`CUSTOMER_DELETED`),
`UploadDocumentCommandHandler.java:63` (`DOCUMENT_UPLOADED`).

Not audited (state-changing, but no audit row):
- `AddAddressCommandHandler.java` (creates an Address; also clears a prior default).
- `UpdateAddressCommandHandler.java` (mutates address fields).
- `SetDefaultAddressCommandHandler.java` (changes which address is default).

These persist state changes to customer address data and, under a strict reading of NFR-12 / MVP brief
Section 13 ("every change is written to an audit_log table"), should emit an audit row. Severity: medium
(address is a customer sub-resource, not the core PII/KYC lifecycle, all of which is covered).

### subscription-service - PASS
All lifecycle transitions audit:
- `ActivateSubscriptionCommandHandler.java:90`, `SuspendSubscriptionCommandHandler.java:62`,
  `ReactivateSubscriptionCommandHandler.java:63`, `TerminateSubscriptionCommandHandler.java:69`,
  `SuspendSubscriptionsForNonPaymentCommandHandler.java:57`.
- `FailSubscriptionActivationCommandHandler` writes its audit row via
  `SubscriptionActivationFailedEmitter` (the single source of truth for both the failure event and the
  audit row, `.../application/SubscriptionActivationFailedEmitter.java:55`), so the pre-activation
  failure path is covered even though the handler has no direct call.

### payment-service - GAP-FLAGGED (no audit infrastructure at all)
payment-service has NO `audit_log` table, NO `AuditLog` entity, NO `AuditLogRepository`, and NO
`AuditLogWriter`. The only "audit-like" artifact is `payment_attempts`
(`V2__create_payment_attempts.sql`), which is a per-charge technical attempt trail, not the mandated
actor/action/entity/correlationId audit record. State-changing operations that currently emit NO audit
row:
- `ChargePaymentCommandHandler.handle` - COMPLETED path (Payment.java `markCompleted`) and FAILED path
  (`markFailed`), `.../application/handler/ChargePaymentCommandHandler.java:105-141`.
- `RefundPaymentCommandHandler.handle` - REFUNDED path,
  `.../application/handler/RefundPaymentCommandHandler.java:56-77`.

This directly violates NFR-12, MVP brief Section 13 (line 666: payment is one of the four services
where "every change is written to an audit_log table"), root `CLAUDE.md` ("audit logging where
mandated"), and the security agent charter (MUST NOT "skip audit logging in ... payment ..."). Severity:
HIGH.

### Why no fix was applied here
The correct, minimal fix for both gaps is to inject `AuditLogWriter` (constructor injection, matching
the other three services) into the affected handlers and add one `log(...)` call per state transition;
for payment it additionally requires a new `V3` Flyway `audit_log` migration plus the `AuditLog`
entity, `AuditLogRepository`, and `AuditLogWriter` (a byte-for-byte copy of the subscription/customer
implementation). However, the affected handlers are directly instantiated in unit tests owned by the
parallel qa agent and outside this agent's edit scope:
- `microservices/payment-service/src/test/java/com/telco/payment/application/handler/ChargePaymentCommandHandlerTest.java:43`
  and `RefundPaymentCommandHandlerTest.java:42` call `new ChargePaymentCommandHandler(...)` /
  `new RefundPaymentCommandHandler(paymentRepository, pspAdapter, outboxService)`.
- `microservices/customer-service/src/test/java/.../AddAddressCommandHandlerTest.java:36`,
  `UpdateAddressCommandHandlerTest.java:32`, `SetDefaultAddressCommandHandlerTest.java:33` call the
  address handler constructors directly.

Adding a constructor parameter would break these test sources, which this agent must not edit. Applying
the fix without the paired test change would fail the build. The additive parts that would NOT break
tests (new payment `audit_log` migration + entity + repository + writer bean; the payment schema
migration test only asserts table presence, not an exclusive set -
`.../migration/PaymentSchemaMigrationTest.java:36`) were also left unapplied to avoid shipping dead,
unwired infrastructure. Both gaps are therefore flagged for coordinated delivery.

### Result: GAP-FLAGGED (see "Items needing a decision")

### Ready-to-apply recommendation
1. payment-service (HIGH): add `microservices/payment-service/src/main/resources/db/migration/V3__audit_log.sql`
   with the standard DDL (id, actor_id, action, entity, entity_id, details JSONB, correlation_id,
   created_at + indexes on actor_id and created_at); copy `AuditLog`, `AuditLogRepository`, and
   `AuditLogWriter` from subscription-service (same package layout); inject the writer into
   `ChargePaymentCommandHandler` and `RefundPaymentCommandHandler`; emit `PAYMENT_COMPLETED`,
   `PAYMENT_FAILED`, and `PAYMENT_REFUNDED` rows (entity `"Payment"`, entityId = payment id, no PII in
   details - amount/orderId only). Update `ChargePaymentCommandHandlerTest` /
   `RefundPaymentCommandHandlerTest` constructors in the same change (qa agent).
2. customer-service (MEDIUM): inject the existing `AuditLogWriter` into the three address handlers and
   emit `ADDRESS_ADDED` / `ADDRESS_UPDATED` / `ADDRESS_DEFAULT_SET` (entity `"Address"`). Update the
   three address handler tests in the same change (qa agent). Confirm with the product/tech-lead
   whether address sub-resource mutations are in NFR-12 scope before implementing.

## 14.2.4 mTLS posture and security review (NFR-05, ADR-011)

### What was checked
Gateway-behind-trust model and mTLS deferral; token handling; rate limiting; input validation; error
leakage (stack traces).

### Findings (evidence)
- Gateway-behind-trust and anti-spoofing:
  - `microservices/api-gateway/src/main/java/com/telco/gateway/filter/JwtClaimsFilter.java:39-57`
    unconditionally strips any client-supplied `X-User-Id` / `X-User-Roles` and re-injects them from
    the verified JWT subject and `roles` claim. A client cannot spoof identity headers.
  - JWT is validated against the realm JWKS:
    `microservices/api-gateway/src/main/java/com/telco/gateway/config/GatewaySecurityConfig.java:54-95`
    (NimbusJwtDecoder from `jwks-uri`, stateless sessions, CSRF disabled for a stateless resource
    server). `/internal/**` is denied at the edge (routed to a 404 sink) so internal endpoints are not
    reachable from outside (71).
- Token handling: Keycloak is the issuer (ADR-011); refresh-token rotation and reuse detection are
  realm features (`revokeRefreshToken: true`, `refreshTokenMaxReuse: 0`) per
  `docs/architecture/keycloak-and-auth.md:38`. identity-service does not mint tokens. NOTE: the MVP
  brief Section 13 (line 657) says identity-service issues JWT; the delivered platform follows ADR-011
  (Keycloak issuer), reconciled in the brief's own PLATFORM NOTE (lines 670-675). ADR-011 wins.
- Rate limiting (NFR-18): `RateLimitingFilter.java` - Redis fixed-window, 100 req/min, keyed by JWT
  subject (falls back to client IP for unauthenticated allowlisted routes), atomic INCR+EXPIRE Lua
  script (39-44). Fails OPEN on Redis outage (77-80) - a deliberate availability tradeoff, documented
  in the posture doc.
- CORS: explicit origin allowlist (`gateway.cors.allowed-origins`), not wildcard, with
  `allowCredentials=true` (GatewaySecurityConfig.java:108-122). The in-code comment (111-113) correctly
  warns that wildcard + credentials would be a leak; the delivered config uses an allowlist. Production
  must set `CORS_ALLOWED_ORIGINS`.
- Input validation: Bean Validation on request DTOs (e.g. `ChargePaymentRequest` `@NotNull`,
  `@DecimalMin`, `@Size`); `MethodArgumentNotValidException` / `ConstraintViolationException` map to
  400 with field-level detail only (GlobalExceptionHandler.java:98-120).
- Error leakage - no stack traces in responses:
  - `platform/platform-starters/starter-api/src/main/java/com/telco/platform/starter/api/GlobalExceptionHandler.java`
    maps each exception type to a status; the catch-all `handleUnexpected` (122-130) returns a generic
    `INTERNAL_ERROR` / "An unexpected error occurred" body with only `code`, `traceId`, and a `logId`
    correlation handle - never the exception message or stack trace. The full detail is logged
    server-side (124) and referenced by `logId`. Platform business exceptions return their controlled
    message only (132-141). The gateway's own 401/403 handlers likewise return fixed JSON envelopes
    (GatewaySecurityConfig.java:124-150).

### Result: PASS
mTLS deferral for the MVP is documented with a production recommendation (see
`docs/architecture/security-posture.md`). No high-severity findings in token handling, rate limiting,
input validation, or error handling. Error responses never leak stack traces.
