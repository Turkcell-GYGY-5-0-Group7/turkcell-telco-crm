## Part A â€” Bill-Run Throughput (NFR-02)

### What was built

1. **Seed harness** â€” `microservices/billing-service/src/test/java/com/telco/billing/perf/BillRunSeedHarness.java`.
   A reusable, repeatable JDBC-batch seeder (chunked `JdbcTemplate.batchUpdate`, 2000 rows/batch) that
   populates exactly the read models `RunBillCommandHandler`/`BillRunBatchProcessor` consume:
   - `subscriber_billing_records`: N ACTIVE postpaid subscribers spread across 5 tariff codes.
   - `tariff_prices`: one cached price row per tariff code (mirrors what
     `RecordSubscriptionActivatedCommandHandler` caches in production so the bill-run needs no
     synchronous cross-service call).
   - `overage_records`: ~20% of subscribers get a data/voice/SMS overage record for the billing
     period, exercising the overage invoice-line branch (not just the flat tariff-fee line).

   Seeding goes directly to JDBC rather than one `RecordSubscriptionActivatedCommand` per subscriber
   through the mediator, because seeding 100K rows of fixture data is not the thing NFR-02 measures â€”
   driving it through individual commands would dominate the harness's own runtime. The bill-run
   **under test** still goes through the real mediator/command path exactly as production does; the
   harness never touches `invoices`/`invoice_lines`/outbox tables.

2. **Throughput test** â€” `microservices/billing-service/src/test/java/com/telco/billing/perf/BillRunThroughputPerformanceIT.java`.
   Testcontainers Postgres (matching `BillingAC02IntegrationTest`'s existing pattern), seeds via
   `BillRunSeedHarness`, times `mediator.send(new RunBillCommand(...))` end to end, and asserts:
   - `invoicesGenerated == subscriberCount`, `invoicesSkipped == 0`.
   - elapsed wall-clock `< 30 minutes`.
   - **no duplicate invoices**: a direct SQL query
     (`SELECT subscription_id, period_start, COUNT(*) FROM invoices GROUP BY subscription_id,
     period_start HAVING COUNT(*) > 1`) returns zero rows.
   - invoice row count in the database matches the subscriber count exactly.

   Named `*IT` (not `*Test`) deliberately: billing-service has no Failsafe binding (unlike
   `acceptance-tests`), so a `*IT` class is excluded from the default `mvn test`/`mvn verify` Surefire
   run â€” a multi-minute performance validation must not slow down every routine build. It is run
   explicitly:
   ```
   mvn -f microservices/pom.xml -pl billing-service -am test \
       -Dtest=BillRunThroughputPerformanceIT -DfailIfNoTests=false -Dschema.registry.skip=true \
       -DsubscriberCount=100000 -DbatchSize=500 -Dparallelism=8
   ```
   `subscriberCount`/`batchSize`/`parallelism` are system-property-overridable so the same class
   serves both small-scale tuning runs and the full 100K validation.

### Batching/parallelism tuning applied

The original `RunBillCommandHandler` loaded every ACTIVE subscriber into one `List`, then processed
them one at a time in a single-threaded loop. Critically, the platform mediator's own
`TransactionBehavior` (`platform-core`, not modifiable) already wraps **every** command in one
transaction end to end â€” so the original implementation ran the entire 100K-subscriber bill-run as
one multi-hundred-thousand-row transaction on a single thread: no parallelism, and a single DB
connection held open (with no partial commits visible) for the full run.

Refactored into two classes, staying inside the existing Domain Orchestration mode (ADR-004) and the
existing outbox pattern (ADR-009) â€” no new architecture, no bypass of either:

- `BillRunCommandHandler` (`RunBillCommandHandler.java`) â€” thin orchestrator. Fetches active
  subscribers, partitions them into bounded batches (`telco.billing.bill-run.batch-size`, default
  500), and runs batches concurrently on a fixed worker pool
  (`telco.billing.bill-run.parallelism`, default 8), propagating the request's `CorrelationContext`
  onto each worker thread (ADR-012) since it is thread-bound and would otherwise be lost.
- `BillRunBatchProcessor` (new) â€” holds the exact per-subscriber generation logic that used to live
  in the handler (idempotency check, invoice+line generation, PDF render, storage, outbox publish),
  moved verbatim, with `processBatch(...)` now `@Transactional(propagation = REQUIRES_NEW)`. This
  deliberately suspends the mediator's own outer transaction and opens an independent one per batch â€”
  standard Spring behavior, safe to call from worker threads because Spring binds transactional
  resources per-thread. Each batch now commits independently instead of the whole run being one
  transaction, and batches run in parallel.

Duplicate-invoice safety is defense-in-depth, not just the application-level idempotency check: the
`uidx_invoices_sub_period` unique index on `(subscription_id, period_start)` (pre-existing in
`V1__billing.sql`) is a real database-level guarantee independent of the parallel-batch refactor.

Config added (`microservices/configs/billing-service/application.yml`, the actual runtime config â€”
the bootstrap `application.yml` under `src/main/resources` is intentionally left untouched per
ADR-010):
```yaml
spring.datasource.hikari.maximum-pool-size: 20   # parallelism (8) + mediator's own idle outer-tx connection + headroom
telco.billing.bill-run.batch-size: 500
telco.billing.bill-run.parallelism: 8
```

### Tuning process â€” measured, not guessed

| Scale | batchSize | parallelism | Elapsed | Throughput | Result |
| --- | --- | --- | --- | --- | --- |
| 1,000 | 200 | 4 | 3.9 s | ~250/s | generated=1000, skipped=0, all 4 workers active |
| 20,000 | 500 | 8 | 48.4 s | 413/s | generated=20000, skipped=0, all 8 workers active |
| **100,000** | **500** | **8** | **6m 20.34s (380,339 ms)** | **262.9/s** | **generated=100000, skipped=0, target PASS** |

Throughput at 100K (262.9/s) is lower than at 20K (413/s) â€” expected, from a larger `invoices` table
(idempotency-check and insert cost grow with table size) and from seeding/JVM warm-up effects at
scale; still comfortably inside the target with a **4.7x** margin, so no further tuning was pursued.
1,000/20,000 runs were used purely to validate correctness (idempotency, worker-thread parallelism
actually engaging) and calibrate before committing to the full 100K run, not as separate deliverables.

### Known limitation observed, not fixed (out of scope for this task)

`minio.enabled` is never set to `true` anywhere in `microservices/configs/billing-service/*.yml`
(dev/docker/staging/prod all omit it), and `LocalStorageService` defaults to active
(`@ConditionalOnProperty(matchIfMissing = true)`) whenever it is unset. This means
`MinioStorageService`/`MinioConfig` are effectively dead code in every configured environment today,
including the "prod" profile â€” PDFs are always written to local disk, not MinIO, regardless of
environment. This is a pre-existing gap, unrelated to this session's changes (confirmed no config
file anywhere sets it), and is flagged here for `devops`/`platform-engineer` rather than silently
fixed, since changing which storage backend activates in "prod" is a deployment-configuration
decision outside a performance-validation task's scope. It does not affect the validity of this
report: `LocalStorageService` is genuinely what runs today, so timing against it reflects real
current behavior, not an artificial test-only shortcut.
