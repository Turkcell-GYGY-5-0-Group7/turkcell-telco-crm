# AC-01 Onboarding Saga - Traceability (Sprint 09 Feature 9.5)

This document maps every hop of the AC-01 new-line onboarding saga - both the happy path and the
compensation path - to the EXISTING per-service Testcontainers test(s) that prove that hop, plus the
new outbox routing regression test that proves the TRANSPORT (Debezium routing + `eventType` header)
the whole saga rides on.

## Scope of this feature (SCOPED real-broker de-risk)

Feature 9.5 delivers the scoped, CI-friendly AC-01 proof. It does NOT boot multiple service Spring
contexts (separate apps + databases) in one JVM - that is impractical and low-signal. Instead each
saga hop is proven piecewise by its owning service's Testcontainers test (real Mediator pipeline,
real inbox/outbox, real state machine, Testcontainers Postgres), and the cross-service TRANSPORT that
stitches those hops together is proven by the new Debezium `EventRouter` routing regression test.

**Full-system compose acceptance - all services containerized, real Debezium connectors registered,
Keycloak-driven REST driving the flow through the gateway - is Sprint 14 / task 14.1**, per the
requirement traceability ("AC-01: 09 built, 14 validated"). This feature builds and de-risks; 14
validates the assembled system.

## Transport proof (the glue): outbox routing regression GATE

The saga is choreographed: each service publishes domain events to its transactional outbox, Debezium
captures the `outbox_event` insert and its `EventRouter` SMT routes the row to a Kafka topic derived
from the `aggregate_type` column (`route.by.field=aggregate_type`,
`route.topic.replacement=${routedByValue}.events`) and places `event_type` as the `eventType` header
that consumers filter on. A real-broker de-risk caught a saga-breaking bug: a PascalCase
`aggregate_type` (e.g. `"Subscription"`) routes to `Subscription.events`, a topic NO consumer
subscribes to, so the saga silently stalls. The fix (lowercase `OUTBOX_AGGREGATE_TYPE` constants;
PascalCase retained only for audit `aggregateType`) is in.

The pre-existing consumer tests could not catch that regression: they construct `ConsumerRecord`s with
hardcoded lowercase topics and never exercise Debezium routing. The new gate closes this by running the
REAL `io.debezium.transforms.outbox.EventRouter` SMT (configured identically to the production
connector `infra/docker/kafka-connect/connectors/outbox-connector.example.json`) against the
`aggregate_type` each producer ACTUALLY persists to its outbox, and asserting it routes to the
lowercase `<domain>.events` topic plus places the `eventType` header.

| Producer service | Real handler exercised (writes real outbox row) | Test |
| --- | --- | --- |
| subscription | `ActivateSubscriptionCommand` -> `subscription.activated.v1` (+ `msisdn.allocated.v1`) | `subscription-service` `com.telco.subscription.routing.OutboxRoutingRegressionTest` |
| order | `CreateOrderCommand` -> `order.created.v1` | `order-service` `com.telco.order.routing.OutboxRoutingRegressionTest` |
| payment | `ChargePaymentCommand` -> `payment.completed.v1` | `payment-service` `com.telco.payment.routing.OutboxRoutingRegressionTest` |

Each test reads the persisted `aggregate_type` back from `outbox_event` and feeds THAT value into the
SMT, so routing is proven DERIVED from the real persisted value. A PascalCase regression fails the
`..._routes_to_the_lowercase_..._events_topic` assertion; the companion
`a_pascalcase_aggregate_type_would_route_to_the_wrong_topic` test documents the exact failure mode
(routes to `<Pascal>.events`). Verified: flipping `CreateOrderCommandHandler.OUTBOX_AGGREGATE_TYPE`
to `"Order"` fails the order gate with `expected: "order" but was: "Order"`.

The routed topics match the consumers' `@KafkaListener(topics = ...)` subscriptions:

| Topic | Subscribed by |
| --- | --- |
| `order.events` | payment-service `OrderCreatedEventConsumer` |
| `payment.events` | subscription-service `PaymentCompletedEventConsumer` / `PaymentFailedEventConsumer`; order-service `PaymentCompletedEventConsumer` / `PaymentRefundedEventConsumer` |
| `subscription.events` | order-service `SubscriptionActivatedEventConsumer`; payment-service `SubscriptionActivationFailedEventConsumer` |

## Compensation path: activation fails -> refund -> order CANCELLED, no dangling MSISDN

| # | Saga hop | Proven by (test class :: method) | Service |
| --- | --- | --- | --- |
| C1 | Activation fails (MSISDN pool exhausted) -> emit `subscription.activation-failed.v1` keyed by orderId, and allocate NO number (no dangling MSISDN) | `SubscriptionActivationFailureTest::exhausted_pool_emits_activation_failed_and_allocates_no_number` (asserts 0 RESERVED, pool unchanged, failure event with reason `MSISDN_POOL_EXHAUSTED`) | subscription |
| C1' | Terminal pre-activation failures also emit `subscription.activation-failed.v1` and activate nothing (order 404, non-404 4xx, multi-item) | `PaymentCompletedConsumerTest::order_not_found_emits_activation_failed_and_does_not_activate`, `::order_lookup_rejected_4xx_emits_activation_failed_with_order_lookup_rejected`, `::multi_item_order_emits_activation_failed` | subscription |
| C2 | `subscription.activation-failed.v1` -> refund the COMPLETED payment -> emit `payment.refunded.v1` | `SubscriptionActivationFailedConsumerTest::activation_failed_refunds_the_completed_payment_and_emits_refunded_event` (+ event-type filter `::wrong_event_type_on_the_same_topic_is_ignored`, `::message_without_event_type_header_is_ignored`; safe no-op `::no_completed_payment_for_order_is_a_safe_no_op`); routing of the refund event: payment `OutboxRoutingRegressionTest::every`* covered via `payment.events` topic proof | payment |
| C3 | `payment.refunded.v1` -> order CANCELLED, saga COMPENSATED (system actor, ownership guard bypassed) | `SagaConsumerTest::payment_refunded_cancels_confirmed_order_and_advances_saga` (+ idempotency `::payment_refunded_redelivery_is_idempotent`, event-type filter `::payment_refunded_consumer_ignores_wrong_event_type`) | order |
| C4 | No dangling MSISDN after a full teardown: a terminated/never-allocated line releases its number to FREE | `SubscriptionIntegrationTest::termination_releases_msisdn_to_free_and_writes_released_outbox_row`; failure-path non-allocation: C1 above | subscription |

Note on C2 routing: the payment refund publishes `payment.refunded.v1` with `aggregate_type="payment"`,
proven to route to `payment.events` by the same payment routing gate that covers `payment.completed.v1`
(both share the `OUTBOX_AGGREGATE_TYPE = "payment"` constant); order-service's
`PaymentRefundedEventConsumer` subscribes to `payment.events`.
