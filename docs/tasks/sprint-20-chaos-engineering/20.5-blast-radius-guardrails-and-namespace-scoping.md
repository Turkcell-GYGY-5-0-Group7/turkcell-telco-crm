# 20.5 Blast-Radius Guardrails and Namespace Scoping

> Sprint: [Sprint 20 - Chaos Engineering](./README.md)
> Feature ID: 20.5
> Subtasks: 20.5.1, 20.5.2, 20.5.3

## Objective

Constrain Chaos Mesh's operational surface to the `telco` namespace with explicit, manual-only
execution, per the Sprint 20 README's Feature 20.5 decision: RBAC/namespace scoping, bounded
`duration` plus documented abort on every experiment, no scheduled/automatic chaos this sprint, and
no change to the CI/CD gate (ADR-014).

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- 20.1 (Chaos Mesh install, `deploy/chaos/values.yaml`)
- 20.3 (the three experiment manifests these guardrails apply to)
- ADR-014 (CI/CD and Deployment Strategy - the build/deploy gate this sprint must not touch)
- 20.5.1

## Subtasks

#### 20.5.1 RBAC and namespace scoping for the Chaos Mesh controller
- ID: 20.5.1
- Title: Restrict Chaos Mesh's RBAC and target-namespace scope to `telco`
- Description: Configure `deploy/chaos/values.yaml` (20.1.1) so the Chaos Mesh controller-manager's
  RBAC (`ClusterRole`/`Role` bindings the chart creates) and its default target-namespace scope are
  limited to `telco` - no cluster-wide `PodChaos`/`NetworkChaos` targeting capability beyond that
  namespace. Where the upstream chart only offers a cluster-scoped mode, document the accepted
  residual risk explicitly (this is a single-cluster, single-namespace platform per
  `deploy/helm/README.md` - there is no `production` namespace on this Kind cluster to accidentally
  target) and record the mitigating control: every 20.3 manifest's `selector.namespaces` is hard-set
  to `["telco"]` (verified in 20.3's acceptance criteria) regardless of the controller's own scope.
- Business Purpose: Prevents a chaos experiment from ever reaching a namespace this sprint does not
  intend it to (defense in depth beyond the per-manifest `selector`), directly implementing the
  README's "gated so experiments never run in a namespace tagged `production`" requirement.
- Inputs: `deploy/chaos/values.yaml` (20.1.1), README Feature 20.1 and 20.5 notes.
- Outputs: RBAC/namespace-scope settings in `deploy/chaos/values.yaml`; a documented residual-risk
  note in `deploy/chaos/README.md` if the chart's RBAC cannot be namespace-scoped below cluster-wide.
- Acceptance Criteria:
  - `kubectl auth can-i --list --as=system:serviceaccount:telco:chaos-controller-manager -n
    telco` (the chart's verified default service-account name, `values.yaml:63` of the vendored
    `chaos-mesh-2.8.3.tgz` chart) shows the controller's effective permissions, and the review
    confirms no target namespace other than `telco` exists on the cluster this permission could
    reach.
  - `deploy/chaos/README.md` states, in one paragraph, the RBAC scoping decision and any residual
    risk plus its mitigation.
- Dependencies: 20.1.1
- Complexity: M

#### 20.5.2 Bounded duration and abort-command audit
- ID: 20.5.2
- Title: Audit all 20.3 experiment manifests for a bounded `duration` and a documented abort command
- Description: Verify, as a checklist pass over the three manifests produced in 20.3, that every one
  sets an explicit, finite `spec.duration` (no manifest relies on an implicit/indefinite duration)
  and carries a header comment with the exact `kubectl delete -f <experiment>.yaml` abort command.
  Produce a short checklist section in `deploy/chaos/README.md` an operator or reviewer can use
  before any future experiment is added to `deploy/chaos/experiments/`.
- Business Purpose: Guarantees every experiment self-terminates even if an operator is not present
  to abort it manually, and that manual abort is always a copy-paste command away - the concrete
  implementation of the README's "every experiment manifest carries a bounded `duration` and a
  documented manual-abort command" rule.
- Inputs: The three 20.3 manifest files.
- Outputs: A "Guardrail checklist" section in `deploy/chaos/README.md` (duration bounded? abort
  command present? selector scoped to `telco`?) plus confirmation each 20.3 manifest passes it.
- Acceptance Criteria:
  - Each of the three 20.3 manifests has `spec.duration` set to a finite value (e.g. `"60s"`,
    `"5m"`) - `grep duration` across `deploy/chaos/experiments/*.yaml` shows no blank or missing
    field.
  - Each manifest's header comment contains the literal `kubectl delete -f` command for that exact
    filename.
- Dependencies: 20.3.1, 20.3.2, 20.3.3
- Complexity: S

#### 20.5.3 Manual-only execution and CI non-interference statement
- ID: 20.5.3
- Title: Confirm no scheduled chaos and no CI/CD gate changes
- Description: Verify that no `Schedule` or `Workflow` Chaos Mesh resource is defined anywhere under
  `deploy/chaos/` (experiments in 20.3 are plain `PodChaos`/`NetworkChaos` objects applied manually,
  per the README's explicit "no scheduled/automatic chaos in this sprint" rule), and that no
  `.github/workflows/*.yml` file references `deploy/chaos/` in any job. Record both checks explicitly
  in `deploy/chaos/README.md` so a future sprint that adds scheduled game days must consciously
  revisit this guardrail rather than silently inherit manual-only assumptions.
- Business Purpose: Keeps this sprint's blast radius bounded to deliberate, observed, single-operator
  runs and leaves the existing CI/CD build-deploy gate (ADR-014, Sprint 15.4) completely unaffected,
  satisfying the README's explicit statement that "CI is untouched by this sprint."
- Inputs: `.github/workflows/` (existing CI/CD pipeline definitions from Sprint 15.4), README
  Feature 20.5 notes.
- Outputs: A "Manual-only, CI-untouched" confirmation note in `deploy/chaos/README.md`, including the
  grep/search commands used to verify both conditions (no `Schedule`/`Workflow` resource; no
  `.github/workflows` reference to `deploy/chaos/`).
- Acceptance Criteria:
  - `grep -rl "kind: Schedule\|kind: Workflow" deploy/chaos/` returns no matches.
  - `grep -rl "deploy/chaos" .github/workflows/` returns no matches.
  - Both grep commands and their expected (empty) output are documented in `deploy/chaos/README.md`.
- Dependencies: 20.1.1, 20.3.1, 20.3.2, 20.3.3
- Complexity: S
