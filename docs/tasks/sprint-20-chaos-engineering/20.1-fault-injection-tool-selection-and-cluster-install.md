# 20.1 Fault-Injection Tool Selection and Cluster Install

> Sprint: [Sprint 20 - Chaos Engineering](./README.md)
> Feature ID: 20.1
> Subtasks: 20.1.1, 20.1.2, 20.1.3

## Objective

Install Chaos Mesh - the tool selected in the Sprint 20 README's Feature notes (chosen over Litmus
and hand-rolled `kubectl`/`tc`/`iptables` scripts; see README Feature 20.1 for the full rationale,
not restated here) - into the `telco` namespace of the existing Kind cluster (Sprint 15,
`deploy/RUNBOOK.md` Section 2), following the same one-chart-per-concern Helm layout precedent as
`deploy/helm/dependencies` (ADR-014).

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- Sprint 15 (Kind cluster, `deploy/helm/` chart layout and install pattern, `deploy/RUNBOOK.md`)
- README Feature 20.1 tool-selection decision (Chaos Mesh - do not re-derive)
- 20.1.1

## Subtasks

#### 20.1.1 Chaos Mesh Helm release under `deploy/chaos/`
- ID: 20.1.1
- Title: Add a `deploy/chaos/` directory with a Chaos Mesh Helm release definition
- Description: Create `deploy/chaos/` following the same shape as `deploy/helm/dependencies`
  (`deploy/helm/README.md`): a values file (`deploy/chaos/values.yaml`) layered on the upstream
  Chaos Mesh chart (`https://charts.chaos-mesh.org`, chart `chaos-mesh/chaos-mesh`), scoped to
  install its controller-manager and chaos-daemon DaemonSet into the existing `telco` namespace
  (no new namespace - this is a single-cluster, single-namespace slice per the README Objective).
  Pin a chart version and container image tag explicitly (no `latest`), consistent with the
  image-tag-pinning discipline in `deploy/RUNBOOK.md` Section 4.1.2. Document the install command
  (`helm repo add chaos-mesh ...`, `helm upgrade --install chaos-mesh deploy/chaos -n telco -f
  deploy/chaos/values.yaml`) in a new `deploy/chaos/README.md` that mirrors the structure of
  `deploy/helm/README.md` (chart layout, install order, config notes).
- Business Purpose: Establishes the fault-injection control plane the 20.3 experiment manifests and
  20.4 game days depend on, without introducing a new cluster or namespace and without touching any
  `microservices/*` image (README rationale: cluster-level controller + daemonset, no service-image
  changes).
- Inputs: Sprint 20 README Feature 20.1 notes (Chaos Mesh decision + CRD list: `PodChaos`,
  `NetworkChaos`, `StressChaos`, `Schedule`, `Workflow`), `deploy/helm/README.md` (chart layout
  precedent), `deploy/RUNBOOK.md` Sections 2 and 4 (existing Kind cluster + install pattern).
- Outputs: `deploy/chaos/values.yaml`, `deploy/chaos/README.md`; no changes to `deploy/helm/*`.
- Acceptance Criteria:
  - `helm upgrade --install chaos-mesh deploy/chaos -n telco -f deploy/chaos/values.yaml` succeeds
    against the Sprint 15 Kind cluster with no changes to any existing `telco-deps` or service
    release.
  - `kubectl -n telco get pods -l app.kubernetes.io/instance=chaos-mesh` shows the
    controller-manager and chaos-daemon pods `Running`.
  - The chart version and image tag are pinned (no `latest`) in `deploy/chaos/values.yaml`.
- Dependencies: none (first task of Sprint 20)
- Complexity: M

#### 20.1.2 CRD verification
- ID: 20.1.2
- Title: Verify the Chaos Mesh CRDs required by 20.3's experiment library are installed
- Description: After 20.1.1's install, verify the `PodChaos`, `NetworkChaos`, and `Schedule` CRDs
  (the three needed by 20.3's pod-kill, latency-injection, and network-partition experiments, plus
  `Schedule` for future scheduled runs even though 20.5 forbids using it this sprint) are registered
  and can be listed with `kubectl api-resources | grep chaos-mesh`. `StressChaos` and `Workflow` are
  installed as part of the chart but are out of scope for 20.3's experiment types this sprint.
- Business Purpose: Confirms the control plane is functionally ready before any 20.3 experiment
  manifest is authored, catching a partial/broken install early rather than at game-day time.
- Inputs: 20.1.1 install output.
- Outputs: A verification note appended to `deploy/chaos/README.md` (the `kubectl api-resources`
  command and its expected output) that an operator can re-run after any future chart upgrade.
- Acceptance Criteria:
  - `kubectl get crds | grep chaos-mesh.org` lists at minimum `podchaos`, `networkchaos`, and
    `schedules`.
  - `kubectl explain podchaos.spec` and `kubectl explain networkchaos.spec` both resolve (proves the
    CRD schema, not just the CRD object, is registered).
- Dependencies: 20.1.1
- Complexity: S

#### 20.1.3 Dashboard access decision (optional Chaos Mesh UI)
- ID: 20.1.3
- Title: Decide and document whether the Chaos Mesh web dashboard is enabled
- Description: Chaos Mesh ships an optional web dashboard (separate from the Grafana dashboards
  reused in 20.2). Decide whether to enable it for this sprint's single-operator, `kubectl apply`
  driven workflow (README Feature 20.5: no scheduled/automatic chaos), and document the decision in
  `deploy/chaos/README.md`. Default recommendation: leave it **disabled** (`dashboard.enabled:
  false` in `deploy/chaos/values.yaml`) to keep the operational surface minimal, matching the
  README's stated reason for rejecting Litmus (avoiding an extra portal-style web app); an operator
  can port-forward to inspect experiment status via `kubectl describe podchaos/<name>` instead.
- Business Purpose: Keeps the chaos control plane's operational footprint minimal and consistent
  with the README's stated preference for CRD-native, `kubectl`-driven operation over an added web
  UI, while leaving a documented path to enable it later if a game day needs richer visualization.
- Inputs: README Feature 20.1 (Litmus-portal rejection rationale).
- Outputs: A short decision note in `deploy/chaos/README.md` and the corresponding
  `dashboard.enabled` value in `deploy/chaos/values.yaml`.
- Acceptance Criteria:
  - `deploy/chaos/values.yaml` explicitly sets `dashboard.enabled` (not left at chart default) and
    `deploy/chaos/README.md` states the reasoning in one paragraph.
- Dependencies: 20.1.1
- Complexity: S
