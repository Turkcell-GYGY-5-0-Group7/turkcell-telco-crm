# 20.3 Experiment Library: Pod-Kill, Latency Injection, Network Partition

> Sprint: [Sprint 20 - Chaos Engineering](./README.md)
> Feature ID: 20.3
> Subtasks: 20.3.1, 20.3.2, 20.3.3

## Objective

Author the three Chaos Mesh experiment manifests named in the Sprint 20 README's Feature 20.3 notes
- pod-kill (`PodChaos`), latency injection (`NetworkChaos`, delay), and network partition
(`NetworkChaos`, partition) - under a new `deploy/chaos/experiments/` directory, each targeting a
concrete domain service or dependency already deployed by Sprint 15, and each carrying the
steady-state hypothesis reference produced by 20.2.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- 20.1 (Chaos Mesh installed and CRDs verified)

## Subtasks

#### 20.3.1 Pod-kill experiment manifest
- ID: 20.3.1
- Title: Author a `PodChaos` manifest killing a domain-service pod
- Description: Create `deploy/chaos/experiments/pod-kill-order-service.yaml`, a `PodChaos` resource
  (`action: pod-kill`, `mode: one`) targeting a pod of `order-service` (chosen as a representative,
  HPA/PDB-managed domain service per `deploy/helm/README.md`'s HPA/PDB section: `minReplicas: 2`,
  PDB `minAvailable: 1`) in the `telco` namespace, with a bounded `duration` and a `selector`
  restricted to `namespace: telco` and the service's pod labels (no cluster-wide selector). Annotate
  the manifest (as a YAML comment header, since Chaos Mesh CRDs have no free-text hypothesis field)
  with the steady-state hypothesis and dashboard/panel from `deploy/chaos/STEADY-STATE.md` (20.2.1)
  and the manual-abort command (`kubectl delete -f
  deploy/chaos/experiments/pod-kill-order-service.yaml`) per the README's Feature 20.5 guardrail.
- Business Purpose: Validates, under a real fault rather than a unit test, that the PDB and HPA
  established in Sprint 15.3 actually keep a domain service serving and reschedule the killed pod -
  the Sprint 20 README's first Exit Criterion.
- Inputs: `deploy/chaos/STEADY-STATE.md` (20.2.1 hypothesis for this experiment),
  `docs/tasks/sprint-15-deployment/15.3-autoscaling-and-resilience.md` (PDB/HPA config this
  experiment exercises), README Feature 20.3 notes (experiment (a) definition).
- Outputs: `deploy/chaos/experiments/pod-kill-order-service.yaml`.
- Acceptance Criteria:
  - `kubectl apply -f deploy/chaos/experiments/pod-kill-order-service.yaml` on the Sprint 15 Kind
    cluster kills one `order-service` pod; `kubectl -n telco get pods -l
    app.kubernetes.io/name=order-service -w` shows a replacement pod scheduled and the PDB
    (`minAvailable: 1`) never violated (at least one
    `order-service` pod `Ready` throughout).
  - The manifest's `duration` is bounded (not indefinite/omitted) and a `kubectl delete -f` abort
    command is documented in the manifest header.
  - The manifest's `selector.namespaces` is exactly `["telco"]` - no cluster-wide or multi-namespace
    selector.
- Dependencies: 20.1.1, 20.2.1
- Complexity: M

#### 20.3.2 Latency injection experiment manifest (CORRECTED)
- ID: 20.3.2
- Title: Author a `NetworkChaos` delay manifest against order-service's real
  Resilience4j-guarded call (order-service -> customer-service)
- Correction note (verified in `deploy/chaos/STEADY-STATE.md`'s "Corrections to upstream
  task text", per `docs/tasks/lessons.md`'s 2026-06-23 rule that an assumption change is
  fixed at the source doc, not just the deliverable): this subtask originally named
  `order-service -> payment-service` as the Resilience4j-guarded pairing. That is
  factually wrong. `order-service <-> payment-service` is Kafka-only/asynchronous (no
  synchronous call, no circuit breaker to trip). Reading
  `microservices/order-service/src/main/java/com/telco/order/config/ResilienceConfig.java`
  shows order-service registers exactly two circuit breakers for its real synchronous
  outbound calls - `customer-service` and `product-catalog-service` - and no
  `slowCallDurationThreshold` is configured (only `failureRateThreshold(50)`,
  `slidingWindowSize(10)`, `waitDurationInOpenState(30s)`,
  `permittedNumberOfCallsInHalfOpenState(3)`). The corrected pairing is
  `order-service -> customer-service`, and the breaker trips via the failure-rate path
  (enough calls fail/time out across the 10-call sliding window to cross 50%), not a
  slow-call-duration threshold that does not exist in this config.
- Description: Create `deploy/chaos/experiments/latency-order-to-customer.yaml`, a
  `NetworkChaos` resource (`action: delay`) injecting latency on the outbound call path
  from `order-service` to `customer-service` (the corrected pairing above, already
  guarded by the `customer-service`-named circuit breaker per Sprint 13.4.1). Set the
  delay magnitude to a concrete, documented, operator-observable value grounded in the
  real config (there is no `slowCallDurationThreshold` to exceed, and no
  connect/read-timeout configured on `RestClientConfig.java`'s `customerRestClient`
  bean either - both absences are documented in the manifest header rather than an
  invented number being used), bounded `duration`, and the same hypothesis/abort
  annotation pattern as 20.3.1.
- Business Purpose: Validates that the circuit breaker opens/half-opens under real injected latency
  instead of the fault cascading upstream - the Sprint 20 README's second Exit Criterion, and the
  concrete proof that Sprint 13.4's resilience configuration works under fault, not just in a mocked
  unit test.
- Inputs: `deploy/chaos/STEADY-STATE.md` (20.2.1 hypothesis, referencing the `circuit-breakers`
  dashboard), `docs/tasks/sprint-13-observability-and-resilience/13.4-resilience.md` (breaker
  threshold configuration; note the corrected config has no slow-call-duration threshold, only
  failure-rate), README Feature 20.3 notes (experiment (b) definition, corrected pairing).
- Outputs: `deploy/chaos/experiments/latency-order-to-customer.yaml`.
- Acceptance Criteria:
  - `kubectl apply -f deploy/chaos/experiments/latency-order-to-customer.yaml` injects delay on the
    order-service -> customer-service path; the `circuit-breakers` Grafana panel (20.2.1) shows the
    `customer-service`-named breaker transition to `OPEN` or `HALF_OPEN` within the experiment's
    `duration` (via the failure-rate path, once injected delay causes enough calls to fail/time out
    across the 10-call sliding window), and `order-service` returns a mapped
    `DependencyFailureException`-based error rather than hanging or cascading a timeout to its own
    caller.
  - The injected delay value is documented against the real Resilience4j config (failure-rate
    threshold, sliding window size - not an invented slow-call-duration threshold, since none is
    configured) and against the absence of any configured HTTP client timeout on the customer
    RestClient, not an arbitrary number presented as if derived from a timeout that exists.
  - The manifest's `duration` is bounded and an abort command is documented.
- Dependencies: 20.1.1, 20.2.1, 20.3.1
- Complexity: L

#### 20.3.3 Network partition experiment manifest
- ID: 20.3.3
- Title: Author a `NetworkChaos` partition manifest isolating a service from Kafka
- Description: Create `deploy/chaos/experiments/partition-billing-service-kafka.yaml`, a
  `NetworkChaos` resource (`action: partition`) isolating `billing-service` from the `kafka`
  StatefulSet's pods/Service in the `telco` namespace (the README's Feature 20.3 experiment (c)),
  with a bounded `duration` after which Chaos Mesh automatically heals the partition, and the same
  hypothesis/abort annotation pattern as 20.3.1/20.3.2. The experiment must be able to run without
  data loss verification requiring any new tooling: verification uses existing outbox-table row
  counts (or existing outbox metrics if instrumented under Sprint 13.3.1) before/after, not a new
  probe.
- Business Purpose: Validates that the outbox/inbox pattern (ADR-009, ADR-019) recovers without
  message loss once a Kafka partition heals - the Sprint 20 README's third Exit Criterion, proving
  the platform's core eventual-consistency mechanism under a real network fault rather than only a
  Testcontainers-based integration test (ADR-013 Section 3).
- Inputs: `deploy/chaos/STEADY-STATE.md` (20.2.1 hypothesis, referencing the `kafka-billing-ops`
  dashboard's consumer-lag panel), ADR-009 (Event Driven Architecture), ADR-019 (outbox/inbox
  pattern), README Feature 20.3 notes (experiment (c) definition).
- Outputs: `deploy/chaos/experiments/partition-billing-service-kafka.yaml`.
- Acceptance Criteria:
  - `kubectl apply -f deploy/chaos/experiments/partition-billing-service-kafka.yaml` isolates
    `billing-service` from `kafka` for the manifest's bounded `duration`; the `kafka-billing-ops`
    dashboard's consumer-lag panel (20.2.1) shows lag rise during the partition and drain back to
    baseline within an operator-observable window after the partition heals.
  - A before/after row-count check on `billing-service`'s outbox table (or existing outbox-lag
    metric) shows no event permanently lost - every event produced during the partition is present
    in Kafka after recovery.
  - The manifest's `duration` is bounded and an abort command is documented.
- Dependencies: 20.1.1, 20.2.1, 20.3.1
- Complexity: L
