# 20.2 Steady-State Hypotheses and Observability Wiring

> Sprint: [Sprint 20 - Chaos Engineering](./README.md)
> Feature ID: 20.2
> Subtasks: 20.2.1, 20.2.2, 20.2.3

## Objective

Define, per experiment in the 20.3 library, a steady-state hypothesis expressed entirely in terms of
the three Sprint 13 Grafana dashboards (`platform-overview`, `kafka-billing-ops`,
`circuit-breakers`) and their existing panels/alert thresholds, per the README's Feature 20.2
decision to reuse rather than duplicate observability (extends ADR-012). No new dashboard, panel, or
alert is introduced by this sprint.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- Sprint 13.3 (Grafana dashboards: `platform-overview`, `kafka-billing-ops`, `circuit-breakers`)
- Sprint 13.4 (Resilience4j circuit breaker/retry/bulkhead - the breaker states 20.3's latency
  experiment observes)
- README Feature 20.2 decision (reuse-only observability surface - do not re-derive)
- 20.2.1

## Subtasks

#### 20.2.1 Steady-state hypothesis mapping table
- ID: 20.2.1
- Title: Author a hypothesis-to-dashboard-panel mapping for each 20.3 experiment
- Description: Produce a table (one row per 20.3 experiment: pod-kill, latency injection, network
  partition) stating, for each: the steady-state hypothesis in plain language (e.g. "p95 latency on
  `order-service` stays under its NFR-07 budget" per the README's example), the exact
  dashboard + panel that shows it (`platform-overview` p95 latency panel, `kafka-billing-ops`
  consumer-lag panel, or `circuit-breakers` breaker-state panel - Sprint 13.3.2), and the existing
  alert rule (Sprint 13.3.3) that would fire if the hypothesis is violated. This mapping is the
  observability contract the 20.3 manifests and 20.4 runbook both reference; it is documentation
  only in this sprint (no manifest or runbook content is authored here).
- Business Purpose: Makes each experiment falsifiable against a metric that already exists,
  satisfying the README's "no experiment requires a new metric or panel to be interpreted" exit
  criterion, and gives 20.4's runbook a single source to cite per experiment.
- Inputs: `docs/tasks/sprint-13-observability-and-resilience/13.3-metrics-dashboards-alerts.md`
  (panel/alert definitions), `docs/tasks/sprint-13-observability-and-resilience/13.4-resilience.md`
  (breaker semantics), Sprint 20 README Feature 20.2 and 20.3 notes (the three experiment types and
  their example hypotheses).
- Outputs: `deploy/chaos/STEADY-STATE.md` - a table with columns: Experiment, Steady-State
  Hypothesis, Dashboard, Panel, Alert Rule, Abort Threshold.
- Acceptance Criteria:
  - The table has exactly one row per 20.3 experiment (pod-kill, latency injection, network
    partition), each row naming one of the three existing Sprint 13 dashboards and citing an actual
    panel/alert from `13.3-metrics-dashboards-alerts.md` (not a placeholder).
  - No row proposes a new panel, alert, or metric.
- Dependencies: none (first task of this feature)
- Complexity: M

#### 20.2.2 Dashboard reachability verification
- ID: 20.2.2
- Title: Verify the three Sprint 13 dashboards are reachable and populated on the target cluster
- Description: Confirm, against the Kind cluster this sprint's experiments will run on
  (`deploy/RUNBOOK.md` Section 10), that Grafana is reachable via
  `kubectl -n telco port-forward svc/grafana 3000:3000` and that `platform-overview`,
  `kafka-billing-ops`, and `circuit-breakers` all render with live (non-empty) data before any
  chaos experiment is run. This is a pre-flight check, not new provisioning - the dashboards
  themselves were provisioned in Sprint 13.3.2 and are out of scope to modify.
- Business Purpose: Prevents a game day from being blocked mid-run by a missing/stale dashboard;
  establishes the pre-experiment baseline the steady-state hypothesis in 20.2.1 is compared against.
- Inputs: `deploy/RUNBOOK.md` Section 10 (Observability access), 20.2.1 mapping table.
- Outputs: A "Pre-flight check" section appended to `deploy/chaos/STEADY-STATE.md` documenting the
  port-forward command and what "populated" looks like for each of the three dashboards.
- Acceptance Criteria:
  - The pre-flight section lists a concrete, runnable command sequence (port-forward + URL) an
    operator can execute before any 20.3 experiment.
  - No dashboard JSON, panel, or Grafana provisioning file under `deploy/helm/dependencies` is
    modified by this task.
- Dependencies: 20.2.1
- Complexity: S

#### 20.2.3 Baseline metric snapshot procedure
- ID: 20.2.3
- Title: Document how to capture a pre-experiment baseline snapshot
- Description: Define the procedure (a short PromQL query list or a documented Grafana panel
  screenshot/export step) for capturing a "before" snapshot of each steady-state metric immediately
  before running a 20.3 experiment, so the 20.4 post-game-day template has a concrete before/after
  comparison rather than a subjective judgment call.
- Business Purpose: Turns "the system stayed healthy" from an impression into an evidence-backed
  comparison, consistent with ADR-013's testing-strategy emphasis on verifiable outcomes.
- Inputs: 20.2.1 mapping table, Prometheus endpoint from `deploy/RUNBOOK.md` Section 10.
- Outputs: A "Baseline capture" section in `deploy/chaos/STEADY-STATE.md` with one PromQL query per
  mapped panel (e.g. the p95 latency query backing the `platform-overview` panel).
- Acceptance Criteria:
  - Each of the three mapped panels in 20.2.1 has a corresponding PromQL query documented that an
    operator can run via `kubectl -n telco port-forward svc/prometheus 9090:9090` before and after
    an experiment.
- Dependencies: 20.2.1, 20.2.2
- Complexity: S
