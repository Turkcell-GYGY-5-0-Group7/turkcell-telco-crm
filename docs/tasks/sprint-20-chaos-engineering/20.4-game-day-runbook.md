# 20.4 Game-Day Runbook

> Sprint: [Sprint 20 - Chaos Engineering](./README.md)
> Feature ID: 20.4
> Subtasks: 20.4.1, 20.4.2, 20.4.3

## Objective

Author `deploy/chaos/GAMEDAY-RUNBOOK.md`, an operational document parallel in spirit and structure
to `deploy/RUNBOOK.md` (Sprint 15.5), that lets a single operator run a full chaos game day -
prerequisites, running each 20.3 experiment, watching the 20.2 dashboards, and recording outcomes -
start to finish without consulting any other document.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- 20.1 (Chaos Mesh installed)
- 20.2 (steady-state hypotheses and dashboard mapping, `deploy/chaos/STEADY-STATE.md`)
- 20.3 (the three experiment manifests this runbook runs)
- `deploy/RUNBOOK.md` (Sprint 15.5) as the structural and tone template
- 20.4.1

## Subtasks

#### 20.4.1 Runbook prerequisites and experiment execution sections
- ID: 20.4.1
- Title: Write the prerequisites and per-experiment execution walkthrough
- Description: Author the first part of `deploy/chaos/GAMEDAY-RUNBOOK.md`: a Prerequisites section
  (tool versions - reuses `deploy/RUNBOOK.md` Section 1's table plus Chaos Mesh from 20.1; cluster
  must already be deployed per `deploy/RUNBOOK.md` Sections 2-4) and one subsection per 20.3
  experiment (pod-kill, latency injection, network partition) each containing: the exact `kubectl
  apply -f deploy/chaos/experiments/<file>.yaml` command, where to watch (the specific Grafana
  dashboard + panel from `deploy/chaos/STEADY-STATE.md`, reached via the `deploy/RUNBOOK.md` Section
  9 port-forward), the steady-state hypothesis being tested, and the abort criteria (what observed
  behavior means "stop now" and the exact `kubectl delete -f` command to do so).
- Business Purpose: Gives a single operator, without prior chaos-engineering experience on this
  platform, a literal step-by-step script to run any of the three 20.3 experiments safely - directly
  satisfying the README's fifth Exit Criterion ("A single operator can run a full game day by
  following `deploy/chaos/GAMEDAY-RUNBOOK.md` alone").
- Inputs: `deploy/RUNBOOK.md` (structure/tone template, Sections 1, 2, 9), `deploy/chaos/README.md`
  (20.1, install verification), `deploy/chaos/STEADY-STATE.md` (20.2, hypotheses and dashboards),
  the three 20.3 manifest files (exact filenames and abort commands).
- Outputs: `deploy/chaos/GAMEDAY-RUNBOOK.md` Sections 1 (Prerequisites) and 2 (Run an experiment),
  with one subsection per experiment (2.1 pod-kill, 2.2 latency injection, 2.3 network partition).
- Acceptance Criteria:
  - Each experiment subsection contains a copy-pasteable `kubectl apply` command, the dashboard URL
    reachable via the documented port-forward, the stated hypothesis (verbatim or summarized from
    20.2.1's table), and a copy-pasteable abort command.
  - The Prerequisites section lists every tool an operator needs beyond `deploy/RUNBOOK.md`'s base
    set (i.e., nothing beyond Chaos Mesh's CRDs already installed by 20.1 - no new CLI tool is
    introduced).
- Dependencies: 20.1.2, 20.2.1, 20.3.1, 20.3.2, 20.3.3
- Complexity: M

#### 20.4.2 Post-game-day findings template
- ID: 20.4.2
- Title: Add a structured post-game-day findings template
- Description: Author a findings template section in `deploy/chaos/GAMEDAY-RUNBOOK.md` (or a
  separate `deploy/chaos/GAMEDAY-FINDINGS-TEMPLATE.md` linked from it) capturing, per experiment run:
  date/operator, experiment name, steady-state hypothesis, observed result (pass/fail against the
  hypothesis), what alerted (from the 20.2.1 alert-rule column) versus what did not, any manual
  abort invoked and why, and follow-up actions with an owner. This is a template only - no filled-in
  findings are produced by this sprint since no experiment has been run yet (README status: TODO,
  nothing built).
- Business Purpose: Turns each game day into a repeatable, comparable record rather than tribal
  knowledge, and gives future sprints (or a later P9 multi-region chaos phase) a consistent input
  format to build on.
- Inputs: README Feature 20.4 notes ("a post-game-day template - what broke, what alerted, what did
  not, follow-up actions"), 20.2.1's alert-rule column (what "what alerted" refers to).
- Outputs: The findings template section/file, with a blank example row per experiment type
  demonstrating the expected format.
- Acceptance Criteria:
  - The template has a field for each of: date, operator, experiment, hypothesis, pass/fail, alerts
    fired, alerts expected-but-not-fired, manual abort (yes/no + reason), follow-up actions + owner.
  - The template is explicitly marked as unfilled/example-only (no fabricated game-day results).
- Dependencies: 20.4.1
- Complexity: S

#### 20.4.3 Cross-link into `deploy/RUNBOOK.md` and Sprint 20 README
- ID: 20.4.3
- Title: Add a pointer from the main operations runbook to the game-day runbook
- Description: Add a one-line cross-reference from `deploy/RUNBOOK.md`'s Section 9 (Observability)
  or a new brief "Chaos game days" note pointing to `deploy/chaos/GAMEDAY-RUNBOOK.md`, so an operator
  reading the primary runbook can discover the chaos runbook exists, without duplicating any chaos
  content into `deploy/RUNBOOK.md` itself. This subtask touches `deploy/RUNBOOK.md`; all other
  Sprint 20 subtasks are confined to `deploy/chaos/` and `docs/tasks/sprint-20-chaos-engineering/`.
- Business Purpose: Discoverability - an operator following the primary deployment runbook should
  not need out-of-band knowledge that a game-day runbook exists.
- Inputs: `deploy/RUNBOOK.md` (current Section 9), 20.4.1 output (`deploy/chaos/GAMEDAY-RUNBOOK.md`
  path).
- Outputs: A one- to two-line addition to `deploy/RUNBOOK.md` Section 9 or its "Related documents"
  list at the top.
- Acceptance Criteria:
  - `deploy/RUNBOOK.md` contains a link to `deploy/chaos/GAMEDAY-RUNBOOK.md` and no other content
    from `deploy/chaos/` is duplicated into it.
- Dependencies: 20.4.1
- Complexity: S
