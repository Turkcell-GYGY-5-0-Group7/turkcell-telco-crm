# 21.4 Campaign Eventing: Outbox (Lifecycle) + Inbox (Order/Tariff Consumption)

> Sprint: [Sprint 21 - Campaign / Catalog Validation](./README.md)
> Feature ID: 21.4
> Subtasks: 21.4.1, 21.4.2, 21.4.3

## Objective

Wire campaign-service into the Kafka/Avro event backbone (ADR-009, ADR-019): publish campaign
lifecycle events via the outbox, and consume order-confirmation/cancellation events (to durably
commit or release a `CampaignRedemption`) and tariff events (to defensively flag a campaign whose
target tariff was retired/repriced) via the inbox, per
[design-note.md Section 4](design-note.md#4-catalog-validation-model) "Where event-driven IS used."

## Dependencies

- [21.2](21.2-campaign-domain-eligibility-and-limits.md) (`CampaignRedemption.reserve/confirm/release`
  behavior, redemption-cap counting)
- [21.3](21.3-campaign-validation-api-and-order-integration.md) subtask 21.3.3 (the
  `order_items.campaign_id` correlation column this feature's consumers rely on to know which
  campaign an order/redemption belongs to)
- ADR-009 Event Driven Architecture, ADR-019 Event Contract and Schema Governance
- `com.telco.platform.outbox.OutboxService`, `com.telco.platform.inbox.*` (via `starter-outbox`/
  `starter-inbox`)
- Existing consumer pattern to mirror: `microservices/order-service/src/main/java/com/telco/order/application/consumer/PaymentCompletedEventConsumer.java`
  (type-filter-first on the `eventType` header, dedicated consumer group per listener, inbox dedup)
- Existing outbox-publish pattern to mirror: `microservices/product-catalog-service/.../domain/event/TariffCreatedEvent.java`
  and order-service's `CreateOrderCommandHandler` (`outboxService.publish(...)`)

## Open items carried forward from design-note.md Section 7 (item 1 now RESOLVED by tech-lead)

1. **Which order-service event means "the order is real" for redemption commit - RESOLVED.** Tech-lead
   ratified ADR-027 on 2026-07-13 (Section 4 amendment) confirming exactly the recommendation below:
   consume `payment.completed.v1` (real, payment-service, `payment.events` topic) for the `CONFIRMED`
   transition - `order.confirmed.v1` is deferred/not produced (verified: no `.avsc`, no publish call
   site, `ConfirmOrderCommandHandler` Javadoc: "No order domain event is published: confirm is a local
   state change") and is not the trigger. The ratification also settled the `RESERVED`-creation timing
   (consume `order.created.v1`, subtask 21.4.3) and added a required `reservedUntil` reservation-expiry
   reaper (`starter-lock`/ADR-024, mirroring `subscription-service`'s MSISDN reaper) for `RESERVED`
   rows left stranded by an abandoned order. See ADR-027 Section 4 ratification note for the full
   reasoning. No further tech-lead confirmation needed for this item; implement as specified below.
2. **Whether campaign eligibility needs a `channel` attribute** (web/app/dealer/call-center) at MVP -
   still open, not addressed by the ADR-027 ratification above.
   `docs/tasks/sprint-21-campaign-catalog-validation/design-note.md` Section 7 defers this to
   ADVANCED.md Section 2.7 (BFF/channels, itself a separate deferred item). **Recommendation: do not
   add a `channel` column in this sprint** - the `Campaign`/`CampaignRedemption` schema (21.1.2) has
   no channel field, and adding one now would be scope creep beyond the ADR-027-approved aggregate
   shape. If channel-scoping is needed later, it is a new ADR/design-note revision, not a change made
   inside this task.

## Subtasks

#### 21.4.1 Campaign lifecycle outbox events

- ID: 21.4.1
- Title: Publish campaign lifecycle events via the transactional outbox
- Description: Extend the admin command handlers from
  [21.2.1](21.2-campaign-domain-eligibility-and-limits.md#2121-campaign-lifecycle-behavior-and-admin-command-handlers)
  to call `OutboxService.publish(aggregateType, aggregateId, eventType, payload)` atomically with each
  state transition (ADR-009 outbox pattern - never call Kafka directly). Candidate event types
  (`domain.event.v1` naming, ADR-019) for this sprint's scope: `campaign.created.v1`,
  `campaign.activated.v1`, `campaign.paused.v1`, `campaign.expired.v1`, `campaign.cancelled.v1`. These
  are a starting proposal consistent with the `tariff.created.v1`/`tariff.price-changed.v1` precedent
  in product-catalog-service and are **not individually named in ADR-027 or the design note** (which
  only says "outbox (campaign lifecycle)" at the README-row level) - finalize the exact set and Avro
  schemas with the `event-integration` agent and register them in Schema Registry + add rows to
  `docs/architecture/event-catalog.md` as part of implementation (both out of scope for this
  planning-only pass).
- Business Purpose: Gives downstream consumers (notification-service, future reporting) visibility
  into campaign state changes without polling campaign-service, matching the outbox convention every
  other CQRS + Mediator service in the platform already follows.
- Inputs: ADR-009, ADR-019, `microservices/product-catalog-service/src/main/java/com/telco/catalog/domain/event/TariffCreatedEvent.java`
  and its outbox-publish call site for the payload/publish-call shape to mirror.
- Outputs:
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/event/{CampaignCreatedEvent,CampaignActivatedEvent,CampaignPausedEvent,CampaignExpiredEvent,CampaignCancelledEvent}.java`
  - `OutboxService.publish(...)` call sites added to each of the 21.2.1 command handlers
  - `docs/architecture/event-catalog.md` new rows for the campaign lifecycle events (implementing
    agent's responsibility at build time, coordinated with `event-integration`)
- Acceptance Criteria:
  - Activating a campaign produces exactly one outbox row with event type `campaign.activated.v1` in
    the same DB transaction as the status-column update (verified by an integration test asserting
    outbox-row count, matching the pattern of order-service/product-catalog-service outbox tests).
  - No direct `KafkaTemplate`/producer usage anywhere in campaign-service - only `OutboxService`.
- Dependencies: 21.2.1
- Complexity: M

#### 21.4.2 Redemption commit and release consumers

- ID: 21.4.2
- Title: Consume the "order is real" event and `order.cancelled.v1` to commit/release redemptions
- Description: Implement two `@KafkaListener` consumers mirroring
  `PaymentCompletedEventConsumer`'s type-filter-first, dedicated-consumer-group, inbox-dedup pattern:
  (a) a redemption-commit consumer on whichever topic/event type is settled by **open item 1 above**
  (recommended: `payment.events` / `payment.completed.v1`), which looks up the `CampaignRedemption`
  by `orderId` (populated when the `RESERVED` row was created - see the 21.2.2 open item on reservation
  timing, which this consumer's sibling, 21.4.3, resolves) and calls `CampaignRedemption.confirm()`;
  (b) an `order.cancelled.v1` consumer on order-service's order-events topic that looks up the
  `CampaignRedemption` by `orderId` and calls `CampaignRedemption.release()`. Both consumers use
  `InboxService`/`IdempotentRequest` (via `starter-inbox`) so Kafka redelivery cannot double-confirm or
  double-release a redemption (ADR-009/ADR-019 idempotent consumption rule). A redemption-commit event
  for an `orderId` with no matching `CampaignRedemption` row (e.g. the order had no campaign applied)
  is a silent no-op, not an error.
- Business Purpose: Implements the durable half of the redemption bookkeeping - "a customer must not
  redeem past their cap even under concurrent order attempts" - by only ever counting a redemption
  once the order it belongs to is confirmed real (payment succeeded) or explicitly releasing it if the
  order is cancelled, exactly as ADR-027 Decision Section 4 specifies.
- Inputs: [design-note.md Section 4](design-note.md#4-catalog-validation-model), "Where event-driven IS
  used" bullets; `microservices/order-service/src/main/java/com/telco/order/application/consumer/PaymentCompletedEventConsumer.java`
  (pattern to mirror, including its distinct-consumer-group lesson - do not share a `groupId` with any
  other campaign-service listener on the same topic).
- Outputs:
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/consumer/RedemptionCommitEventConsumer.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/consumer/OrderCancelledEventConsumer.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/dto/{PaymentCompletedPayload,OrderCancelledPayload}.java`
    (or reuse-by-copy of the relevant fields, since campaign-service cannot import order-service's
    internal DTOs across the service boundary)
- Acceptance Criteria:
  - The redemption-commit event for an `orderId` with a `RESERVED` `CampaignRedemption` row
    transitions it to `CONFIRMED`, idempotently (redelivery does not double-count).
  - `order.cancelled.v1` for an `orderId` with a `RESERVED` `CampaignRedemption` row transitions it to
    `RELEASED`, freeing the cap slot for a subsequent order.
  - An event for an `orderId` with no matching redemption row logs and returns without throwing.
- Dependencies: 21.2.2, 21.3.3, open item 1 (resolved)
- Complexity: L

#### 21.4.3 Redemption reservation trigger and tariff-defensive consumers

- ID: 21.4.3
- Title: Create the `RESERVED` redemption row and consume tariff events defensively
- Description: Resolve the reservation-timing open item flagged in
  [21.2.2](21.2-campaign-domain-eligibility-and-limits.md#2122-redemption-cap-enforcement-campaignredemption):
  add a consumer for `order.created.v1` (published by order-service's `CreateOrderCommandHandler`,
  already carrying `OrderItemPayload.campaignId` once 21.3.3 lands) that creates a `RESERVED`
  `CampaignRedemption` row per campaign-priced order item, keyed by `orderId`. This is what makes the
  per-customer/total redemption-cap race-safety in 21.2.2 real: a `RESERVED` row exists (and is
  counted) from the moment the order is created, not only once payment completes. Also implement two
  defensive consumers per
  [design-note.md Section 4](design-note.md#4-catalog-validation-model): `tariff.created.v1` and
  `tariff.price-changed.v1`, both consumed **not to mirror tariff pricing data** (ADR-027 explicitly
  forbids that - campaign-service stores only tariff codes) but to detect that a campaign's
  `applicable_tariff_codes` entry references a tariff that no longer exists in its expected form, and
  flag or auto-expire the affected campaign (implementer's choice between a logged warning + admin
  flag column vs. auto-`expire()` - document whichever is chosen in
  `docs/api-contracts/campaign-service.md`).
- Business Purpose: Closes the reservation-timing gap so 21.4.2's cap-safety claim is actually true at
  runtime, and gives campaign-service an early-warning mechanism instead of campaigns silently
  referencing a retired tariff until a customer hits a confusing 404 at order time.
- Inputs: [design-note.md Section 4](design-note.md#4-catalog-validation-model) (tariff-defensive
  consumption bullet), `microservices/order-service/src/main/java/com/telco/order/application/event/OrderCreatedEvent.java`
  (payload shape, once 21.3.3's `campaignId` field lands), `docs/architecture/event-catalog.md` rows
  for `tariff.created.v1`/`tariff.price-changed.v1`.
- Outputs:
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/consumer/OrderCreatedRedemptionReservationConsumer.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/consumer/{TariffCreatedEventConsumer,TariffPriceChangedEventConsumer}.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/model/Campaign.java`
    (a defensive flag/auto-expire path added if that implementation choice is taken)
- Acceptance Criteria:
  - An `order.created.v1` event with a non-null `campaignId` on one of its items creates exactly one
    `RESERVED` `CampaignRedemption` row for that `(campaignId, customerId, orderId)`, idempotently.
  - Two concurrent orders each consuming the last available redemption slot (simulated via two
    `order.created.v1` events processed near-simultaneously) result in only one `RESERVED` row within
    cap - proving 21.2.2's cap-safety claim end-to-end through the event path, not just at the
    synchronous validate call.
  - `tariff.price-changed.v1` for a tariff code referenced by an ACTIVE campaign results in an
    observable flag/log/auto-expire action (per whichever behavior is chosen), not silent ignoring.
- Dependencies: 21.3.3, 21.4.2
- Complexity: L

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met, and both open items (order-is-real event choice,
channel-attribute deferral) have been explicitly resolved rather than silently assumed.

## Deliverables

- Campaign lifecycle outbox events (`campaign.created.v1` and related, exact set finalized with
  `event-integration`).
- Redemption reservation (`order.created.v1`), commit (payment/order-confirm event), and release
  (`order.cancelled.v1`) inbox consumers, all idempotent via `starter-inbox`.
- Defensive `tariff.created.v1`/`tariff.price-changed.v1` consumers flagging dangling campaign
  references.

  (coordinated with `event-integration`).
- Build and tests green per the global conventions in [../README.md](../README.md) Section 3.
