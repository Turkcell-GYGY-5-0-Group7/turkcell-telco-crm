# 21.2 Campaign Domain: Eligibility Rules, Redemption Limits, Validity Windows

> Sprint: [Sprint 21 - Campaign / Catalog Validation](./README.md)
> Feature ID: 21.2
> Subtasks: 21.2.1, 21.2.2, 21.2.3

## Objective

Implement the `Campaign`/`CampaignRedemption` domain behavior - lifecycle transitions, validity-window
checks, redemption-cap enforcement, and eligibility evaluation - on top of the bare entities scaffolded
in [21.1](21.1-campaign-service-scaffold-and-schema.md), plus the CQRS + Mediator command/query
handlers needed to create and administer campaigns so that
[21.3](21.3-campaign-validation-api-and-order-integration.md) has real data to validate against. This
feature is domain-layer only: the campaign-validation REST endpoint that order-service calls is
21.3's scope; the outbox/inbox event wiring is 21.4's scope.

## Dependencies

- [21.1](21.1-campaign-service-scaffold-and-schema.md) (`Campaign`/`CampaignRedemption` entities,
  repositories, `campaign-db` schema)
- ADR-027 Decision Section 2 (CQRS + Mediator mode) and Section 4 (catalog validation model)
- [design-note.md Section 5](design-note.md#5-campaign-limits-aggregate-shape) (aggregate shape,
  redemption-cap counting rule)
- `com.telco.platform.cqrs.*` / `com.telco.platform.mediator.Mediator` (via `starter-mediator`)
- Reference shape: `microservices/product-catalog-service/src/main/java/com/telco/catalog/domain/model/Tariff.java`
  (framework-independent behavior + JPA annotations pattern) and
  `.../domain/service/TariffVersioningService.java` (domain service pattern)

## Subtasks

#### 21.2.1 Campaign lifecycle behavior and admin command handlers

- ID: 21.2.1
- Title: Add lifecycle transition methods to `Campaign` and the admin CQRS commands that drive them
- Description: Extend the bare `Campaign` entity (21.1.2) with behavior methods enforcing the
  `DRAFT -> ACTIVE -> PAUSED -> EXPIRED -> CANCELLED` state machine
  ([design-note.md Section 5](design-note.md#5-campaign-limits-aggregate-shape)):
  `activate()` (DRAFT/PAUSED -> ACTIVE, requires `validFrom`/`validTo` set and `validTo > validFrom`,
  mirroring `Tariff.create`'s `effectiveTo > effectiveFrom` invariant), `pause()` (ACTIVE -> PAUSED),
  `cancel()` (any non-terminal state -> CANCELLED), and `expire()` (ACTIVE/PAUSED -> EXPIRED, called
  both explicitly and defensively when `validTo` has passed - see 21.2.3's validity-window check).
  Each illegal transition throws `BusinessRuleException` (platform-provided, do not hand-roll - see
  `docs/architecture/platform-capabilities.md` Section 1). Add the Mediator commands/handlers to
  create and administer a campaign: `CreateCampaignCommand`/Handler, `ActivateCampaignCommand`/Handler,
  `PauseCampaignCommand`/Handler, `CancelCampaignCommand`/Handler. Add a minimal admin
  `CampaignController` (`POST /api/v1/campaigns`, `POST /api/v1/campaigns/{id}/activate`,
  `POST /api/v1/campaigns/{id}/pause`, `DELETE /api/v1/campaigns/{id}`, `GET /api/v1/campaigns/{id}`,
  `GET /api/v1/campaigns`) wrapping every response in `ApiResult<T>` (ADR-015) - this is the minimum
  surface needed for a campaign to exist and reach ACTIVE so the validate flow (21.3) and the sprint's
  exit criteria (`README.md`) are exercisable end-to-end; it is not a full campaign-marketing admin UI
  (out of scope, no such UI is planned this sprint).
- Business Purpose: Without a way to create and activate a campaign, the validation API (21.3) and
  redemption eventing (21.4) have no data to operate against; this closes that gap while staying
  inside the single-aggregate, single-transaction CQRS + Mediator shape ADR-027 Section 2 mandates.
- Inputs: [design-note.md Section 5](design-note.md#5-campaign-limits-aggregate-shape), ADR-015 (API
  standards, `ApiResult<T>`), ADR-008 (mediator-only business logic, no logic in controllers),
  `docs/architecture/platform-capabilities.md` Section 1 (`BusinessRuleException`, `ApiResult`).
- Outputs:
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/model/Campaign.java`
    (behavior methods added)
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/command/{Create,Activate,Pause,Cancel}CampaignCommand.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/handler/{Create,Activate,Pause,Cancel}CampaignCommandHandler.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/query/{GetCampaign,ListCampaigns}Query.java` + handlers
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/dto/{CreateCampaignRequest,CampaignResponse}.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/api/CampaignController.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/api/CampaignSecurityConfig.java`
    (admin-role authorization, mirroring `CatalogSecurityConfig`)
- Acceptance Criteria:
  - An illegal transition (e.g. `activate()` on a `CANCELLED` campaign, or `activate()` with no
    `validTo`) throws `BusinessRuleException`, not a generic exception.
  - `POST /api/v1/campaigns` -> `POST /api/v1/campaigns/{id}/activate` produces a campaign in status
    `ACTIVE`, retrievable via `GET /api/v1/campaigns/{id}`, wrapped in `ApiResult<T>`.
  - No business logic lives in `CampaignController` - it only calls `Mediator.send`/`Mediator.query`
    (ADR-008).
- Dependencies: 21.1.2
- Complexity: M

#### 21.2.2 Redemption-cap enforcement (`CampaignRedemption`)

- ID: 21.2.2
- Title: Implement per-customer and total redemption-cap counting against `CampaignRedemption`
- Description: Implement a domain service (e.g. `CampaignEligibilityService` or a same-package
  `RedemptionCapEvaluator`) that counts `CampaignRedemption` rows to enforce the two caps from
  [design-note.md Section 5](design-note.md#5-campaign-limits-aggregate-shape):
  `perCustomerRedemptionCap` (count `CONFIRMED` + `RESERVED` rows for `(campaignId, customerId)`) and
  `totalRedemptionCap` (count `CONFIRMED` + `RESERVED` rows for `campaignId`, skipped entirely when
  `totalRedemptionCap` is `null` = unlimited). Counting `RESERVED` alongside `CONFIRMED` is
  load-bearing: it is what prevents two concurrent in-flight orders from both being told "eligible"
  and jointly exceeding the cap before either confirms (design-note.md Section 5, "to prevent a race
  between two concurrent orders"). Add repository query methods
  (`countByCampaignIdAndCustomerIdAndStatusIn`, `countByCampaignIdAndStatusIn`) backed by the indexes
  created in 21.1.2. Use the `Campaign.version` optimistic-lock column (21.1.2) or a
  `SELECT ... FOR UPDATE`-equivalent (Spring Data `@Lock`) on the redemption-count read/decide/write
  path to keep the cap check race-safe under concurrent order attempts targeting the same campaign and
  customer.
  - **RESOLVED by tech-lead ADR-027 ratification (2026-07-13, Section 4 amendment)**: the `RESERVED`
    row is created by consuming `order.created.v1` in [21.4](21.4-campaign-eventing-outbox-inbox.md)
    (subtask 21.4.3), not by the synchronous `/campaigns/validate` call. `CONFIRMED` is set by
    consuming `payment.completed.v1` (not `order.confirmed.v1`, which is confirmed deferred/unproduced
    - see ADR-027 Section 4 ratification note), `RELEASED` by consuming `order.cancelled.v1`. A
    `reservedUntil`-based reaper (mirroring `subscription-service`'s MSISDN reaper, `starter-lock`/
    ADR-024) is now also required to release `RESERVED` rows left stranded by an abandoned order.
    Implement per ADR-027 Section 4 as ratified; no further tech-lead confirmation needed.
- Business Purpose: This is the sprint's core business rule - "an order placed against an ACTIVE
  campaign ... under its redemption caps is priced with the campaign discount" (Sprint 21 README exit
  criteria) - and the concurrency-safety half of it (no double-redemption past the cap) is what makes
  the feature production-credible rather than a happy-path demo.
- Inputs: [design-note.md Section 5](design-note.md#5-campaign-limits-aggregate-shape), ADR-027
  Decision Section 4 ("Redemption is not counted at validation time").
- Outputs:
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/service/CampaignEligibilityService.java`
    (or equivalently named per implementer's class-naming choice)
  - `microservices/campaign-service/src/main/java/com/telco/campaign/infrastructure/persistence/CampaignRedemptionRepository.java`
    (count/lock query methods added)
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/model/CampaignRedemption.java`
    (behavior methods: `reserve(...)`, `confirm()`, `release()`, each validating the current
    `RedemptionStatus` before transitioning)
- Acceptance Criteria:
  - A customer at `perCustomerRedemptionCap` is evaluated as ineligible for a further redemption of
    the same campaign.
  - A campaign at `totalRedemptionCap` is evaluated as ineligible for any further customer.
  - `totalRedemptionCap = null` never blocks on the total-cap check (unlimited case), unit-tested
    explicitly.
  - A concurrency test (two simulated simultaneous reservation attempts at `perCustomerRedemptionCap -
    1` remaining) results in exactly one succeeding, not both (see [21.5](21.5-tests.md)).
- Dependencies: 21.1.2, 21.2.1
- Complexity: L

#### 21.2.3 Validity-window and eligibility evaluation

- ID: 21.2.3
- Title: Implement validity-window checks and the combined eligibility decision
- Description: Implement the validity-window check (`validFrom <= now <= validTo`, `status == ACTIVE`)
  and the tariff-code applicability check (`tariffCode` present in the campaign's
  `applicable_tariff_codes`), and combine them with the redemption-cap checks (21.2.2) into a single
  eligibility evaluation method returning either an eligible decision (`campaignId`, `discountType`,
  `discountValue`) or an ineligible decision with a specific, distinguishable reason (`EXPIRED`,
  `NOT_YET_ACTIVE`, `NOT_ACTIVE_STATUS`, `TARIFF_NOT_APPLICABLE`, `PER_CUSTOMER_CAP_EXCEEDED`,
  `TOTAL_CAP_EXCEEDED`, `CAMPAIGN_NOT_FOUND`) - this method is the domain-layer piece that
  [21.3](21.3-campaign-validation-api-and-order-integration.md)'s controller/handler calls directly;
  it does not itself do HTTP or Mediator wiring. Also implement the defensive auto-expire check
  (`expire()` from 21.2.1 invoked when `now > validTo` is observed during evaluation, not only via an
  explicit admin action) so a campaign whose window has lapsed does not linger in `ACTIVE` status.
- Business Purpose: Directly implements the Sprint 21 exit criteria: "an order placed against an
  ACTIVE campaign within its validity window and under its redemption caps is priced with the campaign
  discount; an ineligible order is priced at the full catalog price with a clear reason surfaced."
- Inputs: [design-note.md Section 4](design-note.md#4-catalog-validation-model) (step 3: "evaluates
  eligibility ... and returns `{ eligible, discountType, discountValue, campaignId }` or
  `{ eligible: false, reason }`").
- Outputs:
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/service/CampaignEligibilityService.java`
    (validity-window + tariff-applicability logic added to the cap-checking service from 21.2.2 -
    same class, single cohesive eligibility evaluator)
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/model/EligibilityDecision.java`
    (or equivalent value type/record carrying the eligible/ineligible outcome and reason code)
- Acceptance Criteria:
  - A campaign with `validFrom` in the future is evaluated `NOT_YET_ACTIVE`; one with `validTo` in the
    past is evaluated `EXPIRED` (and is auto-transitioned out of `ACTIVE` if it was still marked so).
  - A tariff code not in `applicable_tariff_codes` is evaluated `TARIFF_NOT_APPLICABLE` regardless of
    validity window or caps.
  - Every ineligible outcome carries a specific reason code, never a bare boolean.
- Dependencies: 21.2.1, 21.2.2
- Complexity: M

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met.

## Deliverables

- `Campaign` lifecycle behavior (DRAFT/ACTIVE/PAUSED/EXPIRED/CANCELLED) with admin CQRS commands and
  a minimal `CampaignController`.
- Redemption-cap enforcement (`perCustomerRedemptionCap`, `totalRedemptionCap`) with race-safe
  counting of `CONFIRMED` + `RESERVED` rows.
- A single eligibility-evaluation entry point (validity window + tariff applicability + redemption
  caps) ready for 21.3's validate endpoint to call.

  assumed) before 21.4 is implemented.
