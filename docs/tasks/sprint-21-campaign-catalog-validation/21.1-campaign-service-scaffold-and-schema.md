# 21.1 campaign-service Scaffold and Schema

> Sprint: [Sprint 21 - Campaign / Catalog Validation](./README.md)
> Feature ID: 21.1
> Subtasks: 21.1.1, 21.1.2, 21.1.3

## Objective

Stand up `campaign-service` as a new ADR-017-compliant microservice with its own `campaign-db`
(PostgreSQL, ADR-006 database-per-service) and the bare persistence skeleton for the `Campaign` and
`CampaignRedemption` aggregates defined in
[design-note.md Section 5](design-note.md#5-campaign-limits-aggregate-shape). This feature delivers
scaffolding and schema only - no business rules (eligibility, redemption-cap enforcement, validity
windows: see [21.2](21.2-campaign-domain-eligibility-and-limits.md)) and no API/eventing wiring (see
[21.3](21.3-campaign-validation-api-and-order-integration.md) and
[21.4](21.4-campaign-eventing-outbox-inbox.md)).

## Dependencies

- ADR-027 Campaign and Catalog Validation (service boundary, port 9011, CQRS + Mediator mode,
  data ownership)
- ADR-017 Service Template Standard (`microservices/service-template`, mandatory starters)
- ADR-006 Database Strategy (database-per-service; no shared access to `campaign-db`)
- `docs/architecture/platform-capabilities.md` Section 2 "New-service checklist"
- Closest existing analogue for file/class shape: `microservices/product-catalog-service/`
  (CQRS + Mediator, single-aggregate-family service)

## Subtasks

#### 21.1.1 campaign-service skeleton from the ADR-017 template

- ID: 21.1.1
- Title: Scaffold `microservices/campaign-service/` from the service template with mandatory starters
- Description: Copy `microservices/service-template` (ADR-017) into `microservices/campaign-service/`.
  Set `groupId`/`artifactId` and base package `com.telco.campaign`, inherit `platform-bom` (never
  hardcode dependency versions, ADR-003/ADR-020). Add mandatory starters `starter-api`,
  `starter-security`, `starter-observability`, plus `starter-mediator` (CQRS + Mediator mode per
  ADR-027 Decision Section 2), `starter-outbox` (campaign lifecycle events, needed by
  [21.4](21.4-campaign-eventing-outbox-inbox.md)), and `starter-inbox` (idempotent consumption of
  order/tariff events, needed by 21.4). Create `CampaignServiceApplication.java`, `application.yml`
  (service name `campaign-service`, server port **9011** per ADR-027, config-server/discovery-server
  registration matching the pattern in `microservices/product-catalog-service/src/main/resources/application.yml`),
  `Dockerfile` (per Sprint 15 container pattern), `README.md`, and `CLAUDE.md` declaring
  `Architecture Mode: CQRS + MEDIATOR` (ADR-017 Section 3) and the Infrastructure Profile
  (ADR-006: transactional, per-customer-consistent - not cache-aside, contrast with
  product-catalog-service).
- Business Purpose: Establishes the new bounded context's deployable unit so campaign domain logic,
  API, and eventing (21.2-21.4) have a working service to build inside, per ADR-027's decision to
  isolate campaign/redemption data from product-catalog-service.
- Inputs: ADR-027, ADR-017, `microservices/service-template/`, `docs/architecture/platform-capabilities.md`
  Section 2, `microservices/product-catalog-service/` (naming/shape reference only - do not copy its
  business logic).
- Outputs:
  - `microservices/campaign-service/pom.xml` (platform-bom parent, starters above)
  - `microservices/campaign-service/src/main/java/com/telco/campaign/CampaignServiceApplication.java`
  - `microservices/campaign-service/src/main/resources/application.yml`
  - `microservices/campaign-service/Dockerfile`
  - `microservices/campaign-service/README.md`, `microservices/campaign-service/CLAUDE.md`
  - `docs/architecture/service-catalog.md` new `campaign-service (port 9011)` entry (implementing
    agent's responsibility at build time; out of scope for this planning pass)
- Acceptance Criteria:
  - `campaign-service` builds (`mvn -pl microservices/campaign-service verify`) and starts, registers
    with discovery-server, and reports `UP` on `/actuator/health`.
  - `CLAUDE.md`/`README.md` declares `Architecture Mode: CQRS + MEDIATOR` verbatim (ADR-017 Section 3).
  - No dependency on `platform-core` directly - only platform starters (ADR-018), verified by POM
    inspection.
- Dependencies: none (first task of Sprint 21)
- Complexity: M

#### 21.1.2 `campaign-db` schema and JPA entity skeletons

- ID: 21.1.2
- Title: Create `campaign-db` Flyway migration and bare `Campaign`/`CampaignRedemption` JPA entities
- Description: Add a new PostgreSQL 17 database `campaign-db` (ADR-006, no shared access from any
  other service) and its first Flyway migration creating the `campaigns` and `campaign_redemptions`
  tables per the aggregate shape in
  [design-note.md Section 5](design-note.md#5-campaign-limits-aggregate-shape). Set
  `spring.flyway.locations=classpath:db/migration,classpath:db/migration/platform` so the platform
  outbox/inbox tables apply alongside the domain schema (`docs/architecture/platform-capabilities.md`
  Section 2 item 5). Create the bare JPA entity classes (fields and column mappings only - no domain
  behavior methods; behavior is [21.2](21.2-campaign-domain-eligibility-and-limits.md)'s job) and
  Spring Data repositories.
  - `campaigns`: `id (uuid pk)`, `code (unique, not null)`, `name`, `description`,
    `discount_type (PERCENTAGE|FIXED_AMOUNT)`, `discount_value (numeric)`,
    `applicable_tariff_codes` (a normalized child table `campaign_tariff_codes(campaign_id, tariff_code)`
    is preferred over an array/CSV column for query-ability and referential clarity - implementer's
    call, either satisfies the aggregate shape), `valid_from`, `valid_to` (timestamps),
    `status (DRAFT|ACTIVE|PAUSED|EXPIRED|CANCELLED)`, `total_redemption_cap (nullable int)`,
    `per_customer_redemption_cap (int, default 1)`, `created_at`, `updated_at`, `version` (optimistic
    locking - concurrent redemption-cap checks in 21.2 rely on this).
  - `campaign_redemptions`: `id (uuid pk)`, `campaign_id (fk -> campaigns.id)`, `customer_id (uuid)`,
    `order_id (uuid, nullable until confirmed - see the order-correlation open item in
    [21.3](21.3-campaign-validation-api-and-order-integration.md) and
    [21.4](21.4-campaign-eventing-outbox-inbox.md))`, `status (RESERVED|CONFIRMED|RELEASED)`,
    `redeemed_at`, `confirmed_at (nullable)`.
  - Indexes: `campaigns(code)` unique, `campaign_redemptions(campaign_id, customer_id)` (per-customer
    cap lookups), `campaign_redemptions(campaign_id, status)` (total-cap counting),
    `campaign_redemptions(order_id)` (correlation lookups from 21.4's consumers).
- Business Purpose: Gives the Campaign/CampaignRedemption aggregate family a durable, correctly
  indexed home isolated from `product-catalog-db` and `order-db`, satisfying ADR-006's cross-service
  data-ownership rule before any domain logic is written.
- Inputs: [design-note.md Section 5](design-note.md#5-campaign-limits-aggregate-shape), ADR-006,
  `microservices/product-catalog-service/src/main/resources/db/migration/V1__catalog.sql` (migration
  naming/style reference), `microservices/product-catalog-service/src/main/java/com/telco/catalog/domain/model/Tariff.java`
  (entity mapping style reference).
- Outputs:
  - `microservices/campaign-service/src/main/resources/db/migration/V1__campaign.sql`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/model/Campaign.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/model/CampaignRedemption.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/model/CampaignStatus.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/model/DiscountType.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/domain/model/RedemptionStatus.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/infrastructure/persistence/CampaignRepository.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/infrastructure/persistence/CampaignRedemptionRepository.java`
- Acceptance Criteria:
  - `V1__campaign.sql` applies cleanly against a fresh `campaign-db` (Flyway migration test, mirroring
    `product-catalog-service`'s `CatalogSchemaMigrationTest` pattern).
  - Platform outbox/inbox tables (`outbox_events`, `inbox_*`) exist in `campaign-db` after startup,
    confirming the `classpath:db/migration/platform` location is wired.
  - No foreign key or shared-schema reference from `campaign-db` into `product-catalog-db` or
    `order-db` - `campaigns.applicable_tariff_codes` stores opaque tariff **codes**, never a copy of
    tariff price data (ADR-027 Decision Section 3).
  - `Campaign` and `CampaignRedemption` compile as plain JPA entities with no domain behavior methods
    yet (behavior deferred to 21.2 by design, so this task stays schema-scoped).
- Dependencies: 21.1.1
- Complexity: M

#### 21.1.3 Service registration and API-contract stub

- ID: 21.1.3
- Title: Register campaign-service in the platform's cross-cutting docs and stub its API contract
- Description: campaign-service is called synchronously and directly by order-service (mirroring
  `ProductCatalogServiceClient`'s direct-to-service call, not routed through api-gateway - the
  validate endpoint is an internal service-to-service call, ADR-005/ADR-011), so no gateway route is
  required for the MVP validate flow. Create the initial `docs/api-contracts/campaign-service.md`
  stub (port, mode, base path, auth notes) so [21.3](21.3-campaign-validation-api-and-order-integration.md)
  has a contract file to extend rather than create from scratch. If an admin-facing campaign
  management UI is added later (out of scope for Sprint 21), a gateway route and external auth model
  would be a separate, future task - note this explicitly as deferred.
- Business Purpose: Keeps the service catalog and API-contracts index consistent as new services are
  added (project convention across all prior sprints), and unblocks 21.2/21.3 from having to invent
  contract-file scaffolding mid-task.
- Inputs: `docs/architecture/service-catalog.md` (format reference, e.g. the `ticket-service` entry),
  `docs/api-contracts/order-service.md` and `docs/api-contracts/product-catalog-service.md` (format
  reference), ADR-011 (gateway is the only externally exposed service; internal calls bypass it).
- Outputs:
  - `docs/architecture/service-catalog.md` new entry for `campaign-service` (port 9011, CQRS +
    Mediator, transactional/non-cached infrastructure profile) - **authored by the implementing agent
    at build time**, not part of this planning-only sprint pass.
  - `docs/api-contracts/campaign-service.md` stub with port/mode/base-path header table and an empty
    Endpoints/Events section for 21.3/21.4 to fill in.
- Acceptance Criteria:
  - `docs/api-contracts/campaign-service.md` exists with the same header-table shape as
    `docs/api-contracts/order-service.md` (Port, Mode, Base path, Owning sprint, Build status,
    Requirements).
  - `docs/architecture/service-catalog.md` lists `campaign-service` at port 9011 with no port
    collision against any existing service in the catalog.
- Dependencies: 21.1.1
- Complexity: S

## Acceptance Criteria

Each subtask above carries its own objective, testable acceptance criteria. This feature is complete
when every subtask's acceptance criteria are met.

- Working `campaign-service` skeleton (starters, health probe, discovery registration) at port 9011.
- `campaign-db` schema (`campaigns`, `campaign_redemptions`) with platform outbox/inbox tables wired.
- Bare `Campaign`/`CampaignRedemption` JPA entities and repositories (no domain behavior yet).
- `docs/api-contracts/campaign-service.md` stub and a `docs/architecture/service-catalog.md` entry.

- All listed dependencies satisfied.
- Build and tests green per the global conventions in [../README.md](../README.md) Section 3.

  `tech-lead` ruling, not a decision made inside this task).
