# 21.5 Tests (Unit / Integration / Contract)

> Sprint: [Sprint 21 - Campaign / Catalog Validation](./README.md)
> Feature ID: 21.5
> Subtasks: 21.5.1, 21.5.2, 21.5.3

## Objective

Prove the Sprint 21 exit criteria (`README.md`) hold: eligible orders are discounted, ineligible
orders surface a clear reason, campaign-service outages never block order creation, and no service
shares database access with campaign-service (ADR-006). Covers unit tests for campaign-service domain
logic, Testcontainers integration/contract tests for campaign-service, and order-service integration
tests for the fail-open sync-call path, per the global testing convention (`docs/tasks/README.md`
Section 3: "Every service: Flyway migrations (ARC-07), unit tests, and Testcontainers integration
tests," ADR-013).

## Subtasks

#### 21.5.1 campaign-service domain unit tests

- ID: 21.5.1
- Title: Unit-test `Campaign`/`CampaignRedemption` lifecycle, eligibility, and redemption-cap logic
- Description: Mirror `TariffTest.java`/`AddonTest.java`'s framework-independent domain-test style
  (no Spring context) to cover: every legal and illegal `Campaign` lifecycle transition (21.2.1); the
  full eligibility-decision matrix from 21.2.3 (`EXPIRED`, `NOT_YET_ACTIVE`, `NOT_ACTIVE_STATUS`,
  `TARIFF_NOT_APPLICABLE`, `PER_CUSTOMER_CAP_EXCEEDED`, `TOTAL_CAP_EXCEEDED`, and the eligible case);
  the `totalRedemptionCap = null` unlimited case; and `CampaignRedemption.reserve/confirm/release`
  transition validity (each throwing `BusinessRuleException` on an illegal call, e.g. `confirm()` on
  an already-`RELEASED` row). Also unit-test the command handlers from 21.2.1 with mocked repositories
  (mirroring `ChangeTariffPriceCommandHandlerTest.java`'s handler-test style), and
  `CampaignServiceClient`'s fail-open behavior (21.3.2) in isolation with a mocked/failing
  `CircuitBreaker`.
- Business Purpose: Domain-rule correctness (the caps, windows, and state machine) is the feature's
  entire value proposition; unit-level coverage catches a broken eligibility rule before it reaches
  an order-creation integration test, where it is far more expensive to diagnose.
- Inputs: [21.2](21.2-campaign-domain-eligibility-and-limits.md), [21.3.2](21.3-campaign-validation-api-and-order-integration.md#2132-order-service-campaignserviceclient-and-circuit-breaker),
  `microservices/product-catalog-service/src/test/java/com/telco/catalog/domain/model/TariffTest.java`.
- Outputs:
  - `microservices/campaign-service/src/test/java/com/telco/campaign/domain/model/CampaignTest.java`
  - `microservices/campaign-service/src/test/java/com/telco/campaign/domain/model/CampaignRedemptionTest.java`
  - `microservices/campaign-service/src/test/java/com/telco/campaign/domain/service/CampaignEligibilityServiceTest.java`
  - `microservices/campaign-service/src/test/java/com/telco/campaign/application/handler/*CommandHandlerTest.java`
    (one per 21.2.1 command handler)
  - `microservices/order-service/src/test/java/com/telco/order/infrastructure/client/CampaignServiceClientTest.java`
- Acceptance Criteria:
  - Every eligibility reason code enumerated in 21.2.3 has at least one dedicated test case producing
    it.
  - The `totalRedemptionCap = null` (unlimited) case is asserted never to block, explicitly (not just
    incidentally passing).
  - `CampaignServiceClientTest` asserts no exception propagates out of `validate(...)` when the
    underlying call fails or the circuit is open (proves 21.3.2's fail-open encapsulation).
  - Line/branch coverage on `com.telco.campaign.domain.*` meets the project's standing coverage gate
    (per `docs/tasks/README.md` Section 3 / JaCoCo config already applied to every other service - no
    new threshold introduced by this task).
- Dependencies: 21.2 (all subtasks), 21.3.2
- Complexity: M

#### 21.5.2 campaign-service integration and contract tests

- ID: 21.5.2
- Title: Testcontainers integration tests, API contract test, event-schema/outbox/inbox tests
- Description: Add a `CampaignServiceIntegrationTest` (Testcontainers PostgreSQL, mirroring
  `ProductCatalogServiceIntegrationTest`) covering the full create -> activate -> validate ->
  (simulate order.created.v1) -> reserve -> (simulate payment.completed.v1 or the settled
  order-is-real event) -> confirm -> cap-exceeded-on-next-attempt flow end to end against a real
  database. Add a `CampaignSchemaMigrationTest` (mirroring `CatalogSchemaMigrationTest`) proving
  `V1__campaign.sql` applies cleanly and platform outbox/inbox tables are present. Add a
  `CampaignApiContractTest` (mirroring `TariffApiContractTest`) asserting `POST
  /api/v1/campaigns/validate` and the 21.2.1 admin endpoints return the documented
  `ApiResult<T>`-wrapped shapes from `docs/api-contracts/campaign-service.md`. Add a
  `CampaignEventSchemaCompatTest` (mirroring `TariffEventSchemaCompatTest`) for every new Avro schema
  registered in 21.4.1/21.4.2/21.4.3, verifying Schema Registry compatibility (ADR-019). Add
  Testcontainers-Kafka-backed consumer tests for `RedemptionCommitEventConsumer`,
  `OrderCancelledEventConsumer`, `OrderCreatedRedemptionReservationConsumer`, and the two tariff
  defensive consumers (21.4.2, 21.4.3), each asserting idempotent redelivery (send the same message
  twice, assert only one state transition).
- Business Purpose: Proves the feature works against real infrastructure (Postgres, Kafka, Schema
  Registry), not just mocked unit tests - the concurrency and eventing claims in 21.2.2/21.4.2/21.4.3
  are specifically the kind of behavior that only a real-database/real-broker test can actually
  validate.
- Inputs: [21.1](21.1-campaign-service-scaffold-and-schema.md)-[21.4](21.4-campaign-eventing-outbox-inbox.md),
  ADR-013, ADR-019, the four `product-catalog-service` reference test classes listed in Dependencies.
- Outputs:
  - `microservices/campaign-service/src/test/java/com/telco/campaign/CampaignServiceIntegrationTest.java`
  - `microservices/campaign-service/src/test/java/com/telco/campaign/migration/CampaignSchemaMigrationTest.java`
  - `microservices/campaign-service/src/test/java/com/telco/campaign/CampaignApiContractTest.java`
  - `microservices/campaign-service/src/test/java/com/telco/campaign/CampaignEventSchemaCompatTest.java`
  - `microservices/campaign-service/src/test/java/com/telco/campaign/application/consumer/*ConsumerTest.java`
- Acceptance Criteria:
  - `CampaignServiceIntegrationTest`'s end-to-end flow reaches `perCustomerRedemptionCap` and proves
    the next attempt is rejected, entirely through real Postgres-backed repositories.
  - `CampaignSchemaMigrationTest` fails loudly if `V1__campaign.sql` cannot apply cleanly.
  - Every consumer test proves idempotent redelivery (duplicate message -> single effect).
  - `CampaignApiContractTest` fails if any endpoint's response shape drifts from
    `docs/api-contracts/campaign-service.md`.
- Dependencies: 21.5.1, 21.4 (all subtasks)
- Complexity: L

#### 21.5.3 order-service integration tests: fail-open and discounted pricing

- ID: 21.5.3
- Title: Prove order-service's fail-open behavior and discounted pricing end-to-end
- Description: Add order-service integration tests (Testcontainers) covering: (1) order creation with
  campaign-service healthy and returning an eligible decision results in a discounted `unitPrice` on
  the persisted `OrderItem` and outbox `order.created.v1` payload; (2) order creation with
  campaign-service simulated unreachable (e.g. WireMock/stub returning connection failures, or the
  circuit breaker forced open) still succeeds and prices at the full undiscounted `monthlyFee` -
  directly proving the sprint's "campaign-service unavailability does not block order creation"
  exit criterion at the integration level, not just the client-unit-test level covered in 21.5.1; (3) a
  regression check that pre-existing `CreateOrderCommandHandler`/order-creation tests (no campaign
  involved) still pass unmodified in behavior, only extended with the new optional path.
- Business Purpose: This is the direct, end-to-end proof of the sprint's second exit criterion and the
  ADR-027 fail-open guarantee - the most safety-critical property in the whole feature, since a
  regression here would mean a campaign-service outage silently blocks all order creation platform-wide.
- Inputs: [21.3.3](21.3-campaign-validation-api-and-order-integration.md#2133-wire-discounted-pricing-into-createordercommandhandler),
  existing order-service integration test suite (`microservices/order-service/src/test/`) for the
  Testcontainers setup pattern to extend.
- Outputs:
  - New/extended test class(es) under `microservices/order-service/src/test/java/com/telco/order/`
    (e.g. `CampaignDiscountedOrderIntegrationTest.java`, `CampaignServiceFailOpenIntegrationTest.java`
    - exact naming at the implementer's discretion, following the existing suite's conventions)
- Acceptance Criteria:
  - A full order-creation call with campaign-service healthy and eligible produces a discounted
    `OrderItem.unitPrice` verifiable against the campaign's `discountType`/`discountValue`.
  - A full order-creation call with campaign-service down completes successfully with the
    undiscounted price - asserted as a hard integration-level requirement, not a mock.
  - No existing order-service test's behavior changes as a result of this feature (diff-reviewed, not
    just "still green").
- Dependencies: 21.3.3, 21.5.1
- Complexity: M
