# 21.3 Campaign Validation API and order-service Integration

> Sprint: [Sprint 21 - Campaign / Catalog Validation](./README.md)
> Feature ID: 21.3
> Subtasks: 21.3.1, 21.3.2, 21.3.3

## Objective

Expose `POST /api/v1/campaigns/validate` on campaign-service and wire order-service to call it
synchronously at order-creation time, snapshotting the discounted price into the `OrderItem` exactly
as order-service already snapshots the undiscounted catalog price - with a **fail-open** Resilience4j
circuit breaker so campaign-service unavailability never blocks order creation (ADR-027 Decision
Section 4).

## Dependencies

- [21.2](21.2-campaign-domain-eligibility-and-limits.md) (eligibility evaluation method, campaign
  data to validate against)
- ADR-027 Decision Section 4 (catalog validation model, fail-open circuit breaker)
- ADR-005 Service Communication Strategy (internal sync REST/OpenFeign + Resilience4j)
- `docs/api-contracts/order-service.md` ("Order capture validates the customer (ACTIVE/KYC) and
  snapshots catalog price synchronously")
- Existing pattern to mirror: `microservices/order-service/src/main/java/com/telco/order/infrastructure/client/ProductCatalogServiceClient.java`,
  `microservices/order-service/src/main/java/com/telco/order/config/{RestClientConfig,ResilienceConfig}.java`,
  `microservices/order-service/src/main/java/com/telco/order/application/handler/CreateOrderCommandHandler.java`

## Subtasks

#### 21.3.1 `POST /api/v1/campaigns/validate` endpoint

- ID: 21.3.1
- Title: Implement the campaign-validate endpoint on campaign-service
- Description: Implement `POST /api/v1/campaigns/validate` per
  [design-note.md Section 4](design-note.md#4-catalog-validation-model): request
  `{ customerId, tariffCode, campaignCode? }`, response either
  `{ eligible: true, campaignId, discountType, discountValue }` or
  `{ eligible: false, reason }` (reason codes from 21.2.3), wrapped in `ApiResult<T>` (ADR-015). Add
  the Mediator query (`ValidateCampaignQuery`/Handler) that calls the 21.2.3 eligibility-evaluation
  method; if `campaignCode` is omitted, resolve the best-matching ACTIVE campaign(s) whose
  `applicable_tariff_codes` include the given `tariffCode` (implementer's tie-break rule if more than
  one matches - e.g. highest discount value - documented in `docs/api-contracts/campaign-service.md`).
  This endpoint is called service-to-service (order-service -> campaign-service), not via the
  api-gateway (21.1.3); follow the same internal-call trust model
  `ProductCatalogServiceClient`/`CatalogSecurityConfig` already establish for `/internal/tariffs/**`
  (tokenless, network-perimeter trust, tech-lead ruling 2026-07-06) unless the implementing agent finds
  a reason campaign data is more sensitive than tariff data (it carries a `customerId`, unlike the
  tariff lookup) - **flag this specific choice (tokenless internal endpoint vs. JWT-bearing endpoint
  under `/api/v1`) to `security`/`tech-lead` before finalizing**, since ADR-027 literally names the
  path `/api/v1/campaigns/validate` (not `/internal/...`) without settling the auth model explicitly.
- Business Purpose: This is the endpoint the entire sprint exists to deliver - the synchronous
  eligibility+discount decision order-service needs before it can price a discounted `OrderItem`.
- Inputs: [design-note.md Section 4](design-note.md#4-catalog-validation-model), ADR-015, ADR-011
  Section 2 (internal vs. external surface).
- Outputs:
  - `microservices/campaign-service/src/main/java/com/telco/campaign/api/CampaignValidationController.java`
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/query/ValidateCampaignQuery.java`
    + handler
  - `microservices/campaign-service/src/main/java/com/telco/campaign/application/dto/{CampaignValidationRequest,CampaignValidationResponse}.java`
  - `docs/api-contracts/campaign-service.md` Endpoints section filled in (extends the 21.1.3 stub)
- Acceptance Criteria:
  - `POST /api/v1/campaigns/validate` against an ACTIVE campaign within its window and under caps
    returns `eligible: true` with a non-null `discountType`/`discountValue`.
  - The same call against an ineligible case (expired, cap-exceeded, tariff not applicable) returns
    `eligible: false` with a specific `reason`, never a 4xx/5xx for a well-formed ineligible request.
  - No `CampaignRedemption` row is created or mutated by this endpoint (validation must stay
    read-only; see [21.2.2](21.2-campaign-domain-eligibility-and-limits.md#2122-redemption-cap-enforcement-campaignredemption)'s
    reservation-trigger open item - reservation happens via 21.4's event consumption, not here).
- Dependencies: 21.2.3
- Complexity: M

#### 21.3.2 order-service `CampaignServiceClient` and circuit breaker

- ID: 21.3.2
- Title: Add a fail-open `CampaignServiceClient` to order-service
- Description: Add `CampaignServiceClient` to order-service's `infrastructure.client` package,
  mirroring `ProductCatalogServiceClient`'s `RestClient` + Resilience4j `CircuitBreaker` structure,
  with one deliberate behavioral difference: `ProductCatalogServiceClient` is **fail-closed**
  (a missing/unreachable tariff throws and blocks order creation, because a tariff is mandatory data);
  `CampaignServiceClient` must be **fail-open** (ADR-027 Decision Section 4: "if campaign-service is
  unreachable, the order proceeds at the full undiscounted price"). Implement this by catching
  `CallNotPermittedException` (circuit open) and `DependencyFailureException` (any other call failure)
  *inside* `CampaignServiceClient` itself and returning a sentinel "not eligible / no discount" result
  rather than propagating - do not let the fail-open behavior leak into `CreateOrderCommandHandler` as
  a try/catch around the client call, since a future maintainer could easily "fix" that into fail-closed
  by accident; encapsulate the safety property in the client. Add
  `campaignServiceCircuitBreaker()` to `ResilienceConfig` and `campaignRestClient(...)` to
  `RestClientConfig`, sourcing the base URL from `telco.clients.campaign-service.url`
  (config-server-driven, no hardcoded URL, matching the existing two clients).
- Business Purpose: Directly implements the sprint's second exit criterion: "campaign-service
  unavailability does not block order creation (fail-open circuit breaker verified)." Encapsulating
  fail-open inside the client (rather than at the call site) is a deliberate defensive-design choice
  so the property cannot be silently regressed.
- Inputs: `microservices/order-service/src/main/java/com/telco/order/infrastructure/client/ProductCatalogServiceClient.java`,
  `microservices/order-service/src/main/java/com/telco/order/config/{RestClientConfig,ResilienceConfig}.java`,
  ADR-027 Decision Section 4, ADR-005.
- Outputs:
  - `microservices/order-service/src/main/java/com/telco/order/infrastructure/client/CampaignServiceClient.java`
  - `microservices/order-service/src/main/java/com/telco/order/infrastructure/client/CampaignValidationResponse.java`
  - `microservices/order-service/src/main/java/com/telco/order/config/ResilienceConfig.java` (add
    `campaignServiceCircuitBreaker()` bean)
  - `microservices/order-service/src/main/java/com/telco/order/config/RestClientConfig.java` (add
    `campaignRestClient(...)` bean)
  - `microservices/order-service/src/main/resources/application.yml` (add
    `telco.clients.campaign-service.url`)
- Acceptance Criteria:
  - With campaign-service reachable and returning `eligible: true`, `CampaignServiceClient` returns
    the discount decision unchanged.
  - With campaign-service unreachable (connection refused) or its circuit breaker OPEN,
    `CampaignServiceClient.validate(...)` returns a "no discount" result and does **not** throw -
    verified by a unit test that asserts no exception propagates.
  - `campaignServiceCircuitBreaker` uses the same default config shape as the existing two breakers
    (`failureRateThreshold`, `slidingWindowSize`, `waitDurationInOpenState`,
    `permittedNumberOfCallsInHalfOpenState`) unless a documented reason justifies different tuning for
    a fail-open dependency (e.g. a shorter `waitDurationInOpenState` so discounts resume quickly once
    campaign-service recovers).
- Dependencies: 21.3.1
- Complexity: M

#### 21.3.3 Wire discounted pricing into `CreateOrderCommandHandler`

- ID: 21.3.3
- Title: Call campaign validation after the tariff price snapshot and price the `OrderItem` with the
  discount
- Description: In `CreateOrderCommandHandler.handle(...)`, after each tariff is resolved via
  `ProductCatalogServiceClient.getTariff(...)` (existing step), call
  `CampaignServiceClient.validate(customerId, tariff.code(), campaignCode)` (21.3.2) for that line
  item and compute the discounted `unitPrice` (`PERCENTAGE`: `monthlyFee * (1 - discountValue/100)`;
  `FIXED_AMOUNT`: `monthlyFee - discountValue`, floored at zero) when `eligible: true`; otherwise keep
  the undiscounted `monthlyFee`, identical to today's behavior. Persist which campaign (if any) priced
  each `OrderItem` - **this requires a schema change on order-service's side not previously scoped by
  ADR-027/design-note**: add a nullable `campaign_id`/`campaign_code` column to `order_items` (new
  order-db Flyway migration) and to the `OrderCreatedEvent.OrderItemPayload` record. This is the
  mechanism that makes the redemption-confirmation flow in
  [21.4](21.4-campaign-eventing-outbox-inbox.md) possible: without persisting which campaign applied
  to which order/item at order-creation time, campaign-service has no way to later correlate an
  `order.confirmed`/`payment.completed.v1` event (which carries only `orderId`/`customerId`, not
  `campaignId`) back to a specific campaign redemption. **Flag this addition to `architecture`/
  `tech-lead` for confirmation before implementation** - it is a small, order-service-local schema
  addition in service of ADR-027, not a reversal of any ADR-027 decision, but it was not explicitly
  called out in the ADR or design note and order-service's schema is outside this sprint's normal
  authority (`product-owner` does not modify architecture; this is a flag, not a mandate).
- Business Purpose: This is the mechanical center of the feature - "order-service snapshots the
  discounted price into the OrderItem, same as it already does for the undiscounted catalog price"
  (ADR-027 Decision Section 4) - and the addition of a campaign-correlation column is what keeps the
  redemption bookkeeping in 21.4 truthful rather than best-effort.
- Inputs: `microservices/order-service/src/main/java/com/telco/order/application/handler/CreateOrderCommandHandler.java`,
  `microservices/order-service/src/main/java/com/telco/order/domain/model/OrderItem.java`,
  `microservices/order-service/src/main/java/com/telco/order/application/event/OrderCreatedEvent.java`,
  ADR-027 Decision Section 4.
- Outputs:
  - `microservices/order-service/src/main/java/com/telco/order/application/handler/CreateOrderCommandHandler.java`
    (campaign-validate call + discounted pricing added)
  - `microservices/order-service/src/main/java/com/telco/order/domain/model/OrderItem.java`
    (`campaignId`/`campaignCode` nullable fields added)
  - `microservices/order-service/src/main/resources/db/migration/V{n}__order_items_campaign.sql`
    (new migration, next available version number in `order-service`'s migration sequence)
  - `microservices/order-service/src/main/java/com/telco/order/application/event/OrderCreatedEvent.java`
    (`OrderItemPayload` gains a nullable `campaignId` field)
  - `docs/api-contracts/order-service.md` updated to note the optional campaign discount in the order
    capture flow
- Acceptance Criteria:
  - An order placed for a tariff covered by an ACTIVE, eligible campaign is priced at the discounted
    `unitPrice`, and the `OrderItem`/event payload records which `campaignId` applied.
  - An order placed with no matching/eligible campaign is priced identically to today's undiscounted
    behavior (regression-safe: existing `CreateOrderCommandHandler` tests keep passing unmodified in
    intent, only extended).
  - Full campaign-service outage during order creation still results in a successfully created order
    at the undiscounted price (end-to-end proof of the fail-open property from 21.3.2, not just a
    client-level unit test).
- Dependencies: 21.3.2
- Complexity: L
