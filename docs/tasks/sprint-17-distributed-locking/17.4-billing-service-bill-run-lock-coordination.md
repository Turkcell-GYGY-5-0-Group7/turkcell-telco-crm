# 17.4 billing-service: Bill-Run Run-Level Lock Coordination for Scaled-Out Deployments

> Sprint: [Sprint 17 - Distributed Locking](./README.md)
> Feature ID: 17.4
> Subtasks: 17.4.1, 17.4.2

## Objective

Deliver "billing-service: Bill-Run Run-Level Lock Coordination for Scaled-Out Deployments" within
Sprint 17 - Distributed Locking: a run-level lock keyed on the billing period around
`RunBillCommandHandler`, so at most one billing-service pod owns a given period's bill-run
concurrently, closing the wasted-work/silently-swallowed-duplicate-key-exception gap described in
`docs/tasks/sprint-14-testing-and-hardening/14.3.2-bill-run-throughput-report.md`.

## Subtasks

#### 17.4.1 Run-level lock around RunBillCommandHandler
- ID: 17.4.1
- Title: Wrap `RunBillCommandHandler.handle()` in a watchdog-managed, period-keyed `DistributedLock`
- Description: Modify `RunBillCommandHandler.handle(RunBillCommand command)`
  (`microservices/billing-service/src/main/java/com/telco/billing/application/handler/RunBillCommandHandler.java`)
  to acquire a `DistributedLock` keyed on the billing period (e.g.
  `"billing-service:bill-run:" + command.periodStart() + ":" + command.periodEnd()`) before
  partitioning/dispatching batches, using
  `distributedLock.withLock(key, null, () -> { ...existing batch orchestration... })` - watchdog-managed
  (`leaseTime=null`), per ADR-024 Section 4's guidance that variable-duration work (a bill-run can
  legitimately run several minutes, per the 14.3.2 report's measured 6m20s at 100K subscribers) is
  exactly the watchdog case, not an explicit lease that could expire mid-run. On lock-acquisition
  failure (`DependencyFailureException` with `LockErrorCode.LOCK_ACQUISITION_FAILED` - no dedicated
  lock exception type exists, per the ADR-024 Section 5 ratification note), meaning another pod
  already owns this period's run, the handler must surface a distinguishable, logged "another pod
  already owns this run" outcome rather
  than letting the exception propagate as an undifferentiated failure - closing the diagnostic gap
  named in ADR-024's Context section (today a lost invoice attempt logs as a generic error, not a
  distinguishable "another pod already owns this run"). Add `starter-lock` as a new dependency to
  `billing-service/pom.xml`. Do not modify `BillRunBatchProcessor`'s existing
  `@Transactional(propagation = REQUIRES_NEW)` per-batch pattern or the `uidx_invoices_sub_period`
  unique index - this feature adds coordination on top, it does not replace either existing
  safeguard.
- Business Purpose: Prevents two billing-service pods from both partitioning and dispatching the same
  billing period's batches concurrently in a scaled-out deployment, closing the wasted-work/silently-
  swallowed-duplicate-key-exception gap the Sprint 14.3.2 report identified.
- Inputs: ADR-024 Context (billing-service motivation) and Section 4 (watchdog guidance),
  `RunBillCommandHandler.java` (existing orchestration - partitioning/`ExecutorService` dispatch to
  preserve unmodified inside the lock), `docs/tasks/sprint-14-testing-and-hardening/14.3.2-bill-run-throughput-report.md`
  (measured 6m20s@100K run duration - the lower bound the lock's watchdog must comfortably outlast),
  `uidx_invoices_sub_period` unique index (existing defense-in-depth, unaffected by this change).
- Outputs: `RunBillCommandHandler.java` (updated: constructor takes `DistributedLock`, `handle()`
  wraps the existing body in `withLock`), `microservices/billing-service/pom.xml` (`starter-lock`
  dependency), `microservices/configs/billing-service/application.yml` (`telco.platform.lock.*`
  wiring per 17.1), a new/updated `RunBillResult` field or log line distinguishing "another pod owns
  this run" from a generic failure.
- Acceptance Criteria:
  - Two concurrent `RunBillCommand` invocations for the *same* billing period (simulating two pods):
    exactly one runs the full batch-partition-and-dispatch flow; the other returns/logs a
    distinguishable "run already owned" outcome without partitioning or dispatching any batches
    (verified by asserting `BillRunBatchProcessor.processBatch` is never invoked on the losing side,
    not just by the absence of duplicate invoice rows).
  - Two concurrent `RunBillCommand` invocations for *different* billing periods both proceed without
    blocking each other (the lock key is period-scoped, not global).
  - The lock uses a watchdog-managed lease (`leaseTime=null`), not an explicit `Duration`, consistent
    with ADR-024 Section 4's rationale for variable-duration work.
  - `uidx_invoices_sub_period` remains untouched as the existing defense-in-depth guarantee; this
    task adds coordination on top, not a replacement.
- Dependencies: 17.1.2, 17.1.3
- Complexity: M

#### 17.4.2 Multi-instance concurrency verification for bill-run
- ID: 17.4.2
- Title: Prove exactly one pod owns a bill-run for a given period under concurrent invocation
- Description: Add an integration test analogous in shape to `BillRunThroughputPerformanceIT`'s
  harness (`BillRunSeedHarness`) but scoped to concurrency, not throughput: seed a modest subscriber
  set, then fire two-or-more concurrent `mediator.send(new RunBillCommand(...))` invocations for the
  identical period from separate threads (simulating separate pods sharing the same Redis and
  Postgres), and assert exactly one invocation performs the full batch run while the other(s) observe
  the lock as held and complete cleanly.
- Business Purpose: Directly satisfies the Sprint 17 README Exit Criteria ("billing-service's...
  bill-run lock are each verified with a multi-instance (or multi-thread-simulating-multi-instance)
  test showing exactly one winner per lock key, and the losing side degrades safely").
- Inputs: `microservices/billing-service/src/test/java/com/telco/billing/perf/BillRunSeedHarness.java`
  (reusable seeding pattern to adapt at smaller scale), 17.2.1's Testcontainers Redis harness
  (test-jar), 17.4.1's lock-wrapped handler.
- Outputs:
  `microservices/billing-service/src/test/java/com/telco/billing/application/handler/RunBillCommandHandlerConcurrencyIT.java`
  (or under `.../concurrency/`).
- Acceptance Criteria:
  - With 2+ concurrent `RunBillCommand` invocations for the same period against a small seeded
    subscriber set, exactly one invocation generates invoices; the loser generates zero and does not
    throw an unhandled exception.
  - A direct SQL duplicate-invoice check (mirroring `BillRunThroughputPerformanceIT`'s existing query)
    confirms zero duplicates, as additional defense-in-depth evidence alongside the lock-level
    assertion.
- Dependencies: 17.4.1, 17.2.1
- Complexity: M
