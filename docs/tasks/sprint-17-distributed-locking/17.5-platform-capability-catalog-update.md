# 17.5 Platform Capability Catalog and PLATFORM-SPEC.md Update: Distributed Locking Planned to Available

> Sprint: [Sprint 17 - Distributed Locking](./README.md)
> Feature ID: 17.5
> Subtasks: 17.5.1, 17.5.2, 17.5.3

## Objective

Deliver "Platform Capability Catalog and PLATFORM-SPEC.md Update: Distributed Locking Planned to
Available" within Sprint 17 - Distributed Locking: once `starter-lock` has shipped and its two
consumers (17.3, 17.4) are verified, update the documentation surfaces that describe the platform's
reusable capabilities so distributed locking is discoverable as available, not missed or left marked
planned.

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- 17.1 (`starter-lock` must actually exist before it can be documented as available).
- 17.3, 17.4 (first real consumers; their existence is cited as evidence in the catalog update).
- `docs/architecture/platform-capabilities.md` (document being updated; Sections 1-3 structure).
- `platform/PLATFORM-SPEC.md` (document being updated; Section 1 module table and per-module
  sections).
- `docs/architecture/platform-gap-closing-plan.md` (document being updated; existing Tier-table
  format and follow-up-entry convention).
- 17.5.1

## Subtasks

#### 17.5.1 platform-capabilities.md update
- ID: 17.5.1
- Title: Move distributed locking from "planned" to "available" in the capability catalog
- Description: Update `docs/architecture/platform-capabilities.md`: add a new "Locking (via
  `starter-lock`)" row group to Section 1 ("Available now") documenting `DistributedLock`,
  `LockHandle`, `LockAcquisitionException` (import `com.telco.platform.lock.*`, consumed via
  `starter-lock`, matching the existing table format used for outbox/inbox/mediator rows), and
  ensure no stale "planned" reference to distributed locking remains in Section 3 ("Planned - not yet
  available") once this task completes - Section 3 does not list it as of this drafting (a gap ADR-024's
  Context section itself calls out), so this task's job is to make Section 1 correct going forward,
  not to first add-then-remove a Section 3 row that was never actually written.
- Business Purpose: Keeps the capability catalog's "reuse before build" promise accurate - once
  `starter-lock` exists, any service or agent consulting the catalog before writing ad hoc
  coordination code must find it listed as available.
- Inputs: `docs/architecture/platform-capabilities.md` Sections 1-3 (structure to match), ADR-024
  (final, ratified content - library, port shape, property names), 17.1's shipped module names and
  packages.
- Outputs: `docs/architecture/platform-capabilities.md` updated (new Section 1 subsection).
- Acceptance Criteria:
  - Section 1 lists `DistributedLock`/`LockHandle`/`LockAcquisitionException` with the correct import
    path and "via `starter-lock`" attribution, matching the existing table format for other
    capabilities.
  - No reference to distributed locking as "planned" exists anywhere in the document after this
    update.
- Dependencies: 17.1.2, 17.1.3
- Complexity: S

#### 17.5.2 PLATFORM-SPEC.md update
- ID: 17.5.2
- Title: Document `platform-core/lock`/`starter-lock` in PLATFORM-SPEC.md
- Description: Add a new module section to `platform/PLATFORM-SPEC.md` for `platform-core/lock`/
  `starter-lock` (a module/artifactId/base-package row in Section 1's table, plus a new numbered
  section documenting the `DistributedLock` port shape, `LockAutoConfiguration`'s conditional-bean
  rules, and the `telco.platform.lock.*` property table), matching the existing per-module
  documentation depth already given to outbox/inbox in the same spec.
- Business Purpose: PLATFORM-SPEC is the authoritative builder's contract (root `CLAUDE.md`); leaving
  a shipped module undocumented there would let a future platform-engineer re-derive or diverge from
  ADR-024's decisions instead of reading them from the spec.
- Inputs: `platform/PLATFORM-SPEC.md` (existing structure/depth for outbox/inbox sections to mirror),
  ADR-024 full text (source of truth for the content).
- Outputs: `platform/PLATFORM-SPEC.md` updated (Section 1 module table row, new lock section).
- Acceptance Criteria:
  - `platform/PLATFORM-SPEC.md` Section 1's module table includes `platform-lock`/`starter-lock` with
    correct base packages.
  - A new section documents the `DistributedLock` port signature, the two lease modes, and the
    fail-closed behavior, consistent in substance with ADR-024 (no re-derivation, no contradiction).
- Dependencies: 17.5.1
- Complexity: S

#### 17.5.3 platform-gap-closing-plan.md follow-up entry
- ID: 17.5.3
- Title: Record distributed locking as a shipped follow-up in the gap-closing plan
- Description: Add distributed locking as a shipped follow-up entry to
  `docs/architecture/platform-gap-closing-plan.md`, per the Sprint 17 README's stated purpose ("the
  existing tiered gap-closing sequencing this capability slots into as a follow-up entry") and
  ADR-024's Context note that its ratification should feed this document. Add a row (or a small
  "Follow-up" addendum, matching the existing Tier-table format) noting the capability, its module
  (`starter-lock`), its first consumers (subscription-service 17.3, billing-service 17.4), and that it
  shipped outside the original Tier 1-3 sequencing as a post-MVP addition.
- Business Purpose: Keeps the gap-closing plan an accurate historical record of what has been closed
  and how, matching the bookkeeping already done for the Tier 1-3 items.
- Inputs: `docs/architecture/platform-gap-closing-plan.md` (existing Tier table format), ADR-024,
  Sprint 17 README.
- Outputs: `docs/architecture/platform-gap-closing-plan.md` updated.
- Acceptance Criteria:
  - The plan document reflects distributed locking as shipped, with its module name and first
    consumers, in a format consistent with the existing table.
- Dependencies: 17.5.1
- Complexity: S
