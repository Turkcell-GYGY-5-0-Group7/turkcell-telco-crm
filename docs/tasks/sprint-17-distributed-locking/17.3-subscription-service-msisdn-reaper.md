# 17.3 subscription-service: MSISDN Reservation-Expiry Reaper with Cross-Instance Lock Coordination

> Sprint: [Sprint 17 - Distributed Locking](./README.md)
> Feature ID: 17.3
> Subtasks: 17.3.1, 17.3.2

## Objective

Deliver "subscription-service: MSISDN Reservation-Expiry Reaper with Cross-Instance Lock
Coordination" within Sprint 17 - Distributed Locking: a `@Scheduled` reaper that expires stale
`RESERVED` MSISDN holds back to `FREE`, coordinated across replicas via `starter-lock` so exactly one
pod performs the sweep per tick once subscription-service scales out (Sprint 15 HPA), per ADR-024's
Context section and FR-13.

## Subtasks

#### 17.3.1 Reaper scheduled job, lock-coordinated across replicas
- ID: 17.3.1
- Title: Add `MsisdnReservationExpiryReaper`, guarded by an explicit-lease `DistributedLock`
- Description: Add a `@Scheduled` component (e.g.
  `infrastructure/scheduler/MsisdnReservationExpiryReaper.java`) to `subscription-service` that finds
  `MsisdnPool` rows in `RESERVED` status whose `reservedUntil` has elapsed and releases them back to
  `FREE`, calling the existing `MsisdnPool.release()` domain method (do not bypass it with a direct
  SQL update, and do not modify its transition rules), running on a configurable fixed-delay schedule.
  Because `@Scheduled` runs identically on every replica once the service scales (Sprint 15 HPA),
  guard the entire sweep with `DistributedLock` (add `starter-lock` as a new dependency to
  `subscription-service/pom.xml`): wrap the tick in
  `distributedLock.withLock("subscription-service:msisdn-reaper", Duration.ofSeconds(n), this::sweep)`
  using an **explicit** `Duration` lease (not watchdog-managed), per ADR-024 Section 4's guidance that
  a bounded, per-tick sweep is exactly the explicit-lease use case. A replica that loses the race
  (lock not acquired within `wait-time`) simply skips that tick and lets the next scheduled tick
  retry - no error, no duplicate sweep, no unhandled `DependencyFailureException` (the
  `LockErrorCode.LOCK_ACQUISITION_FAILED` case is caught and logged, per the ADR-024 Section 5
  ratification note - no dedicated `LockAcquisitionException` type exists) propagated out of the
  scheduled method.
- Business Purpose: Closes the gap ADR-024's Context section names directly: `reservedUntil` already
  exists on `MsisdnPool` but no reaper exists today, and once built it must not race across replicas
  (FR-13, Sprint 15 HPA).
- Inputs: ADR-024 (reaper motivation, explicit-lease guidance, Section 4),
  `microservices/subscription-service/src/main/java/com/telco/subscription/domain/MsisdnPool.java`
  (`release()` method, existing state machine),
  `microservices/subscription-service/src/main/java/com/telco/subscription/infrastructure/MsisdnPoolRepository.java`
  (existing `FOR UPDATE SKIP LOCKED` allocation query this reaper complements, does not replace),
  `microservices/subscription-service/CLAUDE.md` ("Audit logging is mandatory for state-changing
  operations").
- Outputs:
  `microservices/subscription-service/src/main/java/com/telco/subscription/infrastructure/scheduler/MsisdnReservationExpiryReaper.java`,
  a new query method on `MsisdnPoolRepository` (e.g.
  `findByStatusAndReservedUntilBefore(MsisdnStatus, Instant)`), `microservices/subscription-service/pom.xml`
  (`starter-lock` dependency added), `microservices/configs/subscription-service/application.yml`
  (reaper interval, `telco.platform.lock.*` wiring per 17.1).
- Acceptance Criteria:
  - A `RESERVED` MSISDN whose `reservedUntil` has elapsed is returned to `FREE` on the next reaper
    tick, driven through `MsisdnPool.release()` (not a direct SQL update bypassing the domain method).
  - A `RESERVED` MSISDN whose `reservedUntil` has not elapsed is untouched.
  - Each release writes an `audit_log` row per subscription-service's existing mandatory
    audit-logging rule (ADR-021, NFR-12).
  - The reaper uses an explicit-`Duration` lease (not watchdog-managed), per ADR-024 Section 4's
    stated rationale for bounded sweeps.
  - A losing replica's tick completes without an unhandled `DependencyFailureException` propagating
    out of the `@Scheduled` method (it logs and skips, waiting for the next tick).
- Dependencies: 17.1.2, 17.1.3
- Complexity: M

#### 17.3.2 Multi-instance concurrency verification
- ID: 17.3.2
- Title: Prove exactly one replica sweeps expired reservations per tick
- Description: Add an integration test (e.g.
  `MsisdnReservationExpiryReaperConcurrencyIT.java`, Testcontainers Postgres + the `starter-lock`
  test-jar's Redis harness from 17.2.1) that simulates two-or-more subscription-service "replicas"
  (multiple `MsisdnReservationExpiryReaper` instances, or multiple threads each holding an
  independent `DistributedLock` client bound to the same Redis, all sharing the same underlying
  `msisdn_pool` table) racing to sweep the same set of expired `RESERVED` rows concurrently, and
  asserts exactly one instance performs the sweep per tick - the losing instance(s) observe the lock
  as held and skip, verified via a call-count assertion, not by absence of errors alone.
- Business Purpose: Directly satisfies the Sprint 17 README Exit Criteria
  ("subscription-service's MSISDN reaper... [is] verified with a multi-instance (or
  multi-thread-simulating-multi-instance) test showing exactly one winner per lock key, and the
  losing side degrades safely").
- Inputs: 17.3.1's reaper, 17.2.1's Testcontainers Redis harness (test-jar), Sprint 17 README Exit
  Criteria.
- Outputs:
  `microservices/subscription-service/src/test/java/com/telco/subscription/infrastructure/scheduler/MsisdnReservationExpiryReaperConcurrencyIT.java`.
- Acceptance Criteria:
  - With N (>= 2) concurrently-racing reaper instances/threads triggered against the same set of
    expired rows, exactly one performs the release work per tick; the rest complete without error and
    without performing a duplicate release.
  - The test runs against real Testcontainers Postgres + Redis (no mocked `DistributedLock`).
- Dependencies: 17.3.1, 17.2.1
- Complexity: M
