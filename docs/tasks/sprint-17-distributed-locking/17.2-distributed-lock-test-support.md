#### 17.2.2 Contention and failure-mode test suite
- ID: 17.2.2
- Title: Prove mutual exclusion, watchdog liveness, explicit-lease expiry, and fail-closed behavior
  against a real Redis
- Description: Write the concurrency/failure-mode integration suite (e.g.
  `RedissonDistributedLockIT.java`, or split into
  `RedissonDistributedLockContentionIT.java` + `RedissonDistributedLockFailureModeIT.java`) proving,
  against the real Redis container from 17.2.1, every guarantee ADR-024 makes: (1) **mutual
  exclusion** - two concurrent `acquire`/`withLock` calls on the same key from separate threads
  serialize; the second blocks until the first releases, never both entering the critical section
  simultaneously (asserted via a shared counter, not a timing guess); (2) **watchdog liveness** - a
  watchdog-managed lease (`leaseTime=null`) survives a holder that runs longer than
  `watchdog-timeout`, because Redisson's internal renewal keeps extending it (a competing `tryLock`
  from another thread fails while the first holder is still working, past the nominal
  `watchdog-timeout` window); (3) **explicit-lease hard expiry** - an explicit-`Duration` lease
  hard-expires at exactly that duration even while the holder is still "alive" (a second acquirer
  succeeds once the explicit lease elapses, despite the first holder never having released); (4)
  **fail-closed** - a Redis outage (stop the Testcontainers Redis container, or point at an
  unreachable address) causes `acquire`/`withLock` to throw `DependencyFailureException` (constructed
  with `LockErrorCode.LOCK_ACQUISITION_FAILED`, per the ADR-024 Section 5 ratification - no dedicated
  lock exception type exists), not silently proceed - the caller's guarded action is never invoked
  (verified by an invocation-count assertion, not just absence of an exception).
- Business Purpose: Directly proves the four guarantees the Sprint 17 README Exit Criteria and
  ADR-024 both name explicitly - mutual exclusion, watchdog liveness, explicit-lease hard-expiry, and
  fail-closed - are real behavior, not just documented intent, closing the ADR's stated Consequence
  risk ("two more real consumers required before this capability's API shape can be considered
  validated in practice").
- Inputs: ADR-024 Sections 4 and 5 (the four behaviors under test), 17.2.1's `RedisContainerSupport`,
  the Sprint 17 README Exit Criteria ("`starter-lock` ships with unit and Testcontainers-backed
  integration tests proving the fail-closed behavior on Redis unavailability and correct mutual
  exclusion under real concurrent acquisition - not just mocked Redisson behavior").
- Outputs:
  `platform/platform-starters/starter-lock/src/test/java/com/telco/platform/starter/lock/RedissonDistributedLockIT.java`
  (or the split `ContentionIT`/`FailureModeIT` variant), plus supporting unit tests for
  `RedissonLockHandle`/`LockProperties`/`LockAutoConfiguration`'s conditional-bean behavior.
- Acceptance Criteria:
  - A two-thread contention test asserts serialized access to a shared counter under a lock (no lost
    updates, no both-in-critical-section race).
  - A watchdog-lease test proves a holder running past `watchdog-timeout` keeps the lock (a competing
    `tryLock` attempt from another thread fails while the first holder is still working).
  - An explicit-lease test proves the lock releases automatically at the declared `Duration` even
    without an explicit `close()`/unlock call from a still-running holder.
  - A Redis-outage test proves `DependencyFailureException` (with `LockErrorCode.LOCK_ACQUISITION_FAILED`)
    is thrown and the guarded action is never executed.
  - `mvn -f platform/pom.xml -pl platform-starters/starter-lock -am verify` runs and passes these
    tests as part of the module's own default build (a correctness suite, unlike the manually-invoked
    `*IT` bill-run throughput perf test - this must not be excluded from the default Surefire/Failsafe
    run for `starter-lock`).
- Dependencies: 17.1.1, 17.1.2, 17.1.3, 17.2.1
- Complexity: L
