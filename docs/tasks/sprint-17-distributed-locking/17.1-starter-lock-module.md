# 17.1 starter-lock Module: Redisson Wiring, DistributedLock Port, Config Properties, Fail-Closed Semantics

> Sprint: [Sprint 17 - Distributed Locking](./README.md)
> Feature ID: 17.1
> Subtasks: 17.1.1, 17.1.2, 17.1.3

## Objective

Deliver "starter-lock Module: Redisson Wiring, DistributedLock Port, Config Properties, Fail-Closed
Semantics" within Sprint 17 - Distributed Locking: the new `platform-core/lock` port and
`platform-starters/starter-lock` Redisson-backed implementation, built exactly to ADR-024's decisions
(library choice, module home, API shape, TTL/lease semantics, fail-closed failure mode, ADR-018 fit).

## Dependencies

Aggregated from the subtasks below (prerequisite task IDs and enablers):

- ADR-024 Distributed Lock Strategy must be ratified (Accepted) by tech-lead before implementation
  begins (Sprint 17 README Exit Criteria). **Ratified 2026-07-12**: Section 5's original
  `LockAcquisitionException extends PlatformException` design was infeasible (`PlatformException` is
  `sealed` with a same-package-only `permits` list, no `module-info.java` exists under `platform/`).
  Corrected: fail-closed acquisition failures now throw the existing `DependencyFailureException`
  (already 503-mapped in `starter-api`) constructed with a new `LockErrorCode` defined in
  `platform-core/lock`. This task's subtasks below reflect the corrected design.
- ADR-007 Platform Library Strategy (new-module Extension Rule this feature follows).
- ADR-018 Platform Starter Dependency Model (starter-only consumption rule `starter-lock` must fit).
- `platform/platform-core/outbox` and `platform/platform-starters/starter-outbox` (existing sibling
  modules whose split between Spring-free port and Spring-wired starter this feature mirrors).
- `platform/platform-starters/starter-security` (existing `AutoConfiguration.imports` wiring pattern
  to mirror for `LockAutoConfiguration`).
- 17.1.1

## Subtasks

#### 17.1.1 platform-core/lock: Spring-free DistributedLock port
- ID: 17.1.1
- Title: Add the `platform-core/lock` module (`DistributedLock` port, `LockHandle`, `LockErrorCode`)
- Description: Create a new Maven module `platform/platform-core/lock` (artifactId `platform-lock`,
  base package `com.telco.platform.lock`, registered as a `<module>` in `platform-core/pom.xml`
  alongside `common`/`cqrs`/`mediator`/`outbox`/`inbox`), matching platform-core's Spring-free purity
  constraint (ADR-024 Section 2, `platform/PLATFORM-SPEC.md` Section 1: "Spring lives ONLY in
  `platform-autoconfigure` and `platform-starters/*`"). Define the `DistributedLock` port exactly per
  ADR-024 Section 3 (`LockHandle acquire(String key, Duration leaseTime)` where `LockHandle extends
  AutoCloseable` for try-with-resources use, `<T> T withLock(String key, Duration leaseTime, Callable<T>
  action)`, `void withLock(String key, Duration leaseTime, Runnable action)`), and `LockHandle`
  (extends `AutoCloseable`, exposes at minimum the held key and a `close()`/`release()`). Also define
  `LockErrorCode implements ErrorCode` (mirroring `platform-common`'s existing `CommonErrorCode`
  pattern), with at least `LOCK_ACQUISITION_FAILED`. There is no dedicated lock exception type:
  `PlatformException` is `sealed` with a `permits` list scoped to `com.telco.platform.common.exception`
  and this codebase has no `module-info.java`, so a same-hierarchy subtype cannot live in
  `com.telco.platform.lock` (ADR-024 Section 5 ratification note); fail-closed failures are surfaced by
  `starter-lock` throwing the existing `DependencyFailureException` constructed with `LockErrorCode`
  (see 17.1.3). No Redisson or Spring types appear anywhere in this module's public API or dependency
  graph (ADR-024 Section 2, bullet 1).
- Business Purpose: Establishes the infrastructure-agnostic contract every consumer (starter-lock's
  Redisson implementation, and a future in-memory fake for unit tests) depends on, keeping the
  third-party Redisson client out of platform-core's own dependency graph per ADR-007/PLATFORM-SPEC's
  purity rule.
- Inputs: ADR-024 Sections 2-3 and 5 (ratification note), `platform/PLATFORM-SPEC.md` Section 1
  (module/package table) and Section 2.2 (`PlatformException`/`ErrorCode` conventions),
  `platform/platform-core/outbox/pom.xml` (sibling module shape to mirror: parent/artifactId pattern,
  dependency on `platform-common` + slf4j-api + junit test-scope only),
  `platform/platform-core/common/.../exception/CommonErrorCode.java` (pattern to mirror for
  `LockErrorCode`), `DependencyFailureException.java` (the exception `starter-lock` will construct).
- Outputs: `platform/platform-core/lock/pom.xml` (artifactId `platform-lock`, parent `platform-core`),
  `platform/platform-core/lock/src/main/java/com/telco/platform/lock/DistributedLock.java`,
  `LockHandle.java`, `LockErrorCode.java`, updated `platform/platform-core/pom.xml`
  `<modules>` list (add `<module>lock</module>`).
- Acceptance Criteria:
  - `platform-lock` builds standalone (`mvn -f platform/pom.xml -pl platform-core/lock -am install`)
    with zero Spring/Redisson dependencies, verified via `mvn dependency:tree`.
  - `DistributedLock`, `LockHandle` match the exact method signatures specified in ADR-024 Section 3.
  - `LockErrorCode` implements `ErrorCode` and defines `LOCK_ACQUISITION_FAILED`; no
    `LockAcquisitionException` class is created (ADR-024 Section 5 ratification note).
- Dependencies: none (first task of Sprint 17)
- Complexity: S

#### 17.1.2 starter-lock: RedissonDistributedLock, LockAutoConfiguration, LockProperties
- ID: 17.1.2
- Title: Add `starter-lock`: the Redisson-backed implementation and Spring auto-wiring
- Description: Create `platform/platform-starters/starter-lock` (artifactId `starter-lock`, base
  package `com.telco.platform.starter.lock`, registered in `platform-starters/pom.xml` `<modules>`),
  depending on `platform-lock`, `platform-common`, `platform-autoconfigure`, and the plain
  `org.redisson:redisson` client artifact (not Redisson's own opinionated
  `redisson-spring-boot-starter`, per ADR-024 Section 2 - the platform owns the AutoConfiguration and
  property surface). Implement `RedissonDistributedLock implements DistributedLock` (built on
  `RedissonClient.getLock(key)` / `RLock`, watchdog-managed when `leaseTime` is `null`, explicit
  hard-expiry lease when a `Duration` is passed, per ADR-024 Section 4; on connection failure, timeout,
  or `wait-time` budget expiry it throws `DependencyFailureException` constructed with
  `LockErrorCode.LOCK_ACQUISITION_FAILED` and the contended key in `details`, per ADR-024 Section 5),
  `LockAutoConfiguration`
  (builds the `RedissonClient` bean from `telco.platform.lock.redis.address`, falling back to
  `spring.data.redis.host`/`port` when unset, per ADR-024's property table; conditional on
  `telco.platform.lock.enabled=true` (default) and Redisson on the classpath;
  `@ConditionalOnMissingBean` override points per ADR-018 Section 3), and `LockProperties`
  (`telco.platform.lock.*`: `enabled` default `true`, `redis.address`, `wait-time` default `5s`,
  `watchdog-timeout` default `30s` - the ADR-024 Section 2 table, minus `lease-time`, removed by a
  2026-07-12 code-review correction as dead configuration since `DistributedLock`'s port always takes
  an explicit per-call `leaseTime` argument). Register the auto-configuration in
  `src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`,
  matching `starter-security`'s existing pattern.
- Business Purpose: Delivers the actual Redis-backed lock implementation and auto-wiring so a
  consuming service gets a working `DistributedLock` bean by adding one starter dependency, with zero
  Redisson/Spring exposure through platform-core (ADR-024 Sections 2, 6).
- Inputs: ADR-024 Sections 2, 3, 4, 6 in full;
  `platform/platform-starters/starter-security/src/main/java/com/telco/platform/starter/security/SecurityAutoConfiguration.java`
  and its `AutoConfiguration.imports` file as the wiring template; ADR-018 Section 3 (override points).
- Outputs: `platform/platform-starters/starter-lock/pom.xml`,
  `.../starter/lock/RedissonDistributedLock.java`, `.../starter/lock/LockAutoConfiguration.java`,
  `.../starter/lock/LockProperties.java`, `.../starter/lock/RedissonLockHandle.java` (or equivalent
  `LockHandle` implementation wrapping `RLock`),
  `src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`,
  updated `platform/platform-starters/pom.xml` `<modules>` list.
- Acceptance Criteria:
  - A service that adds only `starter-lock` (no `platform-lock` or `org.redisson` direct dependency)
    gets an injectable `DistributedLock` bean with no other config (ADR-018 Section 2's Dependency
    Rule; ADR-024 Section 6).
  - `acquire(key, null)` produces a watchdog-managed lock verified by holding past
    `watchdog-timeout` while the holder keeps renewing; `acquire(key, Duration.ofSeconds(n))` produces
    a lock that hard-expires at exactly `n` seconds regardless of holder liveness (ADR-024 Section 4).
  - `telco.platform.lock.enabled=false` disables the auto-configuration entirely (no `RedissonClient`
    bean created).
  - `@ConditionalOnMissingBean` allows a consuming service to override the `RedissonClient` or
    `DistributedLock` bean.
- Dependencies: 17.1.1
- Complexity: M

#### 17.1.3 Fail-closed behavior verification and platform-bom wiring
- ID: 17.1.3
- Title: Verify fail-closed Redis-unavailability handling and pin Redisson in platform-bom
- Description: No `GlobalExceptionHandler` (`starter-api`) change is required: per the ADR-024 Section 5
  ratification, `RedissonDistributedLock` (17.1.2) already throws the existing `DependencyFailureException`
  (constructed with `LockErrorCode.LOCK_ACQUISITION_FAILED` and the contended key in `details`), and
  `GlobalExceptionHandler` already maps `DependencyFailureException` -> HTTP `503`
  (`handleDependencyFailure`, existing code, unchanged). This task's job is to (a) verify that mapping
  actually covers the lock path end to end, and (b) pin the Redisson version in `platform-bom` (new
  `redisson.version` property, `org.redisson:redisson` dependency-management entry) and add
  `platform-lock` and `starter-lock` coordinate entries to `platform-bom`'s `<dependencyManagement>`,
  following the exact pattern already used for `platform-outbox`/`starter-outbox`.
- Business Purpose: Confirms the fail-closed behavior ADR-024 Section 5 mandates is externally
  observable as a `503`, not an uncaught `500`, by reusing the platform's existing dependency-failure
  path rather than adding a parallel one, and ensures every consuming service inherits a single pinned
  Redisson version rather than resolving one transitively (ADR-003, ADR-020).
- Inputs: ADR-024 Section 5 (ratified fail-closed semantics via `DependencyFailureException` +
  `LockErrorCode`), `starter-api`'s existing `GlobalExceptionHandler.handleDependencyFailure`
  (already-present mapping, no edit needed), `platform/platform-bom/pom.xml` (existing pattern:
  `<properties>` version + `<dependencyManagement>` entries for `platform-outbox`/`starter-outbox`).
- Outputs: `platform/platform-bom/pom.xml` updates only (`redisson.version` property,
  `org.redisson:redisson` dependency, `platform-lock` and `starter-lock` dependencyManagement entries).
  No `starter-api` source changes.
- Acceptance Criteria:
  - With Redis unreachable, a call through `DistributedLock.acquire`/`withLock` from a
    controller-reachable path returns HTTP `503` with a stable `ApiError.code`
    (`LockErrorCode.LOCK_ACQUISITION_FAILED`), not a generic `500`, via the existing
    `handleDependencyFailure` mapping - confirmed by a test, no handler code changed.
  - `platform-bom` pins exactly one Redisson version; no service or starter POM hardcodes a Redisson
    version (ADR-020).
  - `mvn -f platform/pom.xml -am install` (full platform reactor, including the new modules) succeeds.
- Dependencies: 17.1.1, 17.1.2
- Complexity: S
