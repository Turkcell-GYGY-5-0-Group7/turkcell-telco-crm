# 17.1 starter-lock Module: Redisson Wiring, DistributedLock Port, Config Properties, Fail-Closed Semantics

> Sprint: [Sprint 17 - Distributed Locking](./README.md)
> Feature ID: 17.1
> Subtasks: 17.1.1, 17.1.2, 17.1.3

## Objective

Deliver "starter-lock Module: Redisson Wiring, DistributedLock Port, Config Properties, Fail-Closed
Semantics" within Sprint 17 - Distributed Locking: the new `platform-core/lock` port and
`platform-starters/starter-lock` Redisson-backed implementation, built exactly to ADR-024's decisions
(library choice, module home, API shape, TTL/lease semantics, fail-closed failure mode, ADR-018 fit).

Aggregated from the subtasks below (prerequisite task IDs and enablers):

## Subtasks

#### 17.1.1 platform-core/lock: Spring-free DistributedLock port
- ID: 17.1.1
- Title: Add the `platform-core/lock` module (`DistributedLock` port, `LockHandle`,
  `LockAcquisitionException`)
- Description: Create a new Maven module `platform/platform-core/lock` (artifactId `platform-lock`,
  base package `com.telco.platform.lock`, registered as a `<module>` in `platform-core/pom.xml`
  alongside `common`/`cqrs`/`mediator`/`outbox`/`inbox`), matching platform-core's Spring-free purity
  constraint (ADR-024 Section 2, `platform/PLATFORM-SPEC.md` Section 1: "Spring lives ONLY in
  `platform-autoconfigure` and `platform-starters/*`"). Define the `DistributedLock` port exactly per
  ADR-024 Section 3 (`LockHandle acquire(String key, Duration leaseTime)` where `LockHandle extends
  AutoCloseable` for try-with-resources use, `<T> T withLock(String key, Duration leaseTime, Callable<T>
  action)`, `void withLock(String key, Duration leaseTime, Runnable action)`), `LockHandle` (extends
  `AutoCloseable`, exposes at minimum the held key and a `close()`/`release()`), and
  `LockAcquisitionException` (extends the existing `PlatformException` base from `platform-common`,
  carries the contended lock key and cause). No Redisson or Spring types appear anywhere in this
  module's public API or dependency graph (ADR-024 Section 2, bullet 1).
- Business Purpose: Establishes the infrastructure-agnostic contract every consumer (starter-lock's
  Redisson implementation, and a future in-memory fake for unit tests) depends on, keeping the
  third-party Redisson client out of platform-core's own dependency graph per ADR-007/PLATFORM-SPEC's
  purity rule.
- Inputs: ADR-024 Sections 2-3, `platform/PLATFORM-SPEC.md` Section 1 (module/package table) and
  Section 2.2 (`PlatformException` subtypes), `platform/platform-core/outbox/pom.xml` (sibling module
  shape to mirror: parent/artifactId pattern, dependency on `platform-common` + slf4j-api + junit
  test-scope only).
- Outputs: `platform/platform-core/lock/pom.xml` (artifactId `platform-lock`, parent `platform-core`),
  `platform/platform-core/lock/src/main/java/com/telco/platform/lock/DistributedLock.java`,
  `LockHandle.java`, `LockAcquisitionException.java`, updated `platform/platform-core/pom.xml`
  `<modules>` list (add `<module>lock</module>`).
- Acceptance Criteria:
  - `platform-lock` builds standalone (`mvn -f platform/pom.xml -pl platform-core/lock -am install`)
    with zero Spring/Redisson dependencies, verified via `mvn dependency:tree`.
  - `DistributedLock`, `LockHandle`, `LockAcquisitionException` match the exact method signatures and
    exception hierarchy specified in ADR-024 Sections 2-3.
  - `LockAcquisitionException` extends `PlatformException` and carries the contended key.
- Dependencies: none (first task of Sprint 17)
- Complexity: S

#### 17.1.2 starter-lock: RedissonDistributedLock, LockAutoConfiguration, LockProperties
- ID: 17.1.2
- Title: Add `starter-lock`: the Redisson-backed implementation and Spring auto-wiring
- Description: Create `platform/platform-starters/starter-lock` (artifactId `starter-lock`, base
  package `com.telco.platform.starter.lock`, registered in `platform-starters/pom.xml` `<modules>`),
  depending on `platform-lock`, `platform-common`, `platform-autoconfigure`, and the plain
  `org.redisson:redisson` client artifact (not Redisson's own opinionated
  `redisson-spring-boot-starter`, per ADR-024 Section 2 - the platform owns the AutoConfiguration and
  property surface). Implement `RedissonDistributedLock implements DistributedLock` (built on
  `RedissonClient.getLock(key)` / `RLock`, watchdog-managed when `leaseTime` is `null`, explicit
  hard-expiry lease when a `Duration` is passed, per ADR-024 Section 4), `LockAutoConfiguration`
  (builds the `RedissonClient` bean from `telco.platform.lock.redis.address`, falling back to
  `spring.data.redis.host`/`port` when unset, per ADR-024's property table; conditional on
  `telco.platform.lock.enabled=true` (default) and Redisson on the classpath;
  `@ConditionalOnMissingBean` override points per ADR-018 Section 3), and `LockProperties`
  (`telco.platform.lock.*`: `enabled` default `true`, `redis.address`, `wait-time` default `5s`,
  `lease-time` unset by default, `watchdog-timeout` default `30s` - the exact table in ADR-024 Section
  2). Register the auto-configuration in
  `src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`,
  matching `starter-security`'s existing pattern.
- Business Purpose: Delivers the actual Redis-backed lock implementation and auto-wiring so a
  consuming service gets a working `DistributedLock` bean by adding one starter dependency, with zero
  Redisson/Spring exposure through platform-core (ADR-024 Sections 2, 6).
- Inputs: ADR-024 Sections 2, 3, 4, 6 in full;
  `platform/platform-starters/starter-security/src/main/java/com/telco/platform/starter/security/SecurityAutoConfiguration.java`
  and its `AutoConfiguration.imports` file as the wiring template; ADR-018 Section 3 (override points).
- Outputs: `platform/platform-starters/starter-lock/pom.xml`,
  `.../starter/lock/RedissonDistributedLock.java`, `.../starter/lock/LockAutoConfiguration.java`,
  `.../starter/lock/LockProperties.java`, `.../starter/lock/RedissonLockHandle.java` (or equivalent
  `LockHandle` implementation wrapping `RLock`),
  `src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`,
  updated `platform/platform-starters/pom.xml` `<modules>` list.
- Acceptance Criteria:
  - A service that adds only `starter-lock` (no `platform-lock` or `org.redisson` direct dependency)
    gets an injectable `DistributedLock` bean with no other config (ADR-018 Section 2's Dependency
    Rule; ADR-024 Section 6).
  - `acquire(key, null)` produces a watchdog-managed lock verified by holding past
    `watchdog-timeout` while the holder keeps renewing; `acquire(key, Duration.ofSeconds(n))` produces
    a lock that hard-expires at exactly `n` seconds regardless of holder liveness (ADR-024 Section 4).
  - `telco.platform.lock.enabled=false` disables the auto-configuration entirely (no `RedissonClient`
    bean created).
  - `@ConditionalOnMissingBean` allows a consuming service to override the `RedissonClient` or
    `DistributedLock` bean.
- Dependencies: 17.1.1
- Complexity: M

#### 17.1.3 Fail-closed exception mapping and platform-bom wiring
- ID: 17.1.3
- Title: Wire fail-closed Redis-unavailability handling and pin Redisson in platform-bom
- Description: Wire `LockAcquisitionException` (thrown by `RedissonDistributedLock` on connection
  failure, timeout, or `wait-time` budget expiry, per ADR-024 Section 5) into the platform's existing
  `GlobalExceptionHandler` (`starter-api`) so it maps to HTTP `503` with
  `DependencyFailureException`-equivalent semantics, consistent with how other
  infrastructure-unavailability failures are surfaced (`docs/architecture/platform-capabilities.md`
  Section 1). Pin the Redisson version in `platform-bom` (new `redisson.version` property,
  `org.redisson:redisson` dependency-management entry) and add `platform-lock` and `starter-lock`
  coordinate entries to `platform-bom`'s `<dependencyManagement>`, following the exact pattern already
  used for `platform-outbox`/`starter-outbox`.
- Business Purpose: Makes the fail-closed behavior ADR-024 Section 5 mandates externally observable as
  a `503`, not an uncaught `500`, and ensures every consuming service inherits a single pinned
  Redisson version rather than resolving one transitively (ADR-003, ADR-020).
- Inputs: ADR-024 Section 5 (fail-closed semantics, `LockAcquisitionException` -> `PlatformException`
  mapping), `starter-api`'s existing `GlobalExceptionHandler` (exception-to-status mapping table),
  `platform/platform-bom/pom.xml` (existing pattern: `<properties>` version +
  `<dependencyManagement>` entries for `platform-outbox`/`starter-outbox`).
- Outputs: `GlobalExceptionHandler` update (new `@ExceptionHandler(LockAcquisitionException.class)`
  branch, or an entry in its existing type-to-status mapping table) mapping to `503`,
  `platform/platform-bom/pom.xml` updates (`redisson.version` property, `org.redisson:redisson`
  dependency, `platform-lock` and `starter-lock` dependencyManagement entries).
- Acceptance Criteria:
  - With Redis unreachable, a call through `DistributedLock.acquire`/`withLock` from a
    controller-reachable path returns HTTP `503` with a stable `ApiError.code`, not a generic `500`.
  - `platform-bom` pins exactly one Redisson version; no service or starter POM hardcodes a Redisson
    version (ADR-020).
  - `mvn -f platform/pom.xml -am install` (full platform reactor, including the new modules) succeeds.
- Dependencies: 17.1.1, 17.1.2
- Complexity: S
