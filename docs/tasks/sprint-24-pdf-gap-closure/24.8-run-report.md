## Finding: transient-retry exhaustion drops `subscription.activated.v1` (PRE-EXISTING, open)

`AddonOnboardingAcceptanceIT` failed because the order never left CONFIRMED. Root cause, from
order-service logs:

```
Failed to process subscription.activated.v1 messageId=12fb703a-...: Order 3c5c6441-... not yet
CONFIRMED - payment.completed.v1 has not caught up with subscription.activated.v1; retry on
redelivery
...
Backoff FixedBackOffExecution[interval=0, currentAttempts=10, maxAttempts=9] exhausted
```

`subscription.activated.v1` and `payment.completed.v1` ride independent topics and consumer groups
with no ordering guarantee. The Sprint 14 fix (14.1.1 finding #5) correctly made a not-yet-CONFIRMED
order a TRANSIENT case that re-throws "so Kafka's built-in retry redelivers shortly after" - but the
error handler retries with `interval=0`, so all ten attempts burn inside roughly one second. If the
payment event has not been processed in that window, the offset is committed and **the activation
event is lost permanently**.

Blast radius when it fires (it is a race, not deterministic - 5 of 6 order-driven scenarios passed
in this run):

- The subscriber IS served: payment COMPLETED, subscription ACTIVE, MSISDN assigned, quota
  provisioned, billing record created. The platform is not down.
- The order row is stuck at CONFIRMED forever, so the onboarding wizard - which polls for
  FULFILLED - hangs for the customer.
- **For bundled-addon orders the paid-for addon is never delivered**: `addon.purchased.v1` is
  published during fulfillment, so no quota top-up and no `addon_charge_record` (hence no invoice
  line) is ever created. The customer is charged for an addon they do not receive.

This is pre-existing platform behaviour, not introduced by Sprint 24; Sprint 24's bundled-addon
flow raised the stakes by hanging addon provisioning off fulfillment, and has enough extra work in
the window to lose the race repeatably.

Recommended fix (NOT applied - it touches the shared consumer error-handling used by every
service, so it deserves its own review): give the transient path a real backoff (non-zero interval
with a bounded number of attempts, or a retry topic) instead of an instant burn-down. A second,
cosmetic finding from the same logs: `payment.completed.v1` for AC-02's direct invoice payments
(which legitimately carry a random `orderId`) makes order-service throw "Order not found" and burn
ten retries - that should be a terminal skip, not an ERROR.

## Browser E2E with addon selection - addon path PROVEN, activation blocked by the finding above

Driven in real Chrome against the live stack with a freshly provisioned SUBSCRIBER:

- PKCE login succeeded on the first attempt (S256 challenge visible in the authorize redirect).
- `POST /api/v1/addons` (Feature 24.1, previously a documented 500) returned **201** live when
  creating a DATA addon linked to a purpose-built tariff.
- The onboarding wizard's step 3 **rendered the linked addon as a checkbox** ("Browser 5GB Data
  Pack, TRY 25.00"), proving catalog seeding -> `GET /api/v1/addons?tariffCode=` -> web-bff
  composition -> UI end to end.
- Selecting it updated the monthly total to **TRY 145.00** (120.00 tariff + 25.00 addon) and the
  review step listed the addon.
- Placing the order produced exactly the intended shape in `order_db.order_items`, which is the
  assertion the browser run exists for (the acceptance ITs bypass the BFF and call the gateway
  directly), proving **Feature 24.3's web-bff `addonCodes` forwarding works from the real UI**:

| item_type | product_code | tariff_name | unit_price | allowance_data_mb |
| --- | --- | --- | --- | --- |
| TARIFF | (null) | Browser E2E Tariff | 120.00 | (null) |
| ADDON | BRWADDON-08EF3A82 | Browser 5GB Data Pack | 25.00 | 5120 |

The order then stalled before activation and the wizard hung on its activation step - the same
class of failure as the sweep, and a direct demonstration of the customer-visible impact described
in the finding above (observed live; screenshots were intentionally not committed to the repo).
Two contributing causes, both understood:

1. **Process error in this run (mine, not the product's):** the stack was brought down and back up
   for the browser phase, and the Debezium connectors were not re-registered afterwards
   (`GET /connectors` returned `[]`), so `order.created.v1` never published and the order sat at
   PENDING. Registering the 11 connectors immediately advanced it to CONFIRMED. **Re-registering
   connectors after every stack restart is mandatory** - noted for future runs.
2. During that connector-less window, subscription-service's circuit breaker to order-service
   tripped OPEN, and the `payment.completed.v1` retries exhausted against the same zero-interval
   backoff, so activation never ran for this order - the finding above, reached from a second
   direction.
