# 24.4 - Plan Change Flow

Status: TODO. Owner: domain-engineer + event-integration. Spec refs: FR-09 (paket degisikligi),
PDF event storming TariffChanged. Design: design-note.md D2/D4.

## Tasks

1. New event `subscription.tariff-changed.v1`: canonical avsc (subscriptionId, customerId,
   msisdn, previousTariffCode, newTariffCode, newTariffVersion, orderId, changedAt); rides the
   existing `subscription.events` topic (third event type - eventType-header filtering and
   per-listener groupIds mandatory); pom subject; compat test; event-catalog row.
2. subscription-service: `Subscription.changeTariff(newCode, newVersion)` guarded ACTIVE-only;
   `ChangeTariffCommand`/handler (re-validate ownership/status, version from order snapshot,
   outbox publish, audit; failure reuses the activation-failed emitter - documented reuse, D2);
   `PaymentCompletedEventConsumer` branches on fetched orderType (NEW_LINE -> activate,
   PLAN_CHANGE -> changeTariff, ADDON -> ignore).
3. usage-service: `TariffChangedEventConsumer` (own groupId, inbox) ->
   `ReprovisionQuotaCommand`/handler: reset current-period quota to new allowances,
   `remaining = max(0, newTotal - used)`; notified flags recomputed.
4. billing-service: `TariffChangedBillingConsumer` (own groupId, inbox) updates
   `SubscriberBillingRecord.tariffCode` so the next bill uses the new fee.
5. order-service: `SubscriptionTariffChangedEventConsumer` (own groupId, inbox) fulfills
   PLAN_CHANGE orders.
6. Update `docs/api-contracts/order-service.md` and subscription contract for the new kinds.

## Acceptance criteria

- Plan-change order: payment charged (new monthly fee), subscription tariffCode/version updated,
  quota re-provisioned, order FULFILLED, next bill run bills the new tariff fee.
- changeTariff failure refunds and cancels the order via the existing compensation saga.
