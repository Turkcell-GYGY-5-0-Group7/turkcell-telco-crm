# 24.2 - Order Model Generalization

Status: DONE. Owner: domain-engineer. Spec refs: FR-09, PDF data model OrderItem(productCode,
productType). Design: design-note.md D1/D2.

## Tasks

1. `V8__order_item_types.sql`: `orders.order_type` (default NEW_LINE); `order_items.item_type`
   (default TARIFF), `product_code`, `target_subscription_id`, allowance snapshot columns; relax
   tariff column NOT NULLs (ADDON items carry no tariff).
2. Entities + factories; `OrderItemRequest` gains `itemType` (default TARIFF), `productCode`,
   `targetSubscriptionId`; responses expose all new fields.
3. `CreateOrderCommandHandler` validation matrix: NEW_LINE = exactly 1 TARIFF + 0..N ADDON;
   ADDON = 1..N ADDON items, same required `targetSubscriptionId`, validated ACTIVE + owned;
   PLAN_CHANGE = exactly 1 item with tariffId + targetSubscriptionId, tariff differs from
   current; `campaignCode` rejected on non-TARIFF items; ADDON items priced from the 24.1
   internal snapshot.
4. New tokenless `GET /internal/subscriptions/{id}` on subscription-service
   (`SubscriptionInternalController` + query/handler: id, customerId, status, tariffCode,
   tariffVersion, msisdn) + `SubscriptionServiceClient` in order-service.
5. Relax the one-line invariant in the activation consumer: exactly one TARIFF item, ADDON items
   allowed; two TARIFF items still fail activation (keep the regression test, add the
   addon-allowed inversion).
6. `order-created.avsc`: additive item fields itemType (default "TARIFF"), productCode,
   targetSubscriptionId (nullable defaults) + compat test + event-catalog payload row; mirror in
   `OrderCreatedEvent` and the order internal endpoint + subscription-service client DTOs.

## Acceptance criteria

- All three order types validate per the matrix with platform validation errors on violations.
- payment-service and campaign-service consumers unaffected (their tests stay green).
- Avro compat test green (BACKWARD).
