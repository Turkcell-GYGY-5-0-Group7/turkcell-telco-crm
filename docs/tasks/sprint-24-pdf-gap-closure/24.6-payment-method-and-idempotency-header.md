# 24.6 - Payment Method + Idempotency-Key Header

Status: DONE. Owner: domain-engineer. Spec refs: FR-25 (kredi karti, banka transferi, cuzdan),
FR-26, PDF Section 12 (Idempotency-Key header on Payment/Order POSTs; orders already comply).
Design: design-note.md D6.

## Tasks

1. `V5__payment_method.sql`: `method varchar(20) not null default 'CREDIT_CARD'` on `payments`.
2. `PaymentMethod` enum {CREDIT_CARD, BANK_TRANSFER, WALLET}; `Payment` column + factory;
   optional `method` on `ChargePaymentRequest` (handler defaults CREDIT_CARD); exposed in
   `PaymentResponse`; `OrderCreatedEventConsumer` auto-charge uses CREDIT_CARD;
   `MockPspAdapter` untouched (ignores method).
3. `PaymentController.charge`: `@RequestHeader(value = "Idempotency-Key", required = false)` -
   when present it wins as `paymentRequestId`; body field stays for back-compat (header
   plumbing in the controller, not business logic - ADR-008 compliant).
4. Update `docs/api-contracts/payment-service.md`.

## Acceptance criteria

- Charge with each method persists and echoes it; default applied when omitted.
- Same Idempotency-Key header twice -> one payment, second call returns the original (existing
  idempotency semantics via header).
- payment-service suite green.
