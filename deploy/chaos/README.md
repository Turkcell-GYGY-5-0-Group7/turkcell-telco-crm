# telco-chaos-mesh Helm chart

Chaos Mesh, in-cluster, as the platform's fault-injection control plane (Sprint 20,
task 20.1.1). Wraps the official `chaos-mesh/chaos-mesh` chart as a dependency (see
`Chart.yaml`), the same vendored-subchart pattern already used by
[`deploy/helm/vault`](../helm/vault) and [`deploy/helm/csi-driver`](../helm/csi-driver)
(`dependencies:` block, `helm dependency update`-vendored `charts/chaos-mesh-2.8.3.tgz` +
`Chart.lock`, both committed - see `deploy/helm/README.md`'s chart-layout precedent).

Tool selection (Chaos Mesh over Litmus and hand-rolled `kubectl`/`tc`/`iptables` scripts)
is decided and documented in
[`docs/tasks/sprint-20-chaos-engineering/README.md`](../../docs/tasks/sprint-20-chaos-engineering/README.md)
Feature 20.1 notes - not re-derived here.

## Chart layout

```
deploy/chaos/
  Chart.yaml    # dependencies: chaos-mesh/chaos-mesh, pinned to 2.8.3
  Chart.lock    # generated by `helm dependency update`, committed
  charts/       # vendored charts/chaos-mesh-2.8.3.tgz, committed
  values.yaml   # telco-namespace scoping, pinned image tags, dashboard.create: false
  README.md     # this file
```

No new namespace is created by this chart - it installs into the existing `telco`
namespace (`deploy/RUNBOOK.md` Section 2 already creates and annotates it). No
`microservices/*` image or Dockerfile changes: Chaos Mesh is a cluster-level
controller-manager + chaos-daemon DaemonSet that injects faults from outside the
target pods, nothing is baked into any service image.

## Install

```sh
helm dependency update deploy/chaos
helm upgrade --install chaos-mesh deploy/chaos -n telco -f deploy/chaos/values.yaml
kubectl -n telco rollout status deploy/chaos-mesh-controller-manager --timeout=180s
kubectl -n telco rollout status daemonset/chaos-mesh-chaos-daemon --timeout=180s
```

Run this after the Sprint 15 cluster/namespace and the `telco-deps` stateful backends
are up (`deploy/RUNBOOK.md` Section 2); it does not depend on Vault, the CSI driver, or
Linkerd, and installing/upgrading it never touches any existing `telco-deps` or service
release (separate Helm release name, `chaos-mesh`, distinct from every other release in
the namespace).

Chart version and every image tag are pinned explicitly in `values.yaml` (no `latest`),
consistent with `deploy/RUNBOOK.md` Section 4.1.2's image-tag-pinning discipline.

### Kind-specific note (`chaosDaemon.runtime`)

A Kind node's container runtime is containerd, not Docker, even though the chart's own
default (`chaosDaemon.runtime: docker`, `socketPath: /var/run/docker.sock`) targets
Docker. `values.yaml` overrides `chaosDaemon.runtime: containerd` and
`chaosDaemon.socketPath: /run/containerd/containerd.sock` - the exact override the
upstream chart's own `values.yaml` comments recommend for Kind. Without this override,
chaos-daemon cannot reach the node's runtime socket and pod-kill/network-partition
experiments (20.3) fail at the daemon rather than in the experiment itself. A
non-Kind/production cluster running containerd needs the same override; one running
Docker Engine directly as its CRI would not.

## CRD verification (20.1.2)

Run after install (or after any future chart upgrade) to confirm the CRDs the 20.3
experiment library depends on are actually registered with a usable schema, not just
present as an object:

```sh
kubectl get crds | grep chaos-mesh.org
```

Expected output includes at minimum (full list is longer - `stresschaos`, `workflow`,
`iochaos`, `timechaos`, etc. also ship with the chart but are out of scope for 20.3):

```
networkchaos.chaos-mesh.org
podchaos.chaos-mesh.org
schedules.chaos-mesh.org
...
```

```sh
kubectl api-resources | grep chaos-mesh
```

Expected output lists each CRD's short name/kind/namespaced column, e.g.:

```
podchaos                                        chaos-mesh.org/v1alpha1               true         PodChaos
networkchaos                                     chaos-mesh.org/v1alpha1               true         NetworkChaos
schedules                                        chaos-mesh.org/v1alpha1               true         Schedule
```

```sh
kubectl explain podchaos.spec
kubectl explain networkchaos.spec
```

Both must resolve to a field list (not `error: couldn't find resource`) - this proves
the CRD's OpenAPI schema is registered, not merely the CRD object itself. `PodChaos` and
`NetworkChaos` back 20.3's pod-kill, latency-injection, and network-partition
experiments; `Schedule` is verified for future scheduled runs even though 20.5 forbids
using it this sprint (manual `kubectl apply` only).

## Dashboard access decision (20.1.3)

The Chaos Mesh web dashboard is left **disabled** (`dashboard.create: false` in
`values.yaml` - the upstream chart's actual toggle for the dashboard Deployment/
PVC/Service; there is no separate `dashboard.enabled` key). This sprint's operating
model is a single operator applying/deleting Chaos Mesh CRD manifests directly
(`kubectl apply -f <experiment>.yaml`, `kubectl delete -f <experiment>.yaml`) with no
scheduled or automatic chaos (README Feature 20.5), so a second portal-style web app
adds operational surface - another Deployment, Service, and (optionally) Ingress to
secure and keep patched - without adding capability the CRDs and `kubectl` do not
already provide. This mirrors the README's stated reason for rejecting Litmus for this
scope (its separate Litmus Portal web app). An operator inspects a running or completed
experiment's status the same way any other Kubernetes custom resource is inspected,
without a dashboard:

```sh
kubectl -n telco get podchaos
kubectl -n telco describe podchaos/<name>
kubectl -n telco get networkchaos
kubectl -n telco describe networkchaos/<name>
```

`kubectl describe` surfaces the experiment's `status.experiment` phase transitions and
events, which is sufficient for this sprint's game days (20.4). If a later game day
needs richer visualization, the documented path is to set `dashboard.create: true` (and
`dashboard.service.type`/`ingress` as needed) in a values override at install time - no
chart change required.

## Blast-radius guardrails (20.5)

### 20.5.1 RBAC and namespace scoping

Inspected directly (this session, Docker Desktop down so no live cluster check was
possible - see the deferred item below): the real vendored chart at
`deploy/chaos/charts/chaos-mesh-2.8.3.tgz` was extracted to a scratch directory (not
committed) and its `templates/controller-manager-rbac.yaml` was read in full, along
with the chart's own default `values.yaml`.

Findings, factual, not assumed:

- The chart's own default is `clusterScoped: true` and `controllerManager.targetNamespace:
  chaos-mesh` (unused while `clusterScoped: true`).
- `templates/controller-manager-rbac.yaml` defines three RBAC surfaces:
  1. A ClusterRole (`<release>-chaos-controller-manager-target-namespace`) granting
     `get/list/watch/delete/update/patch/create` on `pods/configmaps/secrets`, `get`
     on `pods/log`, and `"*"` on all `chaos-mesh.org` resources - this is the
     permission set that actually lets the controller inject/kill faults. Its binding
     is conditional: `{{- if .Values.clusterScoped }} ClusterRoleBinding {{- else }}
     RoleBinding {{- end }}`, namespaced to `.Values.controllerManager.targetNamespace`
     in the `RoleBinding` branch. This means, contrary to a "the chart can only be
     cluster-scoped" assumption, this specific permission set CAN be namespace-scoped
     by the chart itself when `clusterScoped: false`.
  2. A second ClusterRole (`<release>-chaos-controller-manager-cluster-level`)
     granting `get/list/watch` on `nodes/persistentvolumes/persistentvolumeclaims`
     (plus `namespaces`/`services` only when `clusterScoped: true`) and `create` on
     `authorization.k8s.io/subjectaccessreviews`. Its `ClusterRoleBinding` in the
     template carries **no** `{{- if .Values.clusterScoped }}` guard - it is always
     cluster-wide regardless of the setting. This is the one piece of RBAC that
     genuinely cannot be namespace-scoped by this chart: read-only node/PV/PVC
     watch plus SAR creation, cluster-wide, unconditionally.
  3. A namespaced `Role`/`RoleBinding` for the controller's own control-plane
     namespace (services/secrets/endpointslices/leases/configmaps/pods-exec) -
     already namespace-scoped by chart design, not affected by `clusterScoped`.
  4. `templates/chaos-daemon-rbac.yaml` only renders a `ClusterRole`/`RoleBinding`
     for a PodSecurityPolicy when `chaosDaemon.podSecurityPolicy: true`; the chart
     default is `podSecurityPolicy: false`, so chaos-daemon adds no additional
     cluster-wide RBAC in this install.

- This install's previous `values.yaml` (20.1) left `clusterScoped`/
  `controllerManager.targetNamespace` at chart defaults, meaning permission set (1)
  above - the one that matters for fault injection - was cluster-wide via
  `ClusterRoleBinding`. `values.yaml` now sets `clusterScoped: false` and
  `controllerManager.targetNamespace: telco` (20.5.1), which changes template branch
  (1) to a `RoleBinding` scoped to the `telco` namespace, so the controller's actual
  fault-injection permissions (pods, configmaps, secrets, chaos-mesh.org CRs) are now
  namespace-scoped to `telco` by RBAC itself, not only by the experiment manifests'
  own `selector`.
  Note the earlier README text "20.1's `values.yaml` scopes `controllerManager`'s
  own operational namespace to `telco`" referred only to where the controller Pod is
  scheduled (via `-n telco` on `helm upgrade --install`), not what the controller's
  RBAC could reach - those are two different things, and 20.5.1 is what closes that
  gap for the reachable-permissions side.

- **Accepted residual risk**: permission set (2) above - cluster-wide read/watch of
  `nodes`/`persistentvolumes`/`persistentvolumeclaims` and cluster-wide `create` on
  `subjectaccessreviews` - cannot be namespace-scoped by this chart; its
  `ClusterRoleBinding` template carries no conditional. This is accepted as low risk
  because: (a) the permissions are read-only list/watch on cluster infrastructure
  objects (not pods, not secrets in other namespaces, no delete/update/patch verbs),
  not a fault-injection capability; (b) every 20.3 experiment manifest's
  `selector.namespaces` (and, for the two `NetworkChaos` manifests, `target.selector.namespaces`
  too) is hard-set to `["telco"]` - verified by direct read of all three files this
  session (`pod-kill-order-service.yaml`, `latency-order-to-customer.yaml`,
  `partition-billing-service-kafka.yaml`) - so even if the controller could
  theoretically reach objects outside `telco`, no experiment Custom Resource in this
  repository actually targets outside it; and (c) this is a single-cluster, single
  Kind-node dev environment (`deploy/RUNBOOK.md`) with no `production` namespace for
  anything to accidentally reach.

- **Deferred to a live cluster (NOT verified this session - Docker Desktop is
  down)**: once the cluster is reachable, an operator must run

  ```sh
  kubectl auth can-i --list --as=system:serviceaccount:telco:chaos-controller-manager -n telco
  kubectl auth can-i --list --as=system:serviceaccount:telco:chaos-controller-manager -n default
  kubectl get ns
  ```

  to confirm empirically that (i) the effective permission list matches the static
  analysis above (namespace-scoped `RoleBinding` for the fault-injection ClusterRole,
  cluster-wide only for the read-only node/PV/PVC/SAR ClusterRole), and (ii) no
  namespace other than `telco` (plus the cluster's own system namespaces) exists on
  this Kind cluster for that residual cluster-wide read to reach. Also run
  `helm template chaos-mesh deploy/chaos -n telco -f deploy/chaos/values.yaml` and
  confirm the rendered `controller-manager-rbac.yaml` output actually contains a
  `RoleBinding` (not `ClusterRoleBinding`) named
  `chaos-mesh-chaos-controller-manager-target-namespace` in namespace `telco` -
  `helm` was not available in this session's shell, so this rendering was not run;
  the conclusion above is from reading the chart's raw Go template source instead.

### 20.5.3 Manual-only execution and CI non-interference

Both checks below were run for real this session (local grep, no cluster required)
and both returned no matches, confirming the required (empty) result:

```sh
grep -rl "kind: Schedule\|kind: Workflow" deploy/chaos/*.yaml deploy/chaos/experiments/*.yaml deploy/chaos/values.yaml
# (no output - no Schedule or Workflow resource defined anywhere in the
# hand-authored manifests; the vendored charts/chaos-mesh-2.8.3.tgz is excluded
# as instructed, since it is upstream binary content, not an authored manifest)

grep -rl "deploy/chaos" .github/workflows/
# (no output - no CI/CD workflow references deploy/chaos in any job)
```

This confirms: every 20.3 experiment is a plain `PodChaos`/`NetworkChaos` object
applied and deleted manually (`kubectl apply -f` / `kubectl delete -f`), never a
`Schedule` or `Workflow` resource, and the existing CI/CD build-deploy gate
(ADR-014, Sprint 15.4) is completely untouched by this sprint. Any future sprint
that introduces scheduled/automatic chaos (a `Schedule` or `Workflow` resource, or
a CI job that touches `deploy/chaos/`) must consciously revisit and update this
section rather than silently inherit the manual-only assumption recorded here.
